# Policy Bundle Runtime PoC Planning v0

Status: v0_implemented
Production-ready: no
Runtime policy engine: `StaticPolicyEngine`

This planning package records the runtime PoC path that follows `docs/README.md`: detailed production-adjacent policy runtime plans live under `docs/roadmaps/`, while reusable policy domain inventories live under `docs/reference/`.

## What v0 Implements

- A planning-only runtime PoC direction for a future candidate policy runtime.
- `StaticPolicyEngine` remains the only runtime policy engine.
- Policy Bundle / OPA-Cedar Planning v0 remains the engine comparison and bundle governance source.
- Golden policy decisions are treated as the future compatibility baseline for any candidate runtime.
- Runtime enforcement, signed bundle verification, and shadow evaluation remain false.

## Current Runtime Baseline

`packages/policy` owns provider-neutral policy models, static rules, `StaticPolicyEngine`, policy audit, DTOs, and service-boundary checks. The static engine is deny-by-default, deterministic, and does not execute dynamic policy code.

The runtime PoC does not replace or wrap the static engine. All Git, LLM, MCP, Runner, Registry, Improvement, SecretRef, Secrets/Sandbox, Provider, Local Agent, Auth, Dashboard, and Deployment Readiness gates must continue to use existing static decisions until a separately approved runtime implementation exists.

## Future Candidate Direction

The recommended first candidate remains a schema-driven signed JSON/YAML policy bundle evaluator because it can be compared against static decisions without introducing Rego, Cedar, WASM, remote policy loading, or an external policy service.

OPA/Rego and Cedar remain future candidate runtime kinds for evaluation only. They are not implemented in v0 or v1 planning.

## Out Of Scope

- OPA/Rego runtime.
- Cedar runtime.
- Signed bundle verification runtime.
- Candidate runtime execution.
- Dynamic policy execution, `eval`, `Function`, WASM, remote policy loading, or hot reload.
- Production policy rollout, rollback, break-glass execution, or enforcement change.
- External policy service calls.
- Secret or env value exposure.

## Follow-up

Policy Runtime PoC Golden Test Harness v1 defines the deterministic source-of-truth cases. Policy Runtime Shadow Evaluation Planning v1 defines how a future candidate runtime would be compared without changing enforcement.
