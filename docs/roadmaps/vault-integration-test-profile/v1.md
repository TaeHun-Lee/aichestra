# Vault Integration-Test Profile v1

Status: `v1_implemented`

Vault Integration-Test Profile v1 defines a controlled, skipped-by-default readiness surface for validating Vault-backed SecretRef resolution against a non-production Vault KV v2 path. It does not make Vault the default backend, does not require Vault for default tests, and does not make secret management production-ready.

Vault Live Integration Enablement v1 builds on this profile with stricter manual-run readiness metadata, test-only path enforcement, a safe runbook, and `/readiness/secrets/vault/live-*` endpoints. The profile remains skipped by default.

## What v1 Implements

- `VaultIntegrationTestProfile`, `VaultIntegrationTestCase`, `VaultIntegrationTestSafetyCheck`, and `VaultIntegrationTestReadinessSummary` read models in `packages/deployment-readiness`.
- Read-only readiness service methods:
  - `getVaultIntegrationTestProfile()`
  - `listVaultIntegrationTestCases()`
  - `listVaultIntegrationTestSafetyChecks()`
  - `getVaultIntegrationTestReadinessSummary()`
  - `canRunVaultIntegrationLiveTests()`
- Read-only API endpoints:
  - `GET /readiness/vault-integration/profile`
  - `GET /readiness/vault-integration/test-cases`
  - `GET /readiness/vault-integration/safety-checks`
  - `GET /readiness/vault-integration/summary`
- Safe `/health` metadata under `vaultIntegrationTests`.
- Dashboard read model and panel under `GET /dashboard/vault-integration`.
- Vault Live Integration Enablement v1 read-only live readiness/check/runbook/summary metadata under `/readiness/secrets/vault/live-*`.
- Deterministic tests for readiness models, API routes, dashboard rendering, health metadata, no-secret/no-env exposure, unsafe-gate classification, path allowlist rules, and a skipped live-test skeleton.

## What v1 Does Not Implement

- No Vault call in default runtime or default tests.
- No production Vault rollout, HA, unseal, storage backend, cluster bootstrap, Vault Agent, Vault CSI, sidecar, Kubernetes auth, AppRole production rollout, Terraform, Helm, Kubernetes manifest, or cloud infrastructure.
- No Vault secret write, delete, rotate, migration, broad path listing, or mutation.
- No BYOK, OAuth, device-code, WIF, IAM, or cloud identity token exchange.
- No credential cache reads.
- No production secret rotation jobs.
- No production-ready secret backend claim.

## Required Environment Gates

Live Vault tests skip unless all required gates are configured:

```text
AICHESTRA_VAULT_INTEGRATION_TESTS=true
AICHESTRA_SECRET_BACKEND_PROVIDER=vault
AICHESTRA_ENABLE_VAULT_SECRET_PROVIDER=true
AICHESTRA_VAULT_ADDR=<configured but never returned>
AICHESTRA_VAULT_AUTH_METHOD=token
AICHESTRA_VAULT_TOKEN=<configured but never returned>
AICHESTRA_VAULT_KV_MOUNT=<test KV mount>
AICHESTRA_VAULT_ALLOWED_PATH_PREFIXES=<non-production test prefix list>
AICHESTRA_TEST_VAULT_SECRET_PATH=<non-production test-only path>
AICHESTRA_TEST_VAULT_SECRET_KEY=<test key name>
```

Optional:

```text
AICHESTRA_VAULT_NAMESPACE=<configured but never returned>
AICHESTRA_VAULT_REQUEST_TIMEOUT_MS=<bounded timeout>
```

Missing required gates skip live tests. Unsafe gates set `canRunLiveTests=false` and classify the profile as blocked until corrected.

## Test-only Vault Path Requirements

`AICHESTRA_TEST_VAULT_SECRET_PATH` must:

- be under one configured `AICHESTRA_VAULT_ALLOWED_PATH_PREFIXES` prefix;
- be clearly non-production and test-only, for example containing `test`, `tests`, `dev`, `development`, `integration`, `sandbox`, `nonprod`, or `ci`;
- avoid production-like path segments;
- refer to one small test secret and one configured key;
- never be listed broadly, written, deleted, rotated, or migrated by this profile.

Readiness surfaces return configured booleans and prefix counts only. They do not return Vault address values, namespace values, token values, raw secret paths, raw key names, env values, or secret values.

## SecretRef Test Pattern

The profile models this metadata-only pattern:

```json
{
  "provider": "vault",
  "secretKind": "provider_api_key",
  "metadata": {
    "vaultMount": "<AICHESTRA_VAULT_KV_MOUNT>",
    "vaultPath": "<AICHESTRA_TEST_VAULT_SECRET_PATH>",
    "vaultKey": "<AICHESTRA_TEST_VAULT_SECRET_KEY>",
    "dataShape": "single_key"
  }
}
```

The SecretRef stores references only. The live test path, key, and value must not appear in API, health, dashboard, audit, test output, snapshots, errors, or docs examples.

## Path Allowlist Policy

The test path must match a configured prefix by exact match or child path. For example, a prefix of `aichestra/test` can allow `aichestra/test/vault-smoke`, but not `aichestra/prod/vault-smoke`.

Unsafe gate warnings include:

- `vault_auth_method_not_token`
- `test_secret_path_not_allowlisted`
- `test_secret_path_not_test_only`
- `production_like_test_path_configured`
- `path_allowlist_prefix_not_test_only`
- `production_like_path_allowlist_configured`
- `test_secret_key_not_test_marker`
- `vault_write_enabled`
- `vault_delete_enabled`
- `vault_rotate_enabled`
- `vault_broad_list_enabled`
- `credential_cache_reference_configured`

## Live Test Skip Behavior

Default validation uses deterministic mock/readiness checks and does not call Vault. Optional live Vault tests are skipped unless `canRunVaultIntegrationLiveTests()` returns true. A missing gate is a skip reason, not a default-suite failure. Unsafe gates block live execution.

The existing Vault-backed Secret Backend v1 gated live skeleton owns the actual one-secret KV v2 read path. This profile records whether that live path is safe to attempt and exposes the readiness state as first-class API, health, and dashboard data.

## Safety Policy

Allowed operations:

- config validation;
- SecretRef metadata validation;
- path allowlist validation;
- one gated KV v2 read status check from an allowlisted test-only path;
- one gated CredentialHandle resolution status check;
- Auth/RBAC and Policy gate validation;
- audit/redaction validation;
- no-secret/no-env exposure validation.

Forbidden operations:

- Vault write;
- Vault delete;
- Vault rotate;
- broad Vault path listing;
- production path access;
- raw secret return;
- Vault token return;
- env value return;
- credential cache read;
- production Vault rollout.

## API and Dashboard

Readiness endpoints are read-only and do not call Vault:

```text
GET /readiness/vault-integration/profile
GET /readiness/vault-integration/test-cases
GET /readiness/vault-integration/safety-checks
GET /readiness/vault-integration/summary
GET /readiness/secrets/vault/live-readiness
GET /readiness/secrets/vault/live-checks
GET /readiness/secrets/vault/live-runbook
GET /readiness/secrets/vault/live-summary
GET /dashboard/vault-integration
```

`/health` exposes only safe metadata: enabled boolean, required/configured/missing/unsafe gate counts, Vault backend selected boolean, Vault provider enabled boolean, Vault address configured boolean, Vault token configured boolean, path allowlist prefix count, no-write/no-delete/no-rotate/no-broad-list booleans, and no-secret/no-env booleans.

## Known Limitations

- Live Vault validation still requires an operator-managed non-production Vault instance and test-only KV v2 secret.
- v1 uses token auth for the optional live profile only; AppRole, Kubernetes auth, and workload identity remain future work.
- The profile does not create, rotate, migrate, delete, or clean up Vault secrets.
- Production Vault deployment, production secret rotation, production Auth/RBAC, tenant isolation, and incident response procedures remain incomplete.
