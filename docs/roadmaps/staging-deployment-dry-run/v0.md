# Staging Deployment Dry-run Profile v0

Status: v0_implemented
Scope: read-only readiness aggregation only
Production ready: no
Staging deployed: no

## What v0 Implements

Staging Deployment Dry-run Profile v0 aggregates existing readiness surfaces into one deterministic, non-deploying staging dry-run report. It evaluates whether the project appears ready to attempt staging deployment validation, while keeping the default runtime mock-first and safe.

Implemented surfaces:

- Dry-run profile, source, check, blocker, integration-profile, report, and summary models in `packages/deployment-readiness`.
- Read-only aggregation across staging profile, CI/CD, Postgres operations, GitHub App hardening, GitHub and LLM integration-test profiles, LLM Gateway, MCP, Auth/RBAC, Secret Backend Migration, SecretRef credentials, Policy Bundle planning, Observability, Dashboard, Local Agent, Runner, and Git.
- Read-only `/readiness/staging-dry-run/*` API endpoints.
- Safe `/health` dry-run metadata.
- `/dashboard/staging-dry-run` read model and web dashboard panel.
- Deterministic tests for models, aggregation rules, API routes, health metadata, dashboard, skipped integration profiles, and no-secret/no-env behavior.

## What v0 Does Not Implement

- Deployment execution.
- Active CI workflows.
- Kubernetes, Terraform, Pulumi, Helm, cloud infrastructure, artifact registry, or production deployment code.
- Live GitHub, LLM, MCP, auth, cloud, policy engine, secret backend, registry, Temporal, Kubernetes, or observability calls.
- Remote integration-test execution.
- Secret reads, credential-cache reads, token/session/JWT issuance, or production credential issuance.
- Staging deployed or production ready claims.

## Aggregation Rules

- Critical fail in a required source makes the report `blocked`.
- Any critical open blocker makes the report `blocked`.
- Any `blocks_staging_dry_run` blocker makes the report `blocked`.
- Required high/medium fail checks or `blocks_staging_deployment` blockers make the report `not_ready` when no critical blocker exists.
- Warnings, skipped optional profiles, future sources, or not-checked validation status make the report `pass_with_warnings` when no fail/blocker exists.
- Missing production auth is a production blocker and staging warning when staging explicitly allows mock auth for read-only dry-run.
- Missing real secret backend is a production blocker and staging warning unless the staging profile requires real provider validation.
- Missing Postgres is a staging fail because Staging Deployment Profile v0 requires or strongly recommends Postgres.
- Missing GitHub App live gates and missing LLM live gates are skipped, not failures, unless the dry-run requirements explicitly require the live profile.
- Remote merge, force push, branch deletion, vendor CLI execution, real MCP transport without readiness, secret/env exposure, or validation command failure are critical blockers.

## Guarantees

No-deployment guarantee:

- No infrastructure resources are created.
- No deployment command is run.
- No staging or production traffic is enabled.
- No Kubernetes, Terraform, Pulumi, Helm, or cloud provider code is added.

No-external-call guarantee:

- The dry-run does not call GitHub, LLM providers, MCP servers, Vault, cloud services, external IdPs, external policy engines, observability exporters, artifact registries, Temporal, or Kubernetes.
- The dry-run classifies optional integration profiles but does not execute them.

No-secret/no-env guarantee:

- API, health, dashboard, and report data return only booleans, counts, ids, statuses, severities, and documentation references.
- Secret values, env values, database URLs, private keys, webhook secrets, tokens, session IDs, JWTs, raw prompts, raw provider responses, raw logs, raw webhook payloads, and credential-cache paths must not be exposed.

## Relationship To Staging CI/CD

Staging CI/CD Pipeline Planning v0 defines validation jobs and optional integration gates. The dry-run does not create or run CI jobs. It reports whether CI/CD planning is sufficient to attempt a future staging validation.

## Relationship To Staging Release Candidate Checklist

Staging Release Candidate Checklist v0 consumes the dry-run report as one required readiness source. The RC checklist adds validation-gate evidence, allowed skipped integration policy, signoff expectations, release-note requirements, and rollback checklist metadata without creating a release, tag, GitHub release, deployment, or provider call.

## Relationship To Integration-Test Profiles

GitHub App integration-test profile v1, LLM Gateway integration-test profile v1, and Vault Integration-Test Profile v1 stay skipped by default. The dry-run classifies them as `skipped`, `gated`, `ready`, `blocked`, or `future` without executing them.

## Known Limitations

- Current default report is blocked/not ready because staging still lacks real secret backend readiness, real production auth, operational observability, and Postgres staging configuration by default.
- Observability, MCP, Git, LLM Gateway default mode, Dashboard, Local Agent, and Runner sources are deterministic read-only summaries rather than live service probes.
- The dry-run records validation command status only when supplied by tests or future callers; it does not execute validation commands.
