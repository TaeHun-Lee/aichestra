# Production Secret Backend Implementation Option Decision v0

Status: `v0_implemented`

This is a decision and planning milestone. It selects the first production-grade secret backend implementation path for Aichestra without implementing or calling that backend.

## Decision

Recommended first backend: `vault`, promoted from the v0 planning alias `vault_future` by Vault-backed Secret Backend v1.

Second choice: `aws_secrets_manager_future`, if the first production deployment is explicitly AWS-first.

Decision status: `accepted_mock`.

The v1 implementation path should add the production secret backend interface refinements plus one concrete Vault-backed provider behind existing `packages/security` SecretRef, Auth/RBAC, Policy, audit, and redaction boundaries.

## Why Vault First

Vault is preferred because Aichestra has not selected a cloud provider and needs a cloud-neutral, lease-oriented backend that can exercise the strongest SecretRef production contract first. Vault gives the best fit for short-lived access, revocation, namespaces, backend audit, self-hosted deployments, and future dynamic secret patterns.

At the time of the v0 decision, Vault was not implemented. Vault-backed Secret Backend v1 now implements a gated, non-default Vault SecretRef provider boundary. This still does not mean secrets are production-ready: Vault production rollout, service identity, HA, rotation, migration, and env fallback enforcement remain future work.

## What v0 Does Not Implement

- No Vault integration in the v0 decision milestone. Vault-backed Secret Backend v1 is documented separately in `docs/foundations/vault-secret-backend/v1.md`.
- No AWS Secrets Manager, GCP Secret Manager, Azure Key Vault, or custom backend integration.
- No backend calls.
- No secret reads, writes, migration, rotation, export, or credential issuance.
- No BYOK.
- No OAuth, device-code, WIF, IAM, or cloud identity token exchange.
- No vendor CLI or credential cache reads.
- No production Auth/RBAC implementation.
- No production readiness claim.

## Safety Guarantees

Default runtime remains safe and mock-first. The decision surface exposes only deterministic metadata, booleans, counts, ids, statuses, and docs references. It does not expose secret values, env values, token values, private keys, webhook secrets, database URLs, backend credential values, or credential cache paths.

## Known Limitations

- Vault operation ownership is not assigned.
- Production Auth/RBAC remains planning-only.
- Production policy bundle runtime remains planning-only.
- Rotation and incident response are planned, not implemented.
- Env fallback remains supported for local/integration compatibility but is not production-sufficient.

## Follow-up Status

Vault-backed Secret Backend v1 is implemented as a gated backend boundary with default runtime/tests still disabled or mock-only. The selected production backend is not deployed and production secret backend readiness remains false.

Vault Integration-Test Profile v1 is implemented as a skipped-by-default readiness surface for optional live KV v2 smoke validation against a test-only allowlisted path. Vault Live Integration Enablement v1 adds stricter live-readiness checks and a safe runbook for that optional path. These do not change the production decision status.
