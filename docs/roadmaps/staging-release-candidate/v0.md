# Staging Release Candidate Checklist v0

Status: `v0_implemented`

Staging Release Candidate Checklist v0 is a read-only readiness surface for deciding whether a commit or branch can be called a staging release candidate. It aggregates the staging deployment dry-run, staging profile, CI/CD planning, optional integration-test profiles, production readiness, DB operations, auth, secrets, policy bundles, observability, dashboard, Git, LLM, MCP, runner, and Local Agent readiness metadata.

It does not create a release, create a Git tag, create a GitHub release, deploy anything, run deployment commands, run remote integration tests, call external providers, read credential caches, issue sessions/tokens/JWTs, or expose secrets/env values.

## Implemented

- Checklist, gate, blocker, signoff, release-note requirement, rollback checklist, report, and summary read models.
- Deterministic evaluation rules for local validation gates, skipped optional integrations, production-only blockers, safety gates, signoffs, release notes, rollback planning, and known limitations.
- Read-only service methods in `packages/deployment-readiness`.
- DTO-backed read-only API endpoints under `/readiness/staging-rc/*`.
- Safe `/health` metadata for staging RC status, gate counts, blockers, signoffs, release notes, rollback counts, and safety booleans.
- Dashboard read model and web panel for staging RC readiness.
- Deterministic tests for models, evaluation rules, API endpoints, dashboard rendering, health metadata, no-secret/no-env exposure, no release, no deployment, and no external calls.
- Evidence Pack v0 documentation that records validation evidence, skipped optional test evidence, release-note draft evidence, rollback evidence, and signoff readiness without creating a release or deployment.

## Not Implemented

- Release creation, Git tag creation, or GitHub release creation.
- Staging or production deployment execution.
- Active CI/CD workflow creation.
- Production auth, production secret backend rollout, real policy bundle runtime, real MCP transport, vendor CLI execution, artifact registry, Kubernetes, Terraform, Helm, Pulumi, Temporal, Vault infrastructure, or cloud infrastructure.
- Live GitHub, LLM, MCP, external auth, or cloud-provider validation.

## Checklist Model

The checklist records:

- Required validation gates: `pnpm lint`, `pnpm typecheck`, `pnpm test`, `pnpm build`, `git diff --check`, safe integration scan, no-secret/no-env check, docs check, and dashboard/readiness check.
- Optional integration profiles: optional Postgres contracts, GitHub App live profile, LLM live profile, future MCP live profile, and future external auth profile.
- Allowed skipped tests and the reason each skip is acceptable for staging RC v0.
- Blocker policy, required signoff roles, release-note sections, rollback checklist items, known limitations, and safety metadata.

## Gate Categories

Gate categories are `validation`, `docs`, `staging_dry_run`, `ci_cd`, `git_integration`, `llm_integration`, `mcp`, `db`, `secrets`, `auth`, `policy`, `observability`, `dashboard`, `security`, and `rollback`.

Required gates that fail or are not checked prevent RC designation. Optional gates may be `skipped` when the skip is documented and the current staging RC criteria do not require live validation.

## Evaluation Rules

- `pnpm lint`, `pnpm typecheck`, `pnpm test`, `pnpm build`, and `git diff --check` must pass before RC designation.
- A validation command failure is a critical blocker.
- A no-secret/no-env exposure failure is a critical blocker.
- Remote merge, force push, branch deletion, vendor CLI execution, and real MCP transport are critical blockers.
- Release creation, Git tag creation, GitHub release creation, deployment execution, external calls, or staging/production-ready overclaims are critical blockers.
- Missing required docs, release notes, rollback items, signoffs, or unchecked required validation are high blockers.
- Missing production auth and missing real secret backend are production blockers, not staging RC blockers, when documented as limitations.
- Missing staging deployment execution is not a blocker for this checklist because v0 is pre-deployment readiness.

## Skipped Tests Policy

Skipped optional Postgres, GitHub, LLM, MCP, and external auth tests are allowed only when:

- the profile is optional for staging RC v0,
- the missing gates are documented,
- release notes list the skip and reason,
- no skipped profile is silently treated as passed,
- no live provider calls are made by the checklist.

## Signoffs

Expected signoff roles are `engineering_owner`, `platform_owner`, `security_reviewer`, `product_owner`, `qa_reviewer`, and `release_manager`.

Statuses are metadata-only: `pending`, `approved_mock`, `waived`, or `not_applicable`. v0 does not implement production identity, real approval enforcement, sessions, tokens, or durable approval workflow.

Evidence Pack v0 records signoff readiness in [signoff-readiness-v0.md](signoff-readiness-v0.md). It does not fake real human approval. Pending or planning-ready signoffs block `staging_rc_pass` and can support `staging_rc_pass_with_warnings` only if a future audit explicitly accepts that v0 limitation.

## Release Notes

Required sections are Summary, Changed areas, Validation, Skipped tests, Safety gates, Known limitations, Migration notes, Dashboard/readiness notes, Rollback notes, and Follow-ups. The checklist exposes section status and guidance; it does not publish notes or create a release.

Evidence Pack v0 fills a draft at [release-notes-draft-v0.md](release-notes-draft-v0.md). The draft is not a release artifact and does not create a Git tag, GitHub release, or deployment.

## Rollback Checklist

Rollback planning covers code revert, DB/migration considerations, config rollback, feature flags, Git/GitHub integration gates, LLM integration gates, SecretRef/env fallback, dashboard/readiness rollback, observability/audit review, and manual verification. v0 records the plan only and executes nothing.

Evidence Pack v0 records rollback planning evidence at [rollback-evidence-v0.md](rollback-evidence-v0.md). No destructive rollback command is run because no release or deployment happened.

- [evidence-pack-v0-plan.md](evidence-pack-v0-plan.md)

- `GET /readiness/staging-rc/gates`

- `GET /readiness/staging-rc/rollback`
- `GET /readiness/staging-rc/report`
- `GET /readiness/staging-rc/summary`

All endpoints are read-only. Non-GET methods return `405`.

## Dashboard

`/dashboard/staging-rc` exposes overall RC status, required gates, blockers, skipped optional tests, signoff status, release-note status, rollback checklist, recommended next actions, and safety flags. It returns no secrets, env values, tokens, raw logs, raw prompts, or raw provider output.

## Guarantees

- `productionReady = false`
- `stagingDeployed = false`
- `releaseCreated = false`
- `gitTagCreated = false`
- `githubReleaseCreated = false`
- `deploymentExecuted = false`
- `externalCallsEnabled = false`
- `remoteIntegrationTestsExecuted = false`
- no secret values returned
- no env values returned

## Known Limitations

- v0 records validation status; it does not execute validation commands itself.
- Signoffs are metadata-only until production Auth/RBAC and durable approval workflow exist.
- Release notes are requirements/template metadata; no release artifact is created.
- Optional live integrations remain skipped unless a separate gated validation task is requested.
- Vault Integration-Test Profile v1 is now one of those optional skipped profiles; its skip is acceptable for RC criteria only when documented and production readiness remains false.
- Production auth, real secret backend, durable observability, production deployment, and staging deployment execution remain future work.
