# Staging CI/CD Pipeline Planning v0

Status: v0_implemented
Scope: planning and read-only readiness model only
Production ready: no
Staging deployed: no

## What v0 Implements

Staging CI/CD Pipeline Planning v0 defines the validation model for local, pull request, integration, staging, and future release-candidate profiles. It adds:

- CI/CD job matrix planning.
- Optional integration test gate policy.
- Secret and env safety rules.
- Artifact/report policy.
- Staging promotion criteria in CI terms.
- Cleanup and rollback policy.
- Read-only `/readiness/ci-cd/*` APIs.
- Safe `/health` CI/CD metadata.
- Dashboard CI/CD readiness panel.

No workflow deploys anything. No external provider is called by default. No production-ready claim is made.

## What v0 Does Not Implement

- Live deployment.
- Active production GitHub Actions workflows.
- Remote integration tests enabled by default.
- Terraform, Pulumi, Helm, Kubernetes, cloud infrastructure, or production deployment code.
- Artifact registry publishing.
- Real secret backend, real Auth/RBAC, OIDC, SAML, SCIM, or SSO.
- Real MCP transport or vendor CLI execution.
- Cleanup workers that call remote services.

## CI Job Matrix

The canonical job matrix is [job-matrix-v0.md](job-matrix-v0.md).

Required baseline jobs:

- `pnpm install` when dependency metadata changes.
- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`
- `pnpm build`
- `git diff --check`

Safety jobs:

- safe integration scan
- secret exposure scan
- docs path consistency check
- no production-ready overclaim check
- Node/Volta version check
- dependency lockfile check
- dashboard no-secret smoke check
- health no-secret smoke check

Optional jobs remain disabled by default and require explicit gates.

## Integration Test Gates

The integration gate policy is [integration-test-gates-v0.md](integration-test-gates-v0.md).

Optional profiles:

- Postgres repository contracts.
- Remote Git/GitHub integration.
- GitHub App integration-test profile v1, skipped unless every explicit GitHub App live-test gate and allowlist is configured.
- GitHub webhook fixture/integration tests.
- LLM Gateway integration-test profile v1, skipped unless every explicit OpenAI-compatible live-test gate, model allowlist, budget cap, SecretRef/credential gate, safe prompt class, and fallback-disabled condition is configured.
- Vault Integration-Test Profile v1, skipped unless every explicit Vault live-test gate, non-production KV v2 test path, path allowlist, token-auth test gate, and no-write/no-delete/no-rotate/no-broad-list safety check is configured.
- Remote MCP, external auth, and vendor CLI tests as future/blocked.

Every remote profile requires non-production scope, explicit env gates, approvals, cleanup expectations, and no-secret assertions.

## Secret / Env Safety

The secret/env safety policy is [secret-env-safety-v0.md](secret-env-safety-v0.md).

CI/CD readiness APIs and dashboard panels expose only:

- booleans
- counts
- statuses
- command names
- env var names
- documentation references

They must not expose env values, secret values, DB URL values, private keys, webhook secrets, raw provider credentials, raw prompts/outputs, raw webhook payloads, or credential-cache paths.

## Artifact / Report Policy

The artifact policy is [artifact-report-policy-v0.md](artifact-report-policy-v0.md).

Allowed artifacts are redacted summaries only. Raw logs, raw env dumps, raw provider outputs, raw webhook payloads, raw prompts, credential-cache contents, and secret-bearing files must not be uploaded.

## Staging Promotion Criteria

The staging promotion policy is [staging-promotion-v0.md](staging-promotion-v0.md).

Promotion to staging requires:

- baseline validation jobs pass
- safety scans are reviewed
- staging readiness APIs show no secret/env exposure
- optional remote integration tests are either explicitly gated and passed or clearly skipped
- destructive Git, remote MCP, vendor CLI, deployment, and production traffic remain disabled

Promotion does not imply production deployment.

## Cleanup / Rollback

The cleanup and rollback policy is [cleanup-rollback-v0.md](cleanup-rollback-v0.md).

v0 documents manual cleanup and rollback expectations only. It does not call remote services or delete remote resources.

## Read-Only APIs

- `GET /readiness/ci-cd/profiles`
- `GET /readiness/ci-cd/jobs`
- `GET /readiness/ci-cd/integration-gates`
- `GET /readiness/ci-cd/checks`
- `GET /readiness/ci-cd/risks`
- `GET /readiness/ci-cd/summary`

All endpoints are read-only and do not execute CI jobs.

## Dashboard Panel

`/dashboard/ci-cd` shows:

- CI/CD readiness status
- job matrix summary
- required baseline jobs
- optional integration gates
- secret/env safety status
- staging promotion status
- artifact policy
- cleanup/rollback readiness
- risks and blockers
- no-secret/no-env status

Staging Deployment Dry-run Profile v0 consumes this CI/CD planning surface as one readiness source under `/readiness/staging-dry-run/*` and `/dashboard/staging-dry-run`. The dry-run does not run CI jobs or create workflows; it only reports whether CI/CD planning is sufficient to attempt a future staging validation.

Staging Release Candidate Checklist v0 also consumes this CI/CD planning surface. It requires local validation evidence to be recorded before RC designation, but the checklist itself does not create workflows, run jobs, upload artifacts, deploy, or call providers.

Staging Deployment Execution Plan v0 consumes this CI/CD planning surface as a pre-deployment gate source. It records validation expectations, optional integration decisions, and future smoke placeholders without creating workflows, running jobs, deploying, or calling providers.

## Known Limitations

- No live CI workflow is created.
- No staging deployment exists.
- Optional integration tests are modeled but not activated.
- Cleanup and rollback are manual/planning-only.
- Node version checks are read-only metadata; CI enforcement is future work.
- Staging remains blocked by production Auth/RBAC, real secret backend, and operational DB evidence.
