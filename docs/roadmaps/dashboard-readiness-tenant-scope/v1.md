# Dashboard/Readiness Tenant Scope Planning v1

Status:

- Dashboard/Readiness Tenant Scope Planning: `v1_implemented`
- Dashboard/Readiness Tenant Scope Implementation: `v1_implemented`
- Tenant Scope Enforcement: `v1_implemented_partial`

Dashboard/Readiness Tenant Scope Planning v1 defines the planning/readiness surface for future dashboard and readiness filtering by tenant, team, project, repo, provider, model, SecretRef, MCP tool, registry package, local-agent host, and audit query scope.

## What v1 Implements

- Dashboard panel inventory: `docs/reference/dashboard-tenant-scope-inventory.md`.
- Readiness endpoint inventory: `docs/reference/readiness-tenant-scope-inventory.md`.
- Dashboard role visibility matrix: `docs/reference/dashboard-role-visibility-matrix.md`.
- Readiness role visibility matrix: `docs/reference/readiness-role-visibility-matrix.md`.
- Scope fallback behavior: `scope-fallback-behavior-v1.md`.
- Future filtering architecture: `future-filtering-architecture-v1.md`.
- Deterministic read-only planning models and `DashboardReadinessTenantScopePlanningService`.
- Read-only planning endpoints:
  - `GET /readiness/tenant-scope/dashboard-plans`
  - `GET /readiness/tenant-scope/dashboard-scope-summaries`
  - `GET /readiness/tenant-scope/readiness-plans`
  - `GET /readiness/tenant-scope/readiness-scope-summaries`
  - `GET /readiness/tenant-scope/role-visibility`
  - `GET /readiness/tenant-scope/fallback-behavior`
  - `GET /readiness/tenant-scope/summary`
- Dashboard read model and panel:
  - `GET /dashboard/tenant-scope`
- Safe `/health` metadata for planning status.

Implementation v1 adds `ScopedReadModelMetadata`, `DashboardPanelScopeSummary`, and `ReadinessEndpointScopeSummary` derived from the planning inventory. Dashboard/readiness responses now expose safe required scopes, applied mock/readiness scope dimensions, missing-scope warnings, role visibility hints, redaction labels, and explicit `tenantFilteringImplemented: false` and `productionEnforcementImplemented: false` metadata.

Tenant Scope Enforcement v1 adds partial representative helper metadata through `TenantScopeEnforcementDecision`, enforcement modes, mismatch records, `scopeDecisionSummary`, `/readiness/tenant-enforcement/*`, and `/dashboard/tenant-enforcement`. It does not change the planning boundary: production tenant enforcement remains false.

## Role Visibility Matrix

The role matrices define target visibility for:

- `viewer`
- `developer`
- `reviewer`
- `security_admin`
- `platform_admin`
- `audit_reader`
- `release_manager`
- `service_account_runner`
- `system_admin_future`

Roles define visibility and redaction expectations only. They do not grant access in this milestone.

## Scope Fallback Behavior

Target behavior is:

- local/mock runtime: metadata-only display with warning;
- staging planning: display with warning and production blocker;
- production: deny scoped details or require explicit tenant/global admin scope;
- audit views: require explicit audit query scope before production;
- secret-adjacent views: always redact raw values.

## API and Dashboard Surface

The new API surface is read-only and deterministic. It returns planning records, role visibility rows, fallback behavior rows, and a summary with:

- dashboard panel count;
- readiness endpoint count;
- panels requiring tenant scope;
- endpoints requiring tenant scope;
- secret-adjacent surface count;
- audit-scoped surface count;
- production blocker count;
- `tenantFilteringImplemented: false`;
- `productionTenantEnforcement: false`;
- `productionEnforcementImplemented: false`;
- `productionReady: false`.

The dashboard panel displays those same values and states that enforcement is planning-only. Implementation v1 also displays `Scope / Visibility / Redaction` hints for applied scope dimensions, missing scope warnings, allowed role summaries, redaction classes, and no-secret/no-env status.

Tenant Scope Enforcement v1 adds read-only summaries:

- `GET /readiness/tenant-enforcement/modes`
- `GET /readiness/tenant-enforcement/mismatches`
- `GET /readiness/tenant-enforcement/summary`
- `GET /dashboard/tenant-enforcement`

These summaries are partial representative metadata only. They report `tenantFilteringImplemented: false`, `productionTenantEnforcement: false`, and no-secret/no-env status.

## No-secret / No-env Policy

The planning surface must never expose:

- raw secret values;
- environment variable values;
- provider API keys;
- GitHub tokens, installation tokens, private keys, or webhook secrets;
- Vault token values, secret values, or sensitive path/key values;
- cookies, sessions, JWTs, passwords, client secrets, or raw identity assertions;
- credential cache paths or contents;
- raw prompts or raw provider responses.

## Production Blockers

Production remains blocked until an explicit future implementation adds:

- production Auth/RBAC;
- durable tenant/scope repository design;
- tenant/team/project/repo/provider/model/package/tool/host/audit query filters;
- role redaction tests;
- cache partitioning;
- audit query filtering;
- security review for secret-adjacent and audit surfaces.

## Known Limitations

- Planning records are deterministic static metadata.
- Current dashboard/readiness responses are not tenant-filtered.
- Current scope catalog is mock/readiness metadata.
- Role matrices are target behavior, not active authorization.
- Production auth and tenant isolation are not implemented.
