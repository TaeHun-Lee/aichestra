# Persistent DB Production Operations v1

Status: `v1_implemented` as a non-destructive operations/readiness foundation. Persistent DB operations are not production-ready.

## What v1 Implements

- Production DB operations runbook and supporting plans.
- Read-only database deployment profiles, readiness checks, migration metadata, risk register, schema inventory, index review, retention/audit-growth plan, and webhook persistence plan in `@aichestra/deployment-readiness`.
- Safe local migration file inventory with deterministic SHA-256 checksums.
- Read-only API endpoints under `/readiness/database/*`.
- `/health` database operations metadata with booleans and counts only.
- Dashboard API-backed DB Operations panel and web rendering.
- Regression tests for no DB URL exposure, no destructive operations, migration metadata, API endpoints, and dashboard rendering.

## What v1 Does Not Implement

- No production database provisioning.
- No production database connection.
- No Terraform, Pulumi, Helm, Kubernetes, or cloud infrastructure.
- No backup jobs or restore execution.
- No automatic migration execution.
- No migration rollback SQL execution.
- No destructive retention deletion jobs.
- No live table partitioning jobs.
- No external DB monitoring/export integration.
- No real secret backend, SSO, SCIM, MCP, Kubernetes, Temporal, artifact registry, GitHub, LLM, or provider calls.

## Production Postgres Assumptions

Future production deployment should use managed or reviewed self-operated Postgres with:

- TLS-enforced network access;
- credentials supplied through a future real SecretRef backend, not raw env fallback;
- pooled API and worker connections;
- a separate migration job;
- automated encrypted backups and point-in-time recovery where available;
- restore drills in an isolated environment;
- query limits on dashboard/readiness/audit APIs;
- retention, legal hold, archive/export, and audit controls before production traffic.

## DB Profiles

- `local`: in-memory default, no Postgres required, no production traffic.
- `integration`: optional Postgres contract testing through `AICHESTRA_TEST_DATABASE_URL`; no production traffic.
- `staging`: Postgres, migrations, pooling, backup, restore drill, retention plan, and monitoring required; not implemented.
- `production`: all staging controls plus tenant scoping, production auth, real secret backend, observability, and reviewed operational runbooks; blocked.

## Migration Process

Current behavior:

- SQL migrations live under `infra/migrations/`.
- `scripts/db/migrate.mjs` runs only when explicitly invoked.
- `pnpm db:migrate` requires `AICHESTRA_DATABASE_URL` or `DATABASE_URL`.
- app startup, build, and tests do not run migrations automatically.

Future production process:

1. Confirm application release and migration bundle checksum.
2. Verify latest backup and rollback compatibility window.
3. Acquire migration lock.
4. Run migration in a dedicated job.
5. Run post-migration smoke checks.
6. Record migration audit metadata.
7. Release API/worker only after migration success.

## Migration Rollback Strategy

- Prefer forward fixes for non-destructive schema mistakes.
- Keep pre-migration backup available.
- Define app rollback compatibility before rollout.
- Rehearse restore-based rollback in staging.
- Require review for any manual rollback SQL.
- Never run destructive rollback automatically.

## Backup Strategy

See `docs/roadmaps/persistent-db-production-operations/backup-restore-runbook-v1.md`.

Production requires encrypted backups, point-in-time recovery where available, retention by data class, restore drills, and audit data restore checks. v1 creates no backup job.

## Restore Test Strategy

Restore into an isolated non-production database, run schema verification, run repository contract tests, verify audit table readability, verify no secret material exists in metadata, and record drill evidence. v1 does not execute restore tests automatically.

## Connection Pooling Plan

See `docs/roadmaps/persistent-db-production-operations/connection-pooling-v1.md`.

Persistent DB v1 currently uses a `psql` CLI process per query to preserve synchronous repository contracts. Production requires an async pooled Node Postgres client behind the existing `StorageProvider` and repository boundaries.

## Index Review Plan

See `docs/roadmaps/persistent-db-production-operations/index-review-v1.md`.

The current schema has core indexes for tasks, task runs, Git webhook metadata, PR/branch sync state, registry, security, policy, and local-agent skeletons. Production still needs review for time-based queries, high-growth audit/event tables, webhook replay/dedupe uniqueness, LLM/MCP/Local Agent future tables, and common observability audit envelopes.

## Retention-Aware Table Strategy

See `docs/roadmaps/persistent-db-production-operations/retention-and-audit-growth-v1.md`.

v1 aligns table planning with Observability / Audit Retention v0 classes:

- `short_debug`
- `operational`
- `security`
- `compliance`
- `ephemeral`

No deletion, archive, export, partition, or legal hold implementation is added in v1.

## Audit/Event Table Growth Plan

High-growth candidates include `audit_events`, `git_webhook_events`, `git_webhook_audit_events`, `usage_ledger_entries`, future common `observability_audit_events`, LLM audit/routing records, MCP invocation/audit records, security audit, policy audit, and Local Agent stream/audit tables.

Future work should add time-based indexes, query limits, partitions, archive/export checkpoints, legal hold, and retention enforcement after production identity and secret backend controls exist.

## Webhook Replay/Dedupe/Dead-Letter Persistence

See `docs/roadmaps/persistent-db-production-operations/webhook-persistence-v1.md`.

Current Git webhook tables store delivery metadata, payload hashes, verification results, sync states, and audit. Production needs durable uniqueness for delivery id, payload hash replay classification, retry counters, and sanitized dead-letter records. Raw webhook payloads remain out of scope.

## Production DB Health/Readiness Checks

v1 exposes read-only data:

- storage provider kind;
- database URL configured boolean only;
- optional Postgres test URL configured boolean only;
- migration runner availability;
- migration file count;
- pooling status;
- backup/restore status;
- retention deletion disabled;
- production DB connection attempted false.

## Optional Postgres Test Profile

Default tests do not require Postgres. Optional contract tests run only when `AICHESTRA_TEST_DATABASE_URL` is configured. That URL must point to a local/test database, never production.

## Known Limitations

- DB readiness is deterministic/read-only planning data, not live infrastructure probing.
- Migration status reports local files as pending/readiness metadata; it does not inspect production migration history.
- The current Postgres runtime path is not production-grade because it uses `psql` per query.
- No backup/restore job exists.
- No pooler or async DB driver exists.
- No retention deletion/legal hold/archive/export exists.
- No durable common audit envelope table exists.
- No durable GitHub App replay/dead-letter table implementation exists.
- Tenant scoping and production Auth/RBAC remain blockers.
