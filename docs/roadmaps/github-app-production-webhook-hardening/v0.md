# GitHub App / Production Webhook Hardening Planning v0

Status: `v0_implemented` planning/read-model foundation. GitHub App integration is not implemented and webhook hardening is not production-ready.

## What v0 Implements

- Planning docs for GitHub App credentials, production webhook endpoint hardening, replay protection, and retry/dead-letter handling.
- `docs/reference/github-app-permission-matrix.md`.
- `docs/reference/github-webhook-event-allowlist.md`.
- Read-only GitHub App / webhook hardening planning models in `packages/deployment-readiness`.
- Read-only API endpoints under `/readiness/github-app/*`.
- Dashboard GitHub App / Webhook Hardening panel.
- Planned observability metric definitions for GitHub webhook and sync operations.
- Deterministic tests for permission matrix, event allowlist, replay classification, API read-only behavior, dashboard rendering, and no-secret exposure.

## What v0 Does Not Implement

- No GitHub App creation or installation flow.
- No GitHub App private key generation, parsing, signing, or storage.
- No installation access token generation.
- No real GitHub API calls.
- No remote Git operations.
- No production webhook enablement.
- No merge, rebase push, force push, branch deletion, workflow dispatch, deployment action, GitLab, or Bitbucket.
- No LLM Gateway, MCP, Vault, Kubernetes, Temporal, artifact registry, vendor CLI, or credential cache integration.

## Target Architecture

Future production GitHub integration should use:

- GitHub App descriptor metadata;
- installation records scoped to account and repository selection;
- repository grants with least-privilege permission sets;
- SecretRef-backed private key and webhook secret references;
- short-lived installation tokens resolved inside a narrow credential boundary;
- `GitHubClient` as the only GitHub HTTP boundary;
- Auth/RBAC, Policy-as-code, repo allowlist, SecretRef, and audit checks before every operation.

## Installation Model

`GitHubAppDescriptor` models the planned app. `GitHubAppInstallation` models account/installation metadata. `GitHubAppRepositoryGrant` models selected repository permissions. These are read models and must not contain raw secrets.

## Permission Matrix

The matrix denies workflows, administration, secrets, and deployments by default. Contents write and pull request write are future branch/PR capabilities only; they do not imply merge.

See `docs/reference/github-app-permission-matrix.md`.

## Webhook Event Allowlist

Current supported/read-model events include `ping`, `pull_request`, `push`, `check_run`, `check_suite`, `status`, and `pull_request_review`. Installation and repository events are planned. `workflow_run` is ignored and `deployment` is denied by default.

See `docs/reference/github-webhook-event-allowlist.md`.

## Replay Protection

Delivery id is the idempotency key. Payload hash detects replay mismatches. Same delivery id and same payload hash is duplicate/ignored; same delivery id and different hash is replay rejected. Production needs durable shared storage.

See `docs/roadmaps/github-app-production-webhook-hardening/webhook-replay-protection-v0.md`.

## Retry And Dead Letter

v0 models retryable/non-retryable classes and dead-letter records. No background retry worker exists. Production needs queue-backed processing, dead-letter review, and alerting.

See `docs/roadmaps/github-app-production-webhook-hardening/webhook-retry-dead-letter-v0.md`.

## Credential / SecretRef Plan

GitHub App private keys and webhook secrets must be stored in a future real secret backend behind SecretRef. Legacy env tokens are not production-ready. v0 does not sign JWTs or exchange installation tokens.

See `docs/roadmaps/github-app-production-webhook-hardening/github-app-credentials-v0.md`.

## Production Endpoint Plan

Target endpoint path: `/git/github/webhooks`. Production requires TLS, raw-body preservation, payload and rate limits, durable idempotency, and queue-backed processing before enabling traffic.

See `docs/roadmaps/github-app-production-webhook-hardening/production-webhook-endpoint-v0.md`.

## Dashboard Panel

`/dashboard/github-app` and the web dashboard show:

- planning status;
- permission matrix summary;
- webhook event allowlist summary;
- replay protection status;
- dead-letter status;
- credential readiness;
- production endpoint readiness;
- blockers and risks;
- no-secret status.

## Observability And Audit Requirements

Planned metrics include webhook deliveries received/verified/rejected, duplicate deliveries, dead-letter count, PR sync success/failure, branch sync success/failure, GitHub API rate-limit warnings, and webhook processing latency.

Planned audit events include:

- `github_app_installation_created_future`
- `github_app_installation_suspended`
- `github_app_repo_grant_changed`
- `github_webhook_duplicate_rejected`
- `github_webhook_dead_lettered`
- `github_installation_token_requested_future`
- `github_installation_token_issued_future_metadata_only`

No audit event or read model may contain private keys, webhook secrets, installation tokens, PATs, raw payloads, or credential cache contents.

## Test Strategy

- Permission matrix enforces least privilege and denies workflow/admin/secrets/deployment permissions.
- Event allowlist requires signatures and read-model-only side effects.
- Replay classification covers first seen, duplicate, and replay rejected.
- API endpoints are read-only and sanitized.
- Dashboard panel renders without secrets.
- Regression suites keep Real Git Adapter, SecretRef, Observability, and Dashboard behavior intact.

## Known Limitations

- No durable replay store.
- No production queue or retry worker.
- No production webhook endpoint deployment.
- No GitHub App private-key signing or installation-token exchange.
- No production observability/export/alerting.
- Production Auth/RBAC, secret backend, policy bundle management, tenant isolation, and production DB operations remain blockers.
