# Staging Deployment Profile v0

Status: v0_implemented
Scope: planning and read-only readiness model only
Production ready: no

## What v0 implements

Staging Deployment Profile v0 defines a non-production validation profile for Aichestra. It adds deterministic staging readiness models, read-only API and dashboard visibility, and documentation for environment gates, promotion criteria, rollback criteria, and integration-test policy.

No deployment is created. No infrastructure code is added. No external provider is contacted.

## What staging is for

Staging is a controlled validation environment between local/integration and production. It is intended to validate:

- Postgres-backed persistence readiness without exposing database URL values.
- API-backed dashboard read models.
- GitHub App controlled boundary metadata and mock token-handle behavior.
- GitHub webhook signature/replay/dead-letter readiness assumptions.
- LLM Gateway remote-call gates without requiring remote calls by default.
- Auth/RBAC, SecretRef, Policy, Observability, Runner, MCP, and Local Agent warnings before production.
- CI validation and optional gated integration-test profiles.

## What staging is not for

Staging v0 is not a production deployment and is not a live provider rollout. It does not enable:

- Production traffic.
- Real production Auth/RBAC, OIDC, SAML, SCIM, SSO, sessions, or JWT issuance.
- Real Vault/cloud secret backend integration.
- Remote MCP transport.
- Vendor CLI execution.
- Remote merge, rebase push, force push, branch deletion, workflow/admin/secrets/deployment GitHub permissions, or webhook-triggered agent execution.
- External observability export, SIEM, alert delivery, or retention deletion jobs.

## Required Runtime Components

The staging profile expects the same control-plane components as production planning, but in a non-production deployment:

- API service.
- Web dashboard.
- Worker.
- Explicit migration job.
- Background job boundary for future retry/dead-letter work.

v0 records these as readiness data only.

## Deployment Topology Expectation

The expected staging topology is a single non-production environment with isolated state and test credentials. It should use API-backed dashboard reads and Postgres persistence. It must not share production secrets, production data, production webhook endpoints, or production identity-provider configuration.

## Required Env Gates

Staging requires or strongly recommends:

- `AICHESTRA_STORAGE_PROVIDER=postgres`
- `AICHESTRA_DATABASE_URL` configured through safe deployment config, never exposed in API/health/dashboard responses.
- `AICHESTRA_DASHBOARD_DATA_SOURCE=api`

Additional integration gates are documented in [staging-environment-gate-matrix.md](../../reference/staging-environment-gate-matrix.md).

## Forbidden Env Gates

Staging v0 forbids:

- `AICHESTRA_ALLOW_REMOTE_MERGE`
- `AICHESTRA_ALLOW_LOCAL_COMMAND_EXECUTION`
- `AICHESTRA_ENABLE_MCP_REAL_TRANSPORT`
- raw GitHub App private key env values such as `GITHUB_APP_PRIVATE_KEY`

Legacy env credential fallback is discouraged for staging and should be blocked except controlled integration-test carve-outs.

## Integration Policy

Staging integrations are classified as:

- Gated: Postgres, GitHub App controlled boundary, GitHub webhook receive/sync, remote LLM tests, Local Agent fixture protocol.
- Blocked: remote MCP transport, external observability export.
- Future: production auth provider, real secret backend, policy bundle runtime.

All live provider tests require explicit gates and must be skipped by default.

## GitHub App Staging Policy

GitHub App Controlled Implementation v1 may be represented in staging as metadata/readiness only. Mock token-handle issuance is allowed in tests. GitHub App integration-test profile v1 adds a skipped-by-default live-test readiness surface for config, installation-token, branch, PR, changed-file, webhook fixture, and cleanup checks. Live installation-token exchange remains disabled unless every explicit integration-test gate is configured and must not run by default.

Remote branch/PR behavior remains gated by Real Git Adapter controls, Auth/RBAC, Policy, SecretRef, repo allowlists, branch prefix checks, and no-destructive-operation rules.

## LLM Staging Policy

Remote LLM completion remains disabled by default. LLM Gateway integration-test profile v1 adds a skipped-by-default live-test readiness surface for the OpenAI-compatible remote path. Live remote completion remains disabled unless every explicit integration-test gate is configured and must not run by default.

- `AICHESTRA_LLM_INTEGRATION_TESTS=true`
- `AICHESTRA_LLM_PROVIDER=openai_compatible`
- `AICHESTRA_ENABLE_REMOTE_LLM=true`
- `AICHESTRA_ALLOW_REMOTE_LLM_COMPLETION=true`
- SecretRef-backed API key configuration preferred.
- Model allowlist/default model and budget cap.
- Safe prompt class and prompt/output redaction checks.
- Fallback disabled by default.
- Auth/RBAC and Policy allow decisions.

BYOK, OAuth/device-code, WIF/IAM, streaming, tool calling, provider credential cache reads, raw provider response storage, and vendor CLI providers remain out of scope.

## MCP Staging Policy

Remote MCP is blocked in staging v0. MCP Gateway remains mock-first. High/critical tools, write/deploy tools, secret forwarding, real transport, and network access remain disabled.

## Local Agent Staging Policy

Local Agent Protocol remains mock/fixture-only. Vendor CLI execution, credential cache reads, PTY automation, real daemon transport, and network tunnels are forbidden.

## Secret Backend Staging Policy

SecretRef metadata and env-backed provider boundaries exist, but a real secret backend is not implemented. A real secret backend remains a blocker for production and for meaningful staging provider validation.

## Auth/RBAC Staging Policy

Production Auth/RBAC v1 is planning-only. Staging must surface mock actor warnings. Real IdP, sessions, JWT issuance, login/logout, password login, SCIM, and service-account credential issuance are not implemented.

## Policy Bundle Staging Policy

StaticPolicyEngine remains the runtime. Policy Bundle / OPA-Cedar Planning v0 is planning-only. No OPA/Rego, Cedar, signed bundle runtime, dynamic policy execution, remote policy loading, or hot reload is enabled.

## DB Staging Policy

Postgres is required or strongly recommended for staging. v0 does not connect to a production DB, run migrations automatically, run backup jobs, run restore jobs, delete retention data, or add live partitioning jobs.

## Observability Staging Policy

Observability / Audit Retention v0 supplies in-memory/read-only taxonomy, retention classes, redaction classes, metrics skeletons, traces skeletons, and dashboard panels. External export, alerting, SIEM integration, and retention deletion remain blocked.

## Dashboard Staging Policy

The dashboard must use API-backed sanitized read models for staging. Demo fallback should be disabled in future deployment config. No secrets, tokens, credential values, database URL values, raw webhook payloads, raw prompts, or raw outputs may appear.

## CI/CD Validation Expectations

Promotion to staging requires:

- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`
- `pnpm build`
- `git diff --check`

Optional Postgres and provider integration tests must skip unless every explicit environment gate is configured.

## API And Dashboard

Read-only endpoints:

- `GET /readiness/staging/profile`
- `GET /readiness/staging/gates`
- `GET /readiness/staging/checks`
- `GET /readiness/staging/promotion-criteria`
- `GET /readiness/staging/rollback-criteria`
- `GET /readiness/staging/summary`

The dashboard includes a Staging Deployment Profile panel showing profile status, integration gates, readiness checks, promotion criteria, rollback criteria, blockers, warnings, mock actor warning, env fallback warning, and no-secret/no-env status. Staging CI/CD Pipeline Planning v0 adds a separate `/dashboard/ci-cd` panel for job matrix, optional integration-test gates, artifact/report policy, staging promotion, cleanup/rollback, and no-secret/no-env CI/CD status. GitHub App integration-test profile v1 adds `/dashboard/github-app-integration` for gate counts, unsafe gate warnings, test-case status, cleanup policy, and no-secret/no-env status. LLM Gateway integration-test profile v1 adds `/dashboard/llm-integration` for gate counts, model allowlist count, budget status, fallback safety, test-case status, and no-secret/no-env/no-raw-response status.

## Known Limitations

- Staging is not deployed.
- Postgres is planned/required but not provisioned.
- Real secret backend is missing.
- Real Auth/RBAC is missing.
- Policy bundle runtime is missing.
- External observability export is missing.
- Live GitHub App token exchange is not enabled.
- Remote MCP remains blocked.
