## Staging Rules

The detailed staging-specific matrix lives in `docs/reference/staging-environment-gate-matrix.md`.

- `AICHESTRA_STORAGE_PROVIDER=postgres` is required or strongly recommended for meaningful staging validation.
- `AICHESTRA_DATABASE_URL` may be configured only through safe deployment config and must be exposed as a boolean/status only.
- `AICHESTRA_DASHBOARD_DATA_SOURCE=api` is required for staging; demo fallback should be disabled in future deployment config.
- `AICHESTRA_ALLOW_REMOTE_MERGE`, remote rebase/push/force-push/delete behavior, real MCP transport, and vendor CLI execution are forbidden.
- GitHub App, webhook, remote Git, and remote LLM integration tests remain gated and skipped by default.
- GitHub App integration-test profile v1 requires GitHub App auth mode, app id metadata, private-key SecretRef metadata, installation/repo allowlists, `AICHESTRA_GITHUB_APP_ALLOWED_BRANCH_PREFIX=ai/`, branch/PR gates, and `AICHESTRA_ALLOW_REMOTE_MERGE=false`; readiness output exposes gate names/counts/status only.
- LLM Gateway integration-test profile v1 requires OpenAI-compatible provider mode, remote LLM/completion gates, base URL configured boolean, SecretRef-preferred API key metadata, model allowlist/default model, routing mode, fallback disabled, small budget cap, and safe prompt class; readiness output exposes gate names/counts/status only.
- Staging Deployment Dry-run Profile v0 exposes a read-only aggregate report under `/readiness/staging-dry-run/*` and `/dashboard/staging-dry-run`; it never deploys, runs CI jobs, executes remote integration profiles, calls providers, exposes secrets/env values, or marks staging/production ready.
- Staging Release Candidate Checklist v0 exposes a read-only checklist/report under `/readiness/staging-rc/*` and `/dashboard/staging-rc`; it never creates releases, Git tags, GitHub releases, deployments, CI jobs, remote integration runs, provider calls, secret/env values, staging-deployed claims, or production-ready claims.
- Staging Deployment Execution Plan v0 exposes read-only planning and Staging Human Signoff Pack v0 pending metadata under `/readiness/staging-execution/*` and `/dashboard/staging-execution`; it never executes deployments, creates releases or tags, runs deployment commands, records fake approval, calls providers, runs remote integration tests, exposes secrets/env values, or marks staging deployed/production ready.
- Env secret provider fallback is discouraged or blocked except tightly controlled integration-test carve-outs.
- Staging v0 must not deploy anything, call external services by default, expose secrets/env values, or mark production ready.

## CI/CD Rules

The detailed CI/CD planning surface lives in `docs/roadmaps/staging-ci-cd-pipeline/`.

- Staging CI/CD Pipeline Planning v0 documents pipeline profile, job, integration gate, artifact/report, promotion, and cleanup/rollback metadata only.
- No active GitHub Actions workflow, deployment job, provider call, vendor CLI call, or default remote integration test execution is created by v0.
- Optional Postgres, remote Git/GitHub App/webhook, remote LLM, remote MCP, external auth, and vendor CLI profiles are disabled by default.
- Staging RC validation may require local command results as evidence, but the RC checklist read model does not run those commands or create release artifacts.
- GitHub App integration-test profile v1 may report missing and unsafe gates but must not print env values, private keys, webhook secrets, installation tokens, or legacy PATs.
- LLM Gateway integration-test profile v1 may report missing and unsafe gates but must not print env values, API keys, base URL values, raw prompts, raw provider responses, vendor CLI state, or credential cache paths.
- CI/CD health and dashboard metadata may expose env var names, booleans, counts, and statuses only; they must not expose env values, secret values, raw logs, raw prompts, or raw provider outputs.

## Production Rules

- `AICHESTRA_STORAGE_PROVIDER=postgres` is required.
- `AICHESTRA_DASHBOARD_DATA_SOURCE=api` is required.
- Persistent DB Production Operations v1 readiness may expose database URL configured booleans and migration file counts/checksums, never DB URL values.
- Secret Backend Migration Planning v0 readiness may expose backend options, migration phases, readiness checks, risks, lease/rotation expectations, SecretRef counts, and env fallback booleans/counts/status only; it must never expose env values or secret values.
- Production Secret Backend Implementation Option Decision v0 may expose selected backend `vault`/`vault_future`, second choice `aws_secrets_manager_future`, criteria, scores, implementation scope, provider mapping, risks, and no-secret/no-env flags only.
- Vault-backed Secret Backend v1 may expose selected backend, enabled/configured booleans, auth method, path allowlist count, client kind/status, readiness checks, and sanitized audit only. It must never call Vault in default runtime/tests, expose Vault token/address/secret values, list broad Vault paths, migrate/rotate/read secrets outside explicit credential resolution, issue production credentials, or mark a production secret backend implemented.
- Vault Integration-Test Profile v1 may expose live-test enabled booleans, missing/unsafe gate counts, backend selected boolean, provider enabled boolean, address/token/mount/path/key configured booleans, path prefix count, test-path allowlisted/test-only booleans, safety checks, and no-secret/no-env booleans only. It must never expose Vault token/address/path/key/secret/env values, write/delete/rotate secrets, list broad paths, call Vault in default runtime/tests, read credential caches, or mark Vault production-ready.
- Vault Live Integration Enablement v1 may expose live-readiness status, validation-check status, runbook steps, run-record metadata, gate names/counts, configured booleans, test path allowlisted/safe booleans, and no-write/no-delete/no-rotate/no-broad-list booleans only. It must never expose Vault token/address/path/key/secret/env values, execute live validation by default, write/delete/rotate/list secrets, read credential caches, or mark production Vault rollout implemented.
- Production Auth/RBAC v1 Planning readiness may expose provider options, migration phases, readiness checks, risks, tenant/service-account plans, permission matrix metadata, mock actor warning, production auth disabled status, and no-token/no-session booleans only; it must never expose tokens, cookies, session ids, passwords, raw identity assertions, IdP client secrets, provider credentials, or env values.
- Production Auth Provider Skeleton v1 readiness may expose selected/active provider kind, disabled future provider statuses, configured booleans, missing config names, blockers, session/token boundary status, identity mapping status, production auth enabled false, token validation enabled false, and no-token/no-session/no-env booleans only; it must never call IdPs, validate tokens, parse cookies, store Authorization headers, issue sessions/JWTs/API keys/service-account credentials, run SCIM sync, expose env values/secrets, or mark production auth enabled.
- Policy Bundle / OPA-Cedar Planning v0 readiness may expose engine options, bundle plans, domain mappings, readiness checks, risks, migration phases, static policy rule counts, and no-dynamic-execution booleans only; it must never execute policy code, load remote bundles, call external policy engines, or expose secrets.
- Policy Runtime Shadow Evaluation Planning v1 readiness may expose plan, comparison rules, mismatch taxonomy, readiness checks, planning reports, counts, rollout stages, and no-execution booleans only; it must never execute candidate runtimes, OPA/Rego, Cedar, signed bundle verification, dynamic policy code, remote bundles, external policy services, or expose secrets/env values.
- GitHub App Controlled Implementation v1 may expose auth mode, app-enabled/configured status, app id configured booleans, private-key/webhook SecretRef configured booleans, installation/repo counts, token provider kind, and mock token-handle status only. It must never expose private keys, installation tokens, webhook secrets, legacy PATs, or raw credential values, and default runtime/tests must not call GitHub.
- LLM Gateway integration-test profile v1 may expose live-test enabled booleans, missing/unsafe gate counts, provider kind, base URL configured boolean, API-key configured boolean, SecretRef configured boolean, model allowlist count, budget configured boolean, fallback safety, and no-secret/no-env/no-raw-response booleans only. It must never expose API keys, env values, raw prompts, raw provider responses, vendor CLI state, or credential cache paths, and default runtime/tests must not call an LLM provider.
- Local CLI Provider Templates v1 may expose template ids, provider ids, compatibility rules, parser profile names, security constraints, and readiness booleans only. It must never expose executable paths, credential cache paths, env values, secret values, vendor CLI state, or mark vendor CLI execution implemented.
- Production pooling, backup/restore, migration governance, retention/legal hold, durable webhook replay/dead-letter persistence, and tenant scoping remain required before production.
- Mock auth, legacy env credentials, env secret fallback, real MCP transport without allowlists, unbounded LLM fallback, local command execution, and vendor CLI credential cache reads must be rejected.
- Health and dashboard may expose booleans, counts, ids, and statuses only.

## OIDC future-provider gates

- `AICHESTRA_AUTH_PROVIDER=oidc_future`: selects disabled future metadata only; must fail closed without production auth.
- `AICHESTRA_ENABLE_PRODUCTION_AUTH=false`: required default.
- OIDC issuer/audience/client-id/JWKS/discovery/scopes/group/tenant placeholders are represented as configured booleans only. Values and client secrets are not exposed or used.
- External calls, discovery fetch, JWKS fetch, token validation, and session issuance remain disabled.
