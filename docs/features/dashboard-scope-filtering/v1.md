# Dashboard Scope Filtering v1

Status: v1_implemented_partial

Dashboard Scope Filtering v1 is a representative, mock-first dashboard filtering layer. It evaluates the existing `DashboardReadinessTenantScopePlanningService` inventory against a role/scope filter context, produces per-panel `DashboardPanelFilterDecision` records, redacts representative sensitive panels, hides panels for callers without the required role, and exposes a `DashboardScopeFilterReadModel` to API/dashboard consumers. It does **not** implement production tenant isolation, production Auth/RBAC, row-level security, OIDC/SAML/SCIM, sessions, JWT, API keys, or any external identity provider.

## What v1 Implements

- Models in `packages/deployment-readiness/src/dashboard-scope-filtering.ts`:
  - `DashboardScopeFilterContext` (roles, tenant/team/project IDs, resource scopes, source, authMode).
  - `DashboardPanelFilterDecision` (`visible | redacted | hidden | warning_only | not_applicable`, requiredRoles, matchedRoles, requiredScopes, missingScopes, sensitivity, redactedFields, redactionClass, `productionFiltering: false`).
  - `DashboardScopeFilterSummary` (`status: v1_implemented_partial`, panel counts, roleApplied, rolesApplied, `productionFiltering: false`, `productionAuthImplemented: false`, `productionTenantEnforcement: false`, `noSecretsExposed: true`, `envValuesExposed: false`, `externalIdentityProviderCalled: false`).
- `DashboardScopeFilteringService` with `buildFilterContext`, `evaluatePanel`, `evaluatePanels`, `summarizeDecisions`, `panelKeyForReadModel`, `redactedFieldHintsFor`, `redactPanelBody`, `filterPanelBody`.
- Safe-only demo header parsing (`parseDashboardFilterHeaders`) that ignores Authorization / cookie / session / JWT / API-key headers entirely and only honors `x-aichestra-demo-role`, `x-aichestra-demo-tenant-id`, `x-aichestra-demo-team-id`, `x-aichestra-demo-project-id`, `x-aichestra-demo-resource-scope`.
- API wiring in `apps/api/src/main.ts` so `GET /dashboard/*` extracts the safe-only headers, builds a filter context, and re-uses it when constructing the dashboard read models.
- New `GET /dashboard/scope-filter` endpoint returning `DashboardScopeFilterReadModel`.
- Read-model assembly in `apps/api/src/dashboard-read-model.ts`: `buildDashboardScopeFilter` runs the filter against the planning service's inventory; representative sensitive panels (`security`, `vaultSecretBackend`, `observability`, `audit`, `githubApp`, `githubAppIntegration`, `llmIntegration`, `mcp`, `staging`, `stagingDryRun`, `stagingReleaseCandidate`, `stagingExecution`) are passed through `filterPanelBody` to apply redaction.
- Shared type `DashboardScopeFilterReadModel` (in `packages/shared`) plus endpoint listing `/dashboard/scope-filter`.
- Dashboard HTML panel ("Dashboard Scope Filtering") in `apps/web/src/render.ts` showing decision counts, redacted/hidden/warning panels, filter context summary, and explicit `production filtering false` flag.
- Tenant-scope plan entry already exists (`tenant_scope_enforcement`); no new tenant-scope plans were added because the filter status is itself the enforcement metadata surface.
- Deterministic tests in `tests/dashboard-scope-filtering-v1.test.ts` (16 tests) covering viewer/developer/security_admin/audit_reader/platform_admin/service_account_runner behaviors, missing-scope warn/redact, secret-adjacent redaction, header parsing safety, API endpoint, dashboard renderer, and the `productionFiltering: false` guarantee.

## Filter Context

`DashboardScopeFilterContext` is built from either:

1. The mock-actor `AuthContext` (default fallback). The service detects this default-mock-admin fallback and treats every panel as `visible` so existing demo behavior is unchanged.
2. Safe demo headers (`x-aichestra-demo-role`, `x-aichestra-demo-tenant-id`, `x-aichestra-demo-team-id`, `x-aichestra-demo-project-id`, `x-aichestra-demo-resource-scope`). These headers are **not auth** — they exist only to exercise the filter. The service records `authMode: "demo_request_header"` and ignores every unknown role.
3. A test fixture (`source: "test_fixture"`) used by deterministic tests.

Filter context never reads `Authorization`, `cookie`, `x-api-key`, `session`, JWT, or any production auth header. There is no session, no token storage, no identity provider call.

## Panel Decision Model

For each panel `summary`:

1. If panel ∈ `{overview, tasks, scope_model, tenant_scope_planning, tenant_scope_enforcement, request_context}` ⇒ `visible` (safety meta-panel).
2. Default mock-admin fallback ⇒ `visible` (preserves existing demo behavior).
3. Service-account-only role ⇒ `hidden` for human panels, `warning_only` for `overview`/`tasks`.
4. `allowedRoles ∩ context.roles == ∅`:
   - sensitive (`secret_adjacent` / `never_store_raw`) ⇒ `hidden`,
   - else ⇒ `warning_only`.
5. Required scopes missing:
   - sensitive ⇒ `redacted`,
   - else ⇒ `warning_only`.
6. Sensitive panel and the matched role is not `{security_admin, platform_admin, audit_reader, system_admin_future}` ⇒ `redacted`.
7. Otherwise ⇒ `visible`.

Redaction replaces `apiKey`, `secret`, `secretValue`, `tokenValue`, `rawSecret`, `rawValue`, `envValue`, `envValues`, `secretRefValue`, `vaultToken`, `credential`, `credentials`, `authorization`, and the panel-class-specific hint fields with the literal `"[redacted]"`. The panel body retains its non-sensitive summary plus a `filterDecision` metadata block.

## Role Behavior

| Role | Behavior |
| --- | --- |
| `viewer` | overview/tasks/scope panels visible; secret-adjacent panels hidden or redacted; sensitive panels warning_only |
| `developer` | tasks/git/llm summaries visible when scoped to tenant/team/project; secret-adjacent panels redacted |
| `reviewer` | registry/approvals/staging RC visible; GitHub app/llm integration redacted |
| `security_admin` | SecretRef/vault/auth/policy/audit visible (still passes through SECRET_LIKE field redaction in `redactPanelBody`) |
| `platform_admin` | operational readiness + staging + cicd visible; never raw secrets |
| `audit_reader` | audit summaries visible; non-audit panels warning_only |
| `release_manager` | staging + deployment profiles visible; auth internals redacted |
| `service_account_runner` | most human panels hidden, overview/tasks warning_only — attribution surfaces only |
| `system_admin_future` | future-only role; v1 behaves like a permissive admin under explicit header opt-in |

## Production Gaps

- Production Auth/RBAC is **not** implemented; the filter context is mock-only.
- Tenant filtering at the database / row-level is **not** implemented.
- No real OIDC/SAML/SCIM/session/JWT/API-key handling exists.
- No external identity provider is called.
- Readiness endpoints (`/readiness/*`) are not yet filtered.
- Audit query scope is represented through Audit Query Scope Enforcement v1 read-model redaction/check metadata. Production audit query filtering is still not implemented.

`DashboardScopeFilterSummary.productionFiltering`, `.productionAuthImplemented`, `.productionTenantEnforcement`, and `.externalIdentityProviderCalled` are typed as the literal `false`. `noSecretsExposed` and `envValuesExposed` are typed `true` / `false` respectively.

## API / Dashboard Behavior

- `GET /dashboard/scope-filter` returns `{ dashboardScopeFilter: DashboardScopeFilterReadModel }` with the filter applied to the request's safe-only header context.
- Other `GET /dashboard/*` endpoints continue to return their panel body but the representative panel bodies pass through `filterPanelBody` when a filter context is present.
- All dashboard endpoints remain read-only. Non-GET methods continue to return `method_not_allowed`.
- The HTML renderer ("Dashboard Scope Filtering") displays the panel decision counts, the redacted/hidden/warning lists, the filter context summary, and the production-safety flags.

## Known Limitations

- Filter rules are deterministic per panel and per role; per-resource filtering (e.g., "show only repos owned by team X") is not implemented.
- The default mock-admin fallback intentionally preserves existing demo behavior — without that fallback the demo dashboard would redact most sensitive panels even when the operator is the local mock admin.
- Secret-like field detection uses a deny-list of known field names; it is not a substitute for proper redaction at the source.
- Demo headers are explicitly **not authentication**; they cannot be used in production deployments.
