# Phase 3 Registry v0

Status note: this document describes the v0 baseline. Registry Hardening v1 is documented in `docs/features/registry/v1-hardening.md`.

## What v0 Implements

Phase 3 Registry v0 adds central, mock-first registries for three separate concepts:

- `SkillPackage`: reusable task capability or workflow.
- `HarnessDefinition`: execution environment and constraints for an agent.
- `InstructionArtifact`: persistent prompt or instruction context.

The implementation is intentionally in-memory and deterministic. It does not add a real artifact registry, external storage service, OCI registry, GitHub/GitLab integration, LLM provider call, MCP integration, Vault, Kubernetes, or Temporal.

## Why The Concepts Stay Separate

Skills, harnesses, and instructions have different responsibilities:

- Skills decide what capability should be invoked.
- Harnesses decide where and how the agent may run.
- Instruction artifacts decide which persistent context is loaded.

They share version references through `RegistryVersionRef`, but they remain separate domain types, APIs, registries, seed data, and tests.

## Domain Models

New or refined core models live in `packages/core/src/domain/models.ts`:

- `SkillPackage`
- `HarnessDefinition`
- `HarnessPackage`

- `RegistryVersionRef`
- `RegistryResolution`

`TaskRun` now records:

- `selectedSkillRefs`
- `selectedHarnessRef`
- `selectedInstructionRefs`
- `registryResolutionWarnings`
- `registryResolutionErrors`

Legacy summary fields such as `selectedSkillId`, `skillVersion`, `selectedHarnessId`, and `harnessVersion` remain for Phase 1 usage ledger compatibility.

## Version Pinning

Registry v0 uses exact version pinning only. A selected record is represented as:

```ts
{
  kind: "skill" | "harness" | "instruction",
  name: string,
  version: string,
  id?: string,
  checksum?: string
}
```

No semver range resolution is implemented in v0.

## Registry Resolver

`packages/registry/src/index.ts` defines provider-agnostic interfaces and in-memory v0 implementations:

- `SkillRegistry`
- `HarnessRegistry`
- `InstructionRegistry`
- `RegistryResolver`
- `InMemorySkillRegistry`
- `InMemoryHarnessRegistry`
- `InMemoryInstructionRegistry`
- `InMemoryRegistryResolver`

`resolveRegistryContextForTask` combines task metadata, requested agent, repo context, default mock configuration, available skills, available harnesses, and applicable instructions.

Default behavior:

- Backend tasks select `backend-node20@1.0.0`.
- Frontend/dashboard/React/Vite tasks select `frontend-node20@1.0.0`.
- Conflict/merge/branch lease tasks include `conflict-risk-reviewer@1.0.0`.
- Auth/login/session tasks select `auth-debugging@1.0.0`.
- Test/Jest/spec tasks select `jest-test-fixer@1.0.0`.
- All tasks include `org-secure-coding-baseline@1.0.0`.
- Repo tasks include `repo-agents-md@1.0.0` when it applies to the repo.
- Conflict tasks include `conflict-manager-guidance@1.0.0`.
- Deprecated or archived entries are not selected by default.

## TaskRun Integration

`apps/worker/src/workflows/run-agent-task-workflow.ts` now resolves registry context before model selection and records selected registry refs on each `TaskRun`.

The workflow still uses mock-first behavior:

- Mock policy engine.
- Mock model router and LLM gateway.
- Mock Git branch and PR provider.
- Mock agent runner.
- Mock or local-only merge simulator.

The usage ledger remains attributed to both `taskId` and `taskRunId`.

Harnesses:

- `POST /registry/resolve`

## Dashboard Changes

The dashboard now shows:

- Active skill, harness, and instruction counts.
- Registry lists for skills, harnesses, and instructions.
- Selected skills, selected harness, selected instructions, and registry warnings on task detail data.

The dashboard still uses seeded local data for MVP demos and tests. It should move to API-backed read models before production work.

## Test Strategy

Tests cover:

- Registry domain validation.
- Skill/Harness/Instruction separation.
- Exact version pinning.
- In-memory registry list/get/create/status update behavior.
- Resolver defaults for backend, frontend, conflict, org, and repo instructions.
- Deprecated entries not selected by default.
- Missing registry prerequisites reported deterministically.
- TaskRun registry ref recording.
- Usage ledger attribution after registry integration.
- Registry API visibility and invalid status handling.
- Dashboard registry assumptions.

## Known Limitations

- Storage is in-memory.
- There is no registry approval workflow.
- There is no signed artifact packaging.
- Instruction checksums are recorded but not enforced against file content.
- There is no semver range resolution.
- APIs return domain-shaped objects rather than stable public DTOs.
- Dashboard registry views are read-oriented and fixture-backed.

## Next Recommended Task
