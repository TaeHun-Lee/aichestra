# Policy-as-code Skeleton v0

## What v0 Implements

Policy-as-code Skeleton v0 adds a central, provider-neutral policy decision layer for protected operations across Git, LLM, Runner, Registry, and Auto-improvement boundaries.

It is intentionally static and mock-first. It does not integrate OPA/Rego, Cedar, external policy services, real auth, real secrets, or production enforcement.

## Policy Models

The policy package defines:

- `PolicySubject`: actor id, optional principal id, actor kind, roles, optional teams, auth mode, optional service account id, mock actor marker, and sanitized metadata.
- `PolicyResource`: resource kind, resource id, and sanitized metadata.
- `PolicyAction`: provider-neutral action names such as `git.remote_operation`, `llm.completion`, `runner.command.execute`, `registry.update`, `improvement.apply`, `local_agent.invoke`, and `credential.cache.read`.
- `PolicyContext`: task/run/repo/model/runner/command context, environment gates, and metadata.
- `PolicyRule`: static allow, deny, or require-approval rule.
- `PolicyDecision`: deterministic decision with matched rule ids.
- `PolicyDecisionAuditEntry`: append-only audit read model for evaluations.

## PolicyEngine Interface

`PolicyEngine` supports:

- `evaluate`
- `evaluateMany`
- `listRules`
- `getRule`
- `validatePolicySet`

`StaticPolicyEngine` is the only v0 implementation. It uses a deterministic in-process rule set and never executes dynamic policy code.

## Default Policy Set

Default rules are restrictive:

- deny remote Git operations unless explicit Real Git Adapter v1 gates, repo allowlists, credentials, and operation-specific policy context pass;
- deny merge, rebase, and branch deletion;
- allow mock Git branch and PR operations;
- allow GitHub branch/PR creation only when remote Git, branch/PR gate, repo allowlist, branch prefix, and credential metadata are all satisfied;
- deny remote LLM completion by default and allow OpenAI-compatible v1/v2 completion only when remote LLM, completion, base URL, credential, model allowlist, routing, and budget gates pass;
- allow `llm.route.select` for mock routes and the gated OpenAI-compatible route, while keeping provider/model/config gates authoritative;
- deny `llm.fallback` unless fallback is explicitly enabled and bounded;
- allow mock LLM completion only after budget checks pass;
- deny disabled/deprecated model usage;
- deny runner command execution by default;
- allow fixture-local runner commands only when local execution, harness, and workspace gates pass;
- deny network commands, remote Git commands, secret reads, legacy MCP tool calls, and unsafe MCP tool paths;
- allow active mock MCP server/tool listing for safe mock roles;
- allow low-risk read-only mock MCP tool invocation only when real transport, secrets, network, write, deploy, and local execution are all disabled;
- deny high-risk MCP tools, critical MCP tools, MCP secret resolution, MCP network access, MCP write tools, and MCP deploy tools by default;
- deny `improvement.apply`;
- deny enterprise provider cloud API invocation, credential cache reads/uploads, PTY invocation, and local CLI shell/file/network access by default;
- allow env-backed SecretRef provider, Git, and LLM credential resolution only when `SecretRef` is active, env secret provider is enabled, and no credential cache access or stored credential material is involved;
- deny direct `local_cli.invoke`, `local_cli.danger_full_access`, `local_cli.network_access`, and `local_cli.shell_execution` by default;
- allow mock Local Agent registration/revoke/cancel/event receipt for system or mock admin roles;
- allow mock Local Agent channel creation, mock handshake verification, capability advertisement, compatibility checks, and stream event receipt for fixture flows;
- deny real Local Agent transport, PTY, incompatible local CLI templates, credential cache reads/uploads, and dangerous local CLI capabilities;
- require consent for `local_agent.invoke`;
- allow read-only `local_agent.invoke` only when policy, consent, connected-agent, mock-transport, sandbox, and redaction gates are satisfied;
- deny invocation for revoked, disconnected, pending, or unknown Local Agents;
- allow mock provider invocation and allow local CLI provider requests only to reach the Local Agent Protocol boundary after consent and compatibility gates;
- allow read-only dashboard/auth metadata for safe mock roles;
- allow authorization check only for security/platform/system admin mock roles;
- deny production auth administration;
- require registry editor/admin/reviewer roles for mutations, approvals, and rollback;
- demonstrate `require_approval` for high-risk task execution.

## Integrations

Policy checks are integrated where practical:

- `GitIntegrationService`: branch creation, PR creation, and remote operation attempts.
- `LLMGatewayService`: model use, route selection, bounded fallback, remote completion, and completion routing.
- `AgentRunnerService`: runner execution and command execution.
- `LocalAgentProtocolService`: agent registration, mock channel/handshake, capability advertisement, compatibility check, consent, invocation, dispatch, stream/event receipt, timeout, cancellation, disconnect, revocation, and blocked credential-cache/direct-local-CLI audit events.
- Registry: `PolicyBackedRegistryMutationAuthorizer` wraps existing mock RBAC.
- Auth/RBAC: `AuthorizationService` maps `AuthContext` to `PolicySubject`, performs RBAC checks, and then calls `PolicyService`; policy denial remains authoritative.
- SecretRef-backed Provider Credentials: `SecurityControlService` evaluates `git.credential.resolve`, `llm.credential.resolve`, `provider.credential.resolve`, and metadata lease policies before any env secret provider read.
- MCP Gateway: `MockMCPGateway` checks Auth/RBAC and Policy-as-code before deterministic low-risk mock tool invocation, and blocked MCP decisions are audited.
- API and dashboard: `/policy/*` endpoints, health metadata, rules, audit entries, and blocked examples.

Existing domain gates remain authoritative. Policy v0 does not weaken resolver approval/eval/checksum gates, runner harness policy, budget gates, or governance apply gates.

## API Endpoints

- `GET /policy/config`
- `GET /policy/rules`
- `GET /policy/rules/:id`
- `POST /policy/evaluate`
- `POST /policy/evaluate-many`
- `GET /policy/audit`

The API does not accept dynamic policy code or upload policy bundles.

Auth/RBAC v0 also adds `/auth/*` endpoints that call Policy-as-code through `AuthorizationService`; no real login/session/token endpoint exists.

## Dashboard Changes

The dashboard now shows:

- policy engine kind;
- loaded rule count;
- audit enabled state;
- recent policy decisions;
- blocked operation examples;
- policy decision audit entries.
- auth/RBAC read models now expose mock auth mode, current actor, role/permission summaries, authorization examples, and auth audit events.

## Schema and Persistence

Policy v0 uses an in-memory audit repository by default. The Postgres schema skeleton includes `policy_decision_audit_entries` for future durable audit persistence.

## Safety Constraints

- No OPA/Rego or Cedar runtime.
- No `eval()` or user-provided policy execution.
- No external policy service.
- No provider calls.
- No real secrets.
- No default remote Git operations. Real Git Adapter v1 remote branch/PR operations require explicit environment gates, allowlists, credentials, and policy allow decisions.
- No real LLM calls by default. LLM Gateway v1/v2 may call an OpenAI-compatible endpoint only behind explicit runtime gates, model allowlists, routing gates, budget checks, credential checks, and policy allow decisions.
- No auto-apply of improvement proposals.

- regression behavior for existing mock-first gates.

## Known Limitations

- Policy rules are static and code-defined.
- Policy audit is in-memory at runtime.
- Production Auth/RBAC Planning v0 exists as a mock-first identity/authorization boundary, but no production identity source exists.
- No OPA/Rego, Cedar, policy bundle loading, or external policy decision service exists.
- Policy decisions are centralizing safety checks, not replacing production security controls.
