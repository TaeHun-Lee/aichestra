# Local CLI Provider Templates v1

Status: `v1_implemented`.

Local CLI Provider Templates v1 adds first-class, metadata-only templates for future local CLI provider integrations. It does not execute vendor CLIs, spawn processes, open PTYs, read credential caches, forward secrets, or implement a real Local Agent daemon.

## What v1 implements

- `LocalCliProviderTemplate` metadata for Claude Code, OpenAI Codex CLI, Gemini CLI, Aider, and a custom local CLI provider.
- `LocalCliCompatibilityRule` metadata for supported mock behavior, blocked behavior, unsupported behavior, and future behavior.
- `LocalCliParserProfile` metadata for expected output shape, patch detection, diff summary, error detection, redaction rules, and output byte limits.
- `LocalCliSecurityConstraint` metadata for credential-cache denial, secret-forwarding denial, direct-execution denial, no network by default, no remote Git, no PTY, no shell escape, and required output redaction.
- Enterprise Provider catalog/readiness/dashboard integration.
- Local Agent Protocol compatibility metadata for v1 template ids.
- Policy actions and default rules for safe template reads and denied execution/cache/secret operations.
- Read-only API endpoints:
  - `GET /providers/local-cli/templates`
  - `GET /providers/local-cli/templates/:id`
  - `GET /providers/local-cli/compatibility`
  - `GET /providers/local-cli/security-constraints`
  - `GET /providers/local-cli/readiness`

## What v1 does not implement

- No Claude Code, Codex CLI, Gemini CLI, Aider, Cursor, Continue, or custom vendor CLI execution.
- No `child_process`, PTY, shell automation, or vendor process spawning.
- No real Local Agent daemon, tunnel, WebSocket, gRPC, HTTP transport, or production pairing flow.
- No credential cache reads or validation.
- No reads of `~/.codex/auth.json`, Claude credentials, Google credential caches, OS keychains, or vendor config files containing credentials.
- No secret forwarding, env value exposure, or credential injection into CLI metadata.
- No OAuth, device-code, WIF, IAM, BYOK, cloud auth, or provider credential exchange.
- No external LLM provider calls and no remote Git operations.
- No production-ready local CLI integration claim.

| `gemini-cli-template-v1` | `gemini-cli-local` | `gemini_cli` | `template_only` | `local_agent_required` |

All templates require Local Agent capabilities for `local_cli`, `workspace_read`, stdout/stderr preview handling, and sandbox metadata. They forbid workspace write, shell execution, network access, secret access, PTY, credential-cache reads, secret forwarding, and remote Git.

## Compatibility matrix

The compatibility matrix is deterministic metadata:

| Capability | Default status | Notes |
| --- | --- | --- |
| `local_agent_channel` | `supported_mock` | Requires Local Agent protocol boundary. |
| `mock_fixture_completion` | `supported_mock` | Fixture daemon may return deterministic `mock_completed` only. |
| `parser_profile` | `supported_mock` | Parser expectations are metadata-only. |
| `direct_execution` | `blocked` | Aichestra must not spawn vendor CLIs. |
| `pty` | `unsupported` | PTY automation remains out of scope. |
| `credential_cache_read` | `blocked` | Vendor credential caches are never read. |
| `secret_forwarding` | `blocked` | Secrets are never passed into CLI processes. |
| `network_access` | `blocked` | Network is denied by default. |
| `remote_git` | `blocked` | Remote Git remains forbidden. |
| `shell_escape` | `blocked` | Shell string execution and escapes are forbidden. |
| `real_local_agent_daemon` | `future` | Production daemon design remains a future task. |

## Parser profiles

Parser profiles describe expected output without parsing live vendor output:

| Profile | Provider id | Expected shape | Patch detection |
| --- | --- | --- | --- |
| `parser_claude_code_v1` | `claude-code-local` | `jsonl_events_future` | `fenced_patch_blocks` |
| `parser_codex_cli_v1` | `codex-cli-local` | `jsonl_events_future` | `unified_diff_markers` |
| `parser_gemini_cli_v1` | `gemini-cli-local` | `json_object_future` | `fenced_patch_blocks` |
| `parser_aider_v1` | `aider-local` | `mixed_text_patch` | `unified_diff_markers` |
| `parser_custom_local_cli_v1` | `custom-local-cli` | `plain_text` | `text_summary_only` |

Profiles include deterministic redaction rules for API-key-like values, token-like values, private key blocks, known credential-cache path strings, and env-value-like shapes. `maxOutputBytes` is capped at `20000`.

## Security constraints

Every v1 provider template includes enforced constraints for:

- `no_credential_cache_read`
- `no_secret_forwarding`
- `no_direct_execution`
- `no_network_by_default`
- `no_remote_git`
- `no_pty`
- `no_shell_escape`
- `output_redaction_required`

Readiness reports `credentialCacheReadAllowed = false`, `secretForwardingAllowed = false`, `directExecutionEnabled = false`, `vendorCliExecutionImplemented = false`, `envValuesExposed = false`, and `secretsExposed = false`.

## Local Agent Protocol dependency

Local CLI templates remain protocol-gated:

- Local Agent channel required.
- Consent required.
- Fixture daemon may return mock results only.
- Real daemon unsupported.
- PTY unsupported.
- Vendor execution unsupported.
- Credential cache reads denied.

## Policy and security behavior

New or verified policy actions:

- `local_cli.template.read`: allowed only for safe read metadata.
- `local_cli.invoke`: blocked unless a mock/local-agent fixture path reaches the existing boundary.
- `local_cli.execute`: denied.
- `local_cli.credential_cache.read`: denied.
- `local_cli.secret.forward`: denied.
- `local_agent.invoke`: still subject to Local Agent protocol, consent, compatibility, and policy requirements.
- `provider.invoke`: still subject to provider and policy gates.

These rules do not weaken Auth/RBAC, Policy-as-code, SecretRef, Git, LLM, MCP, Local Agent, Runner, Dashboard, Observability, Registry, Governance, Staging, CI/CD, Tenant Scope, or Secrets/Sandbox gates.

## API and dashboard surfaces

The provider API exposes read-only template, compatibility, security, and readiness data. The dashboard Enterprise LLM Providers panel now shows:

- template ids and status;
- compatibility rule summaries;
- parser profile summaries;
- security constraints;
- Local Agent requirement;
- credential-cache denied status;
- secret-forwarding denied status;
- PTY unsupported status;
- execution disabled status;
- no-secret/no-env status.

The API and dashboard do not probe the filesystem, read env values, show credential paths, show credential cache paths, expose executable paths, or execute provider commands.

## Known limitations

- Template metadata is not a compatibility guarantee for real vendor CLI versions.
- Parser profiles are expectations only; they are not validated against live vendor output.
- Custom local CLI providers are template-only and cannot be executed.
- Tenant-scoped provider visibility remains metadata-only until future production tenant filtering work.
