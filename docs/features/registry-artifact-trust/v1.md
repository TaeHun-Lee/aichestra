# Registry Signed Package / Artifact Trust v1

Registry Signed Package / Artifact Trust v1 adds mock-first trust metadata for registry packages and artifacts. It models artifact digests, mock signature metadata, provenance metadata, trust policies, and trust decisions without performing real cryptographic signing or verification.

Status:

- Registry Signed Package / Artifact Trust: `v1_implemented`
- Real signing: not implemented
- Real verification: not implemented
- External artifact registry integration: not implemented
- Registry resolver gates: preserved

## What v1 Implements

- `RegistryArtifactDigest`, `RegistryArtifactSignature`, `RegistryArtifactProvenance`, `RegistryArtifactTrustPolicy`, `RegistryArtifactTrustDecision`, and `RegistryArtifactTrustSummary` models.
- `RegistryArtifactTrustService` in `packages/registry`, with deterministic `evaluateArtifactTrust`, `evaluatePackageTrust`, `listTrustPolicies`, `listTrustDecisions`, `getTrustSummary`, `createMockSignatureMetadata`, and `attachProvenanceMetadata` behavior.
- Mock signature/provenance metadata helpers that never create keys, never store raw signatures, and never verify cryptographic material.
- Registry package export metadata for local artifact trust planning.
- Registry package import trust evaluation that warns or blocks only through local metadata decisions and never calls remote registries.
- Resolver trust metadata on `RegistryResolution.artifactTrustDecisions` and `RegistryResolution.artifactTrustSummary`.
- Policy actions for artifact trust reads/evaluation, mock metadata attachment, and denied future sign/verify/import paths.
- Metadata-only API, readiness, health, and dashboard visibility.

## What v1 Does Not Implement

- Real cryptographic signing.
- Real signature verification.
- Signing key generation, storage, KMS, Vault transit, GPG, Sigstore, Cosign, or cloud signing.
- Artifact upload/download, remote registry publishing, or remote import.
- SBOM generation, transparency log verification, or production provenance trust.
- Production resolver enforcement for signed artifacts.
- Auto-improvement mutation of active registry entries.

## Artifact Digest Model

`RegistryArtifactDigest` records artifact id/kind, digest algorithm, digest value, digest status, and sanitized metadata. v1 derives package digest metadata from existing package checksums where present. Missing digests warn or block according to trust policy; digest mismatch is a representative blocker.

## Signature Metadata Model

`RegistryArtifactSignature` records signature kind/status, signer id, signing authority, optional key ref id, timestamps, and sanitized metadata. v1 supports `mock_signature` metadata only. Future signature kinds (`sigstore_future`, `cosign_future`, `gpg_future`, `kms_future`, `vault_transit_future`) are modeled as future statuses and are not executable integrations.

## Provenance Model

`RegistryArtifactProvenance` records source repo/branch/commit, build/task/agent run ids, creator actor/service-account ids, build system, provenance status, and sanitized metadata. Missing provenance warns by default in v1; sensitive representative cases may block.

## Trust Policy And Decision Model

The default policy is `active_mock`:

- digest required;
- mock signature allowed but not required by default;
- provenance recommended but not required by default;
- non-expired and non-revoked signatures required when a signature is present.

Trust decisions include `trusted`, `trusted_with_warnings`, `untrusted`, `blocked_unsigned`, `blocked_digest_mismatch`, `blocked_invalid_signature`, `blocked_missing_provenance`, and `future_verification_required`.

## Resolver Integration

Artifact trust metadata is attached after existing resolver selection. It cannot make an excluded entry selectable. Lifecycle, approval, eval, checksum, semver, policy, governance, and tenant scope gates remain authoritative. Pending approval still excludes candidates and checksum mismatch still blocks selection.

Eval Suite Execution Harness v1 can inspect registry package checksum/trust metadata through the `registry-package-artifact-trust-suite`. Artifact Trust v1 still performs no real signing or verification, and the eval harness cannot bypass trust or resolver blockers.

## Import / Export Trust Model

Package export includes local artifact trust metadata (`realSigningImplemented=false`, `realVerificationImplemented=false`, `externalRegistryCalls=false`). Package import evaluates local trust metadata before mutation. Dry-run import remains non-mutating. v1 does not fetch, upload, publish, or verify remote artifacts.

## Policy / Auth Behavior

New policy actions:

- `registry.artifact_trust.read`
- `registry.artifact_trust.evaluate`
- `registry.artifact_signature.attach_mock`
- `registry.artifact_provenance.attach`
- `registry.artifact_signature.verify_future`
- `registry.artifact.sign_future`
- `registry.artifact.import_trusted_future`

Read/evaluate/mock metadata actions are allowed only under safe metadata-only conditions. Future real sign/verify/trusted import actions are denied by default. Policy deny wins.

## API / Dashboard Surfaces

API endpoints:

- `GET /registry/artifact-trust/policies`
- `GET /registry/artifact-trust/decisions`
- `GET /registry/artifact-trust/summary`
- `POST /registry/artifact-trust/evaluate`
- `POST /registry/artifact-trust/mock-signature`
- `POST /registry/artifact-trust/provenance`
- `GET /readiness/registry/artifact-trust/summary`

Dashboard surfaces:

- `/dashboard/registry` includes the "Registry Artifact Trust" panel.
- `/health.registryArtifactTrust` exposes status and safety booleans.

All surfaces are sanitized metadata only and expose no key material, raw signatures, secrets, env values, credential caches, or provider tokens.

## Safety Guarantees

- No real signing.
- No real verification.
- No signing keys generated or stored.
- No external artifact registry calls.
- No GitHub, LLM, MCP, Vault, KMS, Sigstore, Cosign, GPG, Kubernetes, Temporal, OPA, Cedar, cloud, or external auth calls.
- No registry resolver gate weakening.
- No active registry mutation through auto-improvement.

## Known Limitations

- Trust decisions are in-memory/mock metadata.
- No durable trust decision repository exists yet.
- No production signed bundle verification profile exists.
- No SBOM/provenance attestation format exists.
- No production tenant-isolated artifact trust enforcement exists.
