# Conflict Resolution Assistant v1

## Status

Conflict Resolution Assistant v1 is implemented as a mock-first, metadata-only conflict explanation and review planning layer.

It never applies patches, mutates source files, runs merges, releases merge queue holds automatically, calls remote Git providers, calls real LLM providers, executes vendor CLIs, reads credential caches, or exposes secrets/env values.

## What v1 Implements

- Core models in `packages/core/src/conflicts/conflict-resolution-assistant.ts`:
  - `ConflictResolutionRequest`
  - `ConflictSummary`
  - `ConflictResolutionPlan`
  - `ConflictResolutionRecommendation`
- `ConflictResolutionAssistantService` with deterministic in-memory request, summary, plan, recommendation, review, and merge-queue-hold-link metadata.
- API endpoints under `/git/conflicts/assistant/*`.
- Dashboard conflict panel visibility for assistant requests, summaries, plans, recommendations, safety status, and disabled apply status.
- PR Ownership / Handoff Model v1 linkage so plans can be associated with responsible owner/reviewer metadata.
- `/health` metadata for assistant status and safety flags.
- Demo dashboard data and deterministic tests.

## What v1 Does Not Implement

- automatic conflict resolution
- patch generation or patch application
- source-file reads for conflict content
- source-file writes
- merge, rebase, fetch, push, checkout, branch deletion, force-push, or worktree mutation
- remote PR updates
- real LLM calls
- vendor CLI execution
- credential cache reads
- durable production storage
- production tenant enforcement changes

## Conflict Assistant Model

`ConflictResolutionRequest` records the review scope:

- repo id
- base/source/target branches
- optional dry-run simulation, merge queue entry, and conflict risk ids
- branch lease ids
- optional workspace lease and edit overlap ids
- affected files
- request context/correlation metadata
- sanitized metadata

Requests may derive files from dry-run simulation, conflict risk, branch lease, and edit overlap evidence. Absolute paths and secret-like metadata are sanitized before API/dashboard output.

## Conflict Summary Model

`ConflictSummary` records:

- conflict files
- base/source/target branch metadata
- conflict kind:
  - `same_file_edit`
  - `same_directory_refactor`
  - `rename_modify_future`
  - `delete_modify_future`
  - `generated_file`
  - `dependency_lockfile`
  - `unknown`
- severity:
  - `low`
  - `medium`
  - `high`
  - `critical`
- likely cause
- affected areas
- evidence source metadata

Classification is deterministic. Dry-run text conflicts and same-file overlap classify as same-file edits. Same-directory or broad refactor metadata classifies as same-directory refactor. Lockfile/package conflicts classify as dependency conflicts. Generated/build output paths classify as generated-file conflicts.

## Resolution Plan Model

`ConflictResolutionPlan` records review-only guidance:

- status:
  - `draft`
  - `review_required`
  - `blocked`
  - `future_apply`
- strategy:
  - `manual_review`
  - `choose_branch_a`
  - `choose_branch_b`
  - `combine_changes`
  - `split_commits_future`
  - `rerun_agent_future`
  - `human_resolution_required`
- steps
- suggested validation commands
- suggested tests
- risks
- `applyAllowed: false`

`markPlanReviewed` can record review metadata and move non-blocked plans to `future_apply`, but v1 still keeps `applyAllowed: false` and does not apply changes.

## Recommendations

`ConflictResolutionRecommendation` records reviewer actions:

- review a file
- run a validation command after manual resolution
- request human review
- rerun dry-run merge after manual resolution
- split broad work
- defer branch
- update or link a merge queue hold

Recommendations are metadata only. The assistant does not execute suggested commands.

## Merge Queue Interaction

Merge Queue Policy v2 remains authoritative. The assistant can:

- consume merge queue entry ids and policy evidence;
- recommend keeping or linking holds;
- link a plan to a merge queue hold id for dashboard/reviewer visibility;
- expose assistant plan ids in metadata.

The assistant does not release holds automatically, mark entries ready, execute merges, update PRs, or bypass policy deny decisions.

## PR Ownership / Handoff Interaction

PR Ownership / Handoff Model v1 can link `PrOwnershipRecord.conflictResolutionPlanId` to an assistant plan and expose owner/reviewer metadata for conflict review.

The assistant remains authoritative only for review plans and recommendations. It does not transfer PR ownership, assign reviewers remotely, mark handoffs accepted, update PRs, or release merge queue holds.

## Dry-run Merge Interaction

The assistant consumes `MergeSimulationResult` metadata:

- status
- conflicting files
- changed files
- summary
- risk contribution

It uses this metadata for classification and severity. It does not invoke `MockMergeSimulator`, `LocalGitDryRunMergeSimulator`, or any Git command itself. Rerun dry-merge is only a recommendation.

## Edit Intent Interaction

The assistant can consume Cross-session File Lease / Edit Intent Graph v1 overlap assessments. Same-file, same-directory, same-workspace, broad refactor, future rename, and future delete hints contribute to conflict kind, severity, risks, and recommendations.

The assistant does not lock files, inspect file contents, mutate source files, create branches, create worktrees, call providers, or read credential caches.

## LLM Future Hook

v1 includes metadata for a future `Conflict Resolution Assistant LLM Proposal Profile v1`, but it does not call LLM Gateway or any provider. Default summaries and plans are deterministic local logic only.

All v1 outputs include safe status such as:

- `realLlmUsed: false`
- `noExternalLlmCall: true`
- `applyAllowed: false`
- `noPatchApply: true`

- `GET /git/conflicts/assistant/requests/:id`
- `POST /git/conflicts/assistant/requests/:id/summarize`
- `POST /git/conflicts/assistant/requests/:id/plan`
- `GET /git/conflicts/assistant/plans`
- `POST /git/conflicts/assistant/plans/:id/reviewed`

- `GET /git/conflicts/assistant/summary`

All endpoints are metadata-only. They use RequestContext/AuthContext attribution and sanitized output.

## Dashboard and Health

`/dashboard/conflicts` now includes:

- conflict assistant requests
- conflict summaries
- conflict plans
- recommendations
- assistant summary
- severity, files, strategy, validation, and risks
- `applyAllowed: false`
- `realLlmUsed: false`
- `mergeExecution: false`
- `sourceMutation: false`

`/health` exposes status and safe booleans only.

## No-auto-apply Guarantee

`applyAllowed` is always `false` in v1. The assistant never applies patches, writes files, merges branches, releases holds, updates pull requests, or runs validation commands. Any future apply path must be a separate gated design and implementation task.

## Safety Constraints

- no source mutation
- no patch apply
- no real merge/rebase/fetch/push/checkout/switch/force-push/branch deletion
- no remote Git provider call
- no real LLM provider call by default
- no vendor CLI execution
- no credential cache reads
- no secrets/env values
- policy deny remains authoritative
- merge queue execution remains disabled

## Known Limitations

- Storage is in-memory in the default runtime.
- Classification uses metadata and paths only, not file contents.
- Future rename/delete classification depends on edit-intent metadata hints.
- Suggested validation is heuristic and not executed.
- No durable review workflow or approval evidence repository is added.
- PR owner/reviewer linkage is metadata-only and in-memory through PR Ownership / Handoff Model v1.
- Real Merge Execution Policy v1 consumes unresolved/reviewed conflict resolution plan metadata as future execution precondition evidence. This assistant still never applies plans or authorizes merge execution.
