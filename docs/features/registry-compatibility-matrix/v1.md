# Skill / Harness Compatibility Matrix v1

Status: v1_implemented

Skill / Harness Compatibility Matrix v1 is implemented as a registry-level, advisory metadata layer that helps determine which Skill, Harness, and Instruction artifacts are compatible with a given task, repo, provider/model, runner, MCP, and tenant/repo/provider scope context. The matrix is deterministic and mock-first. It never bypasses the existing Registry resolver gates (lifecycle, approval, eval, checksum, semver, policy), never mutates registry entries, never executes evals/canaries, and never calls external providers/MCP/LLMs/vendor CLIs.

## What v1 Implements

- Models (`packages/registry/src/compatibility.ts`):
  - `RegistryCompatibilityContext` and `RegistryCompatibilityContextInput`
  - `SkillCompatibilityProfile`, `HarnessCompatibilityProfile`, `InstructionCompatibilityProfile`
  - `RegistryCompatibilityRule` (16 default rules covering task_kind, language, framework, provider/model capability, runner capability, MCP tool, security policy, tenant/repo scope, eval/lifecycle/approval/checksum)
  - `RegistryCompatibilityDecision` with `decision ∈ {compatible | compatible_with_warnings | incompatible | blocked_by_policy | blocked_by_registry_gate | future_unknown}`
  - `RegistryCompatibilitySummary` exposing rule/profile counts and explicit `resolverGatesPreserved: true`, `autoApplyEnabled: false`, `registryMutationsExecuted: false`, `externalCallsExecuted: false`, `noSecretsExposed: true`, `envValuesExposed: false`.
- Service: `RegistryCompatibilityService` with `buildContext`, `evaluateSkill`, `evaluateHarness`, `evaluateInstruction`, `evaluatePackage`, `evaluateCandidates`, `summarizeDecisions`, `listRules`, `listSkillProfiles`, `listHarnessProfiles`, `listInstructionProfiles`, `getSummary`.
- Default seed profiles for the three seeded skills (`skill_jest_test_fixer`, `skill_auth_debugging`, `skill_conflict_risk_reviewer`), three seeded harnesses (`harness_backend_node20`, `harness_frontend_node20`, `harness_local_git_dry_run`), and two seeded instructions (`instr_org_secure_coding_baseline`, `instr_conflict_manager_guidance`).
- New policy actions and default rules (`packages/policy/src/types.ts`, `packages/policy/src/default-rules.ts`):
  - `registry.compatibility.read` and `registry.compatibility.evaluate` — allow under mock metadata-only context.
  - `registry.compatibility.matrix.update_future` and `registry.compatibility.override_future` — deny by default.
  - New `registry_compatibility` `PolicyResourceKind`.
- Read-only API endpoints under `/registry/compatibility/*`:
  - `GET /registry/compatibility/rules`
  - `GET /registry/compatibility/summary`
  - `GET /registry/compatibility/candidates`
  - `POST /registry/compatibility/evaluate`
  - `POST /registry/compatibility/evaluate-many`
- Read-only readiness endpoint: `GET /readiness/registry/compatibility/summary`.
- `/health` metadata key `registryCompatibility` exposing planning status, rule/profile counts, and safety flags.
- Dashboard read model and panel: `GET /dashboard/registry-compatibility` and the "Skill / Harness Compatibility Matrix" panel in the rendered HTML.
- Dashboard/Readiness Tenant Scope plan entries for `registry_compatibility` (`docs/reference/runtime-component-inventory.md` and `packages/deployment-readiness/src/dashboard-tenant-scope.ts`).
- Deterministic tests (`tests/registry-compatibility-matrix-v1.test.ts`) covering context construction, every decision pathway, registry gate preservation, policy deny/allow defaults, API endpoints, dashboard panel, and no-secret/no-mutation guarantees.

## What v1 Does Not Implement

- No live LLM, MCP, vendor CLI, GitHub, or any external-provider calls.
- No registry mutation. Compatibility decisions are advisory only.
- No auto-apply or auto-override of registry resolver selections.
- No production tenant enforcement. Scope mismatches surface as warnings.
- No eval/canary execution. `requiredActions` can recommend `attach_eval`/`rerun_eval`, but no eval is run.
- No durable storage. Rules and profiles are deterministic seed data.
- No drift detection or signed-package trust. Those remain separate planned tasks.

## Compatibility Context

`RegistryCompatibilityContext` normalizes string inputs to lower-case, deduplicates, and defaults to deterministic safe values (`taskKind: "general"`, `runnerKind: "mock"`, empty arrays). The context never includes secrets, env values, or full repository paths.

## Skill Profile

`SkillCompatibilityProfile` declares supported task kinds, languages, frameworks, repo kinds, required/forbidden model capabilities, required/forbidden MCP tools, required scopes, and compatibility status (`active_mock | planned | disabled | future`).

## Harness Profile

`HarnessCompatibilityProfile` declares supported runner kinds, languages, frameworks, required/forbidden commands, network/file-write/remote-git/secrets requirements, required policies, and compatibility status. v1 captures local-only harness constraints (e.g., the `local-git-dry-run` harness forbids `git push`, `git fetch`, `git merge`, `git rebase`).

## Instruction Profile

`InstructionCompatibilityProfile` declares supported task kinds, supported skill ids, supported harness ids, supported provider kinds, supported model capabilities, required scopes, and compatibility status.

## Rule Model

`RegistryCompatibilityRule` carries `ruleKind`, `severity` (`info | warning | blocking | critical`), description, and metadata. The 16 default rules cover the documented dimensions and explicitly mark gate-preserved rules (`lifecycle_status`, `approval_status`, `eval_status`, `checksum_status`) with `metadata.gatePreserved: true`.

## Decision Model

For each candidate the service produces a `RegistryCompatibilityDecision` with reasons, warnings, blockers, score, required actions, and sanitized metadata. Decision composition rules:

- registry-gate blockers (lifecycle/approval/eval/checksum) ⇒ `blocked_by_registry_gate`,
- other blockers ⇒ `incompatible`,
- only warnings ⇒ `compatible_with_warnings`,
- no blockers/warnings ⇒ `compatible`,
- profile status `planned`/`future` ⇒ `future_unknown`,
- profile status `disabled` ⇒ `incompatible`,
- policy deny ⇒ `blocked_by_policy`.

Score is a deterministic integer in `[0..100]` derived from the decision class and warning/blocker counts.

## Resolver Integration

The compatibility service consumes the same lifecycle/approval/eval/checksum predicates the resolver uses (`selectableRegistryEntry` semantics) so that excluded registry entries always surface as `blocked_by_registry_gate`. The resolver's `resolveRegistryContextForTask` is unchanged: it continues to enforce its existing gates and select entries deterministically. Callers can read the compatibility decisions alongside resolver output to gain advisory metadata, but the matrix never overrides resolver selection. Tests assert this regression.

## Tenant / Repo / Provider Scope Behavior

Scope mismatch is metadata-only in v1. Missing `repoId` and `tenantId` produce warnings. The matrix records `productionTenantEnforcement: false` and `scopeMismatchSeverity: "warning"` on the dashboard panel and `/health` payload. Production tenant enforcement remains future work in Tenant Scope Enforcement v1 partial.

Registry Tenant Scope Enforcement v1 now adds a separate representative scope layer for registry reads, resolver results, approval queue summaries, audit/readiness queries, and mutation scope checks. Compatibility decisions remain advisory and must not override either the registry resolver gates or the registry scope decisions.

Registry Signed Package / Artifact Trust v1 adds a separate digest/mock-signature/provenance trust metadata layer for registry packages and resolver results. Compatibility remains about fit between task/repo/provider/runner/MCP/scope context and registry candidates; artifact trust remains about package provenance and future signed-bundle readiness. Neither layer can make an entry selectable when lifecycle, approval, eval, checksum, semver, policy, tenant scope, or resolver gates exclude it.

Eval Suite Execution Harness v1 can consume compatibility metadata through `compatibility-matrix-suite` and emit deterministic mock eval run evidence. Compatibility Matrix v1 remains advisory: it can recommend `attach_eval` or `rerun_eval`, but it does not execute evals, call providers, schedule canaries, or bypass resolver gates.

## Policy / Auth Behavior

Default static policy:

- `registry.compatibility.read` — allow for viewer+ when `metadataOnly && !registryMutationExecuted && !autoApplyEnabled && !externalCallExecuted`.
- `registry.compatibility.evaluate` — allow for developer+ under the same context and `resolverGatesPreserved: true`.
- `registry.compatibility.matrix.update_future` — deny by default.
- `registry.compatibility.override_future` — deny by default.

When the policy evaluator returns `deny`, the compatibility service short-circuits the evaluation to `blocked_by_policy`. Policy deny always wins.

## API and Dashboard Surfaces

- `/registry/compatibility/*` endpoints are read-only or evaluation-only; non-GET/POST methods return `method_not_allowed`.
- `/readiness/registry/compatibility/summary` is read-only.
- `/dashboard/registry-compatibility` returns a `RegistryCompatibilityReadModel` with summary, rule list, profile lists, candidate decision lists (compatible/warning/incompatible/blocked-by-policy/blocked-by-registry-gate/future_unknown), resolver gate relationship, scope relationship, and no-auto-apply status. Dashboard sanitization redacts secret-like patterns and credential cache paths via existing helpers.
- `/health.registryCompatibility` exposes counts and explicit `resolverGatesPreserved: true`, `autoApplyEnabled: false`, `registryMutationsExecuted: false`, `externalCallsExecuted: false`.

## No-Auto-Apply Guarantee

Compatibility decisions are advisory metadata only. The service never mutates registry entries, never calls external providers, never auto-applies recommendations, and never relaxes resolver gates. `requiredActions` field contains *human-actionable* suggestions like `request_approval`, `attach_eval`, `select_compatible_model`, but the matrix neither performs nor schedules those actions.

## Known Limitations

- Profiles are deterministic seed data; durable storage and per-tenant customization are future work.
- Capability inference is shallow: `repoLanguages`, `frameworks`, and `modelCapabilities` are caller-supplied tags, not auto-detected.
- High-risk MCP tool detection relies on caller-supplied `highRiskMcpToolIds`; no real MCP probing is done.
- Tenant scope mismatches stay warnings until Tenant Scope Enforcement v1 lands production enforcement.
- Eval/canary integration is metadata-only inside this service; Eval Suite Execution Harness v1 can execute deterministic mock metadata cases separately, while this matrix still performs no eval execution itself.
- Package/bundle evaluation returns `future_unknown` by design until durable package-level compatibility lands.
