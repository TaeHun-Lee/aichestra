# Real Git Adapter v2

## What v2 Adds

Real Git Adapter v2 adds a safe GitHub webhook receiver and synchronization layer on top of the v1 GitHubClient boundary. It records verified webhook metadata, durable PR/branch sync read models, webhook audit events, and dashboard/API visibility.

The default runtime remains mock-first. Webhook handling is disabled by default, default tests do not call GitHub, and v2 does not implement automatic merge, rebase push, force push, branch deletion, GitHub App installation rollout, GitLab, Bitbucket, or LLM Gateway v2.

GitHub App Controlled Implementation v1 now layers a gated GitHub App auth-mode boundary on top of this adapter. It adds runtime/read models, a disabled/mock token provider boundary, metadata-only private-key SecretRef support, and status-only API/dashboard/health visibility. It does not implement private-key signing, JWT issuance, live installation-token exchange, production GitHub App rollout, or new destructive Git behavior.

Agent Workspace Lifecycle v2 now links future per-agent-run workspace leases to optional `branchLeaseId` metadata. This is a runner/workspace lifecycle model only: Real Git Adapter v2 still does not create local worktrees, switch branches, perform provider merges/rebases, push, force-push, or delete branches.

Multi-session Agent Run Coordination v1 now links sessions to optional `branchLeaseId` and `workspaceLeaseId` metadata, flags active sessions that share branch names or workspace leases, and surfaces same-file/same-directory overlap before merge queue work. This remains runner coordination metadata only and does not create Git refs, execute Git commands, or call GitHub.

Multi-user / Multi-session Branch Orchestrator v2 now adds branch allocation and ownership metadata on top of Real Git Adapter v2. It generates deterministic safe `aichestra/` branch names, links branch ownership to `BranchLease` and optional `WorkspaceLease` ids, detects same active branch collisions, models base branch drift, and surfaces dashboard/API readiness. This remains metadata-only and does not create local or remote Git branches, switch branches, push, fetch, merge, rebase, force-push, delete branches, or call GitHub.

Cross-session File Lease / Edit Intent Graph v1 now adds file lease and edit intent graph metadata for planned edits across active sessions. It can flag same-file, same-directory, same-branch, same-workspace, and broad unknown edit risks before merge queue work. This remains runner metadata only and does not create Git refs, execute Git commands, lock files, or call GitHub.

## Webhook Config Gates

GitHub webhook processing requires explicit configuration:

- `AICHESTRA_ENABLE_GITHUB_WEBHOOKS=true`
- `AICHESTRA_GITHUB_WEBHOOK_SECRET_REF` for the preferred SecretRef-backed path, or legacy `AICHESTRA_GITHUB_WEBHOOK_SECRET`
- optional `AICHESTRA_GITHUB_WEBHOOK_ALLOWED_REPOS=owner/repo,...`
- optional real integration gate `AICHESTRA_GITHUB_WEBHOOK_INTEGRATION_TESTS=true`
- `AICHESTRA_GITHUB_WEBHOOK_ACCEPT_UNVERIFIED=false` by default

If webhooks are disabled, the receiver returns a blocked result. If the secret is missing or resolution is blocked, real webhook processing is blocked before dispatch. Config, health, dashboard, and audit responses expose only booleans/status/source metadata, never the secret value.

## Webhook Verifier Design

Webhook verification is isolated in `packages/adapters/src/git/github-webhooks.ts`:

- `GitHubWebhookVerifier` is the provider boundary.
- `NoopGitHubWebhookVerifier` is the disabled/default path.
- `MockGitHubWebhookVerifier` supports deterministic tests.
- `HmacGitHubWebhookVerifier` verifies `x-hub-signature-256` using HMAC SHA-256 and constant-time comparison.

Raw request bodies are used only for signature verification and hashing. The durable event stores `payloadHash`, delivery id, event type, action, repo ref, verification status, and sanitized metadata. Raw webhook payloads are not persisted.

## Supported Events

The v2 receiver recognizes:

- `ping`
- `pull_request` actions `opened`, `synchronize`, `reopened`, and `closed`
- `push` for non-destructive branch metadata sync
- `pull_request_review`, `check_suite`, `check_run`, and `status` as read-only recognized events

Unsupported events or unsupported PR actions are ignored with audit. Malformed payloads and unverified payloads are rejected with audit.

## PR Sync Flow

`GitHubSyncService.syncPullRequestFromWebhook` updates `GitPullRequestSyncState` from a verified `pull_request` payload. It captures:

- repo ref and optional repo id
- PR number/provider id
- state, head/base branch, latest SHA, labels, and mergeable state
- changed-file paths when present in a webhook fixture or later refresh
- mapping to existing `PullRequest`, `TaskRun`, `BranchLease`, and `MergeQueueEntry` when possible

Unmapped external PRs still create sync records instead of failing. This keeps webhook ingestion non-destructive and observable even before Aichestra owns the PR.

## Branch Sync Flow

PR webhooks update the head branch sync state. `push` webhooks update branch metadata from `refs/heads/*`, including `exists=false` for deleted remote refs. This records metadata only; it does not delete local or remote branches.

## Workspace Lease Linkage

Workspace lifecycle metadata can reference a Git/Conflict Manager `BranchLease` through `branchLeaseId`. The link allows dashboard/readiness views to show which per-agent-run workspace lease belongs to which branch lease. It does not allocate a Git worktree and does not call GitHub or local Git.

Future Git worktree allocation remains out of scope for Real Git Adapter v2. A future Agent Worktree Allocation milestone must require repo allowlists, workspace root allowlists, branch lease ownership, Auth/RBAC and Policy allow decisions, path traversal rejection, no shared root, and explicit cleanup policy before any `git worktree` command is allowed.

Multi-session coordination reads branch and workspace lease identifiers as metadata. Same branch produces high severity coordination risk; same workspace produces a critical blocker; same file/directory produces early review or split-file recommendations. Merge Queue Policy v2 consumes this metadata as readiness evidence only. These recommendations do not perform provider merge/rebase, update remote PRs, or change Git adapter execution behavior in v2.

Branch Orchestrator v2 owns branch allocation/ownership metadata for concurrent users and sessions. It may create or link a `BranchLease` record through existing in-memory repository abstractions, but it never asks the Git provider to create/delete/switch branches and never mutates a working tree. Same active branch ownership is blocked, shared active workspace lease ids are blocked, and base branch drift is represented as review metadata.

Cross-session file leases and edit intents also read branch names and workspace lease ids as metadata. Same branch produces a high severity edit overlap assessment, same workspace produces a critical blocker, and same file produces a serialize recommendation. Merge Queue Policy v2 consumes edit overlap summaries for readiness decisions and holds. These assessments do not mutate branch leases, execute merges, update PR sync state, update branch sync state, or change Git adapter execution behavior.

## Changed-Files Refresh Flow

`POST /git/repos/:id/pull-requests/:number/refresh-changed-files` may call `GitIntegrationService.getPullRequestChangedFiles`, which still flows through the v1 `GitHubClient` boundary. The refresh requires policy and v1 remote Git gates before any GitHubClient call. Refreshed files update PR sync state and branch lease file inputs, then refresh merge queue risk fields non-destructively.

## Policy Integration

Policy-as-code includes v2 actions:

- `git.webhook.receive`
- `git.webhook.verify`
- `git.webhook.process`
- `git.pull_request.sync`
- `git.branch.sync`
- `git.changed_files.read`

Default policy denies unverified webhook receive/process and keeps `git.merge` and `git.rebase` denied. Changed-file refresh requires remote Git, repo allowlist, credentials, and a non-destructive operation context.

Production Auth/RBAC Planning v0 adds a mock-first actor context boundary for Git operations. RequestContext Propagation v1 adds request/correlation metadata to branch creation, PR creation, changed-file reads, GitHub App token-handle checks, selected Git policy decisions, and selected Git audit events. Service Account Actor Boundary v1 replaces high-value Git service fallbacks with `git_provider_service`, `git_webhook_service`, `git_sync_service`, and `github_app_token_service` attribution. Tenant/Repo/Provider Scope Model v1 adds additive `RepoScope` metadata to migrated Git policy resources and audit records. SecretRef-backed credential resolution now uses `AuthorizationService` when resolving GitHub tokens or webhook secrets through SecretRef. Auth roles, service accounts, and scope metadata do not bypass Git policy denials: merge, rebase, force push, branch deletion, and unverified webhook processing remain blocked.

Production Auth/RBAC v1 Planning adds read-only service-account and webhook actor readiness models for future GitHub webhook and Git provider actors. It does not enable production webhooks, GitHub App installation tokens, real service-account credentials, tenant enforcement, merge, rebase, force push, branch deletion, or GitHub calls.

## Secrets and Sandbox Integration

Webhook secrets prefer the SecretRef-backed credential path when `AICHESTRA_GITHUB_WEBHOOK_SECRET_REF` is set. Use `secretKind=github_webhook_secret` or `webhook_secret` with the explicit env provider and env-key allowlist. The legacy env secret path is kept as a gated compatibility path. The secret value is never returned through API, health, dashboard, policy audit, Git audit, webhook audit, or errors.

No runner secret injection, Local Agent secret forwarding, credential cache reads, vendor CLI execution, sandbox runtime integration, or network egress change is added by v2.

## Audit Events

Webhook-specific audit records are stored in `git_webhook_audit_events` and mirrored to generic Git audit with sanitized metadata:

- `github_webhook_received`
- `github_webhook_disabled`
- `github_webhook_signature_verified`
- `github_webhook_signature_rejected`
- `github_webhook_unsupported_event`
- `github_webhook_payload_rejected`
- `github_webhook_processed`
- `github_pr_sync_started`
- `github_pr_sync_completed`
- `github_pr_sync_failed`
- `github_branch_sync_started`
- `github_branch_sync_completed`
- `github_branch_sync_failed`
- `github_changed_files_refresh_started`
- `github_changed_files_refresh_blocked`
- `github_changed_files_refresh_completed`
- `github_merge_attempt_blocked`
- `github_rebase_attempt_blocked`

Audit metadata must not contain webhook secrets, GitHub tokens, raw credentials, unredacted raw payloads, or credential cache paths. Selected Git audit records include sanitized requestId, correlationId, actorId, principalId, authMode, and source.

## Health and Dashboard

`GET /health` reports provider kind, remote Git gates, webhook enabled status, webhook secret configured boolean, accept-unverified flag, supported event count, allowed repo count, and secret status. It never returns secrets or tokens.

`/dashboard/git` includes webhook config, recent webhook events, webhook audit events, PR sync states, branch sync states, changed-files refresh status, blocked unverified examples, merge queue policy linkage, and merge/rebase disabled safety flags.

Branch Orchestrator v2 extends `/dashboard/git` and `/git/branches/orchestration*` with branch requests, decisions, ownership records, naming policy, collision blockers, same-workspace blockers, base drift status, merge readiness counts, branch orchestrator audit metadata, and explicit no-destructive-Git/no-remote-Git/no-workspace-mutation status.

GitHub App / Production Webhook Hardening Planning v0 adds `/readiness/github-app/*` and `/dashboard/github-app` planning read models for least-privilege app permissions, webhook event allowlists, replay classification, retry/dead-letter planning, credential readiness, production endpoint readiness, and risks. As an audit follow-up, the v2 receiver also performs local delivery-id idempotency: duplicate deliveries with the same payload hash are ignored, and duplicate delivery ids with a different payload hash are rejected as replay attempts. This remains in-process/runtime-local and is not a production-grade distributed replay cache.

Persistent DB Production Operations v1 adds `/readiness/database/*` and `/dashboard/database` planning read models for durable webhook delivery/dead-letter persistence, index review, retention/audit growth, backup/restore, and pooling requirements. These are planning/readiness surfaces only and do not enable production webhooks, durable replay stores, queue workers, or destructive Git operations.

Secret Backend Migration Planning v0 adds `/readiness/secrets/*`, `/dashboard/secret-backend`, and `/health` metadata for future migration of GitHub token, GitHub webhook secret, and GitHub App private key SecretRefs to a real backend. It does not read private keys, mint installation tokens, call GitHub, rotate secrets, or enable production webhooks.

Vault-backed Secret Backend v1 allows GitHub token, webhook secret, and GitHub App private-key SecretRefs to use `provider: vault` when every Vault gate and path allowlist is configured. This does not change Git behavior: remote Git, webhook processing, and GitHub App operations still require their existing gates, and v2 still does not auto-merge, force-push, delete branches, sign GitHub App JWTs, or exchange live installation tokens by default.

Production Auth/RBAC v1 Planning adds `/readiness/auth/*`, `/dashboard/auth-production`, and `/health` metadata for future GitHub service account and webhook actor scoping. It exposes no tokens, cookies, session ids, raw identity assertions, or provider credentials.

GitHub App Controlled Implementation v1 adds `/git/github-app/*` runtime/status endpoints for the controlled GitHub App boundary:

- `GET /git/github-app/config`
- `GET /git/github-app/installations`
- `GET /git/github-app/repository-grants`
- `POST /git/github-app/validate`
- `POST /git/github-app/installations/:id/token/check`
- `GET /git/github-app/audit`

These endpoints return sanitized config, installation, repository grant, validation, token-check, and audit metadata only. They never expose private keys, installation tokens, webhook secrets, legacy PATs, or raw credential values, and default runtime/tests do not call GitHub.

## Integration Test Setup

Default tests use `MockGitHubWebhookVerifier`, HMAC fixtures, and a mocked `GitHubClient`. Real webhook integration tests are skipped unless every explicit webhook env gate is configured. Default tests do not require a public webhook URL and do not call GitHub.

## Known Limitations

- This is not production-ready Git integration.
- GitHub App / Production Webhook Hardening Planning v0 and Persistent DB Production Operations v1 are implemented as read-only planning data. GitHub App Controlled Implementation v1 adds mock/status-only runtime token-handle checks, but no live GitHub App installation flow, private-key signing, real installation token exchange, production webhook endpoint, durable delivery replay store, background retry worker, or production webhook dead-letter worker is implemented.
- Runtime webhook duplicate/replay checks are local to the current receiver store. Production traffic still requires durable shared idempotency storage, queue-backed processing, rate limits, alerting, and operator review tooling.
- Webhook payload storage is metadata-only; raw payload archiving is out of scope.
- Changed-file refresh can call GitHub only through existing v1 gates.
- PR state sync does not perform reviewer, branch cleanup, merge, rebase, or workflow execution.
- Agent Workspace Lifecycle v2 branch lease linkage is metadata-only; it does not create worktrees or change Git adapter execution behavior.
- Multi-session Agent Run Coordination v1 branch/workspace/session linkage is metadata-only; Merge Queue Policy v2 can consume it for readiness decisions, but it does not create branches, allocate worktrees, or execute merges.
- Multi-user / Multi-session Branch Orchestrator v2 branch ownership is metadata-only; Merge Queue Policy v2 can consume linkage and blockers, but it does not create real branches, inspect remote branch existence, run drift checks, allocate worktrees, or execute merges.
- Cross-session File Lease / Edit Intent Graph v1 file/branch/workspace linkage is metadata-only; Merge Queue Policy v2 can consume edit overlap summaries, but it does not lock files, mutate source files, run Git, or execute merges.
- Merge Queue Policy v2 remains metadata-only and mock-first. It records decisions and holds, ranks queue entries, and denies future merge execution by default.
- SecretRef-backed webhook secrets may use the explicit env provider or the gated Vault v1 provider. Cloud secret managers remain future work.
