# Real Git Adapter v1

## What v1 Adds

Real Git Adapter v1 adds controlled GitHub branch, pull request, and PR changed-file operations behind explicit environment gates, repo allowlists, branch-prefix constraints, policy checks, and sanitized audit records.

The default runtime remains mock-first. Default tests do not call GitHub.

## Safety Gates

Remote Git remains disabled unless all required gates are configured:

- `AICHESTRA_GIT_PROVIDER=github`
- `AICHESTRA_ENABLE_REMOTE_GIT=true`
- `AICHESTRA_ALLOW_REMOTE_BRANCH_CREATE=true` for branch creation
- `AICHESTRA_ALLOW_REMOTE_PR_CREATE=true` for PR creation
- `AICHESTRA_GITHUB_TOKEN` for gated GitHub API access
- `AICHESTRA_GITHUB_OWNER` and `AICHESTRA_GITHUB_REPO` for validation defaults
- `AICHESTRA_GITHUB_ALLOWED_REPOS` containing the target `owner/repo`
- `AICHESTRA_GITHUB_ALLOWED_BRANCH_PREFIX`, default `ai/`
- `AICHESTRA_GITHUB_INTEGRATION_TESTS=true` only for optional real integration tests

`AICHESTRA_ALLOW_REMOTE_MERGE` remains unsupported in v1. Merge, rebase, force push, branch deletion, remote push, webhook handling, GitHub App installation, GitLab, and Bitbucket are out of scope.

## GitHubClient Boundary

GitHub HTTP behavior is isolated behind `GitHubClient` in `packages/adapters/src/git/github-client.ts`.

The service layer depends on `GitProvider`/`GitIntegrationService` only. It does not call `fetch`, Octokit, SDKs, or raw HTTP. `FetchGitHubClient` is created only from gated configuration. `NoopGitHubClient` is the safe default when gates are missing.

## Branch Creation Flow

`POST /git/repos/:id/branches/remote` calls `GitIntegrationService.createRemoteBranch`.

The flow validates:

- remote Git enabled
- branch creation enabled
- token configured
- repo allowlisted
- branch name starts with the allowed prefix
- policy allows `git.remote_operation`
- policy allows `git.branch.create`

On success, the provider creates the branch from the base branch SHA, the service records branch metadata/lease linkage when task/taskRun ids are present, and audit events include `git.github_branch_created`.

## Pull Request Flow

`POST /git/repos/:id/pull-requests/remote` calls `GitIntegrationService.createRemotePullRequest`.

The flow validates:

- remote Git enabled
- PR creation enabled
- token configured
- repo allowlisted
- head branch prefix allowed
- policy allows `git.remote_operation`
- policy allows `git.pull_request.create`

On success, the service stores a durable `PullRequest` record and links a merge queue entry when a taskRun and branchLease are supplied. v1 does not auto-merge, request reviewers, or mutate the branch after PR creation.

## Changed Files Flow

`GET /git/repos/:id/pull-requests/:number/changed-files` reads PR changed files through `GitHubClient.getPullRequestChangedFiles` only when remote Git and repo allowlist gates pass. File paths/statuses are sanitized before returning or auditing. Changed files can feed conflict-risk inputs where the current architecture already accepts file lists.

## Policy Integration

Policy-as-code v0 now has conditional v1 rules:

- `git.remote_operation` is denied unless remote Git, operation-specific gates, repo allowlist, and credentials pass.
- `git.branch.create` allows GitHub only when branch-create, allowlist, prefix, and credential gates pass.
- `git.pull_request.create` allows GitHub only when PR-create, allowlist, prefix, and credential gates pass.
- `git.merge` and `git.rebase` remain denied.

Policy decisions are audited and cannot be bypassed by API routes.

## Secrets and Sandbox

The GitHub token is read only from environment/config for this gated integration. It is never stored in repos, pull request records, health responses, dashboard data, policy audit, or Git audit metadata.

Secrets/Sandbox v0 remains metadata-only. Moving the GitHub credential to `SecretRef`/SecretManager is a future task; v1 does not issue secret leases or inject secrets into runners or Local Agents.

## Audit Events

Git audit includes:

- `git.remote_git_config_validated`
- `git.github_connection_validated`
- `git.github_connection_blocked`
- `git.github_branch_create_requested`
- `git.github_branch_create_blocked`
- `git.github_branch_created`
- `git.github_pr_create_requested`
- `git.github_pr_create_blocked`
- `git.github_pr_created`
- `git.github_changed_files_requested`
- `git.github_changed_files_blocked`
- `git.github_changed_files_read`
- `git.github_merge_attempt_blocked`
- `git.github_rebase_attempt_blocked`

Metadata includes sanitized actor/task/run/repo/branch/PR/policy ids where available. Tokens and credential-like strings are redacted.

## API and Health

Added or extended endpoints:

- `POST /git/github/validate`
- `POST /git/repos/:id/branches/remote`
- `POST /git/repos/:id/pull-requests/remote`
- `GET /git/repos/:id/pull-requests/:number/changed-files`
- `GET /git/remote/audit`

`GET /health` now reports Git provider kind, remote Git gates, GitHub configured status, allowed repo count, branch prefix, integration-test gate, and `remoteMergeEnabled: false`. It never returns the token.

## Dashboard

The dashboard shows remote Git status, GitHub configured/unconfigured state, branch/PR gates, allowed branch prefix, allowed repo count, remote Git audit events, blocked examples, branch/PR records, and merge queue linkage. It does not expose tokens.

## Tests

Coverage in `tests/real-git-adapter-v1.test.ts` verifies:

- default gates are disabled;
- branch/PR creation blocks before client calls when gates are missing;
- mocked GitHub client can create branches and PRs;
- durable PR/branch lease/merge queue linkage is recorded;
- changed files are read through a mocked client;
- token-like strings are redacted from results and audit;
- merge/rebase remain unsupported and policy-audited;
- API remote endpoints are blocked by default;
- optional real GitHub integration tests skip unless every explicit env var is configured.

## Known Limitations

- This is not production-ready Git integration.
- GitHub App installation, GitLab, Bitbucket, webhooks, reviewer automation, branch cleanup, merge, rebase, force push, and branch deletion are not implemented.
- Real GitHub integration tests intentionally leave remote branches/PRs in the configured repo; cleanup is a future controlled fixture task.
- Git audit still uses the generic `audit_logs` store.
