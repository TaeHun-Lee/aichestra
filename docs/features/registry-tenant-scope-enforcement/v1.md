# Registry Tenant Scope Enforcement v1

Status: `v1_implemented_partial`

Registry Tenant Scope Enforcement v1 adds limited, mock-safe tenant scope decisions for registry resources. It applies the existing Tenant/Repo/Provider Scope Model and Tenant Scope Enforcement helper to Skills, Harnesses, Instructions, registry packages, resolver results, approval queue metadata, audit/readiness surfaces, and representative mutation checks.

This is not production registry tenant isolation. Production tenant provisioning, durable grants, row-level security, production Auth/RBAC, and production dashboard/readiness filtering remain out of scope.

## What v1 Implements

- `packages/registry/src/scope-enforcement.ts`:
  - `RegistryScopeDecision`
  - `RegistryScopeEnforcementSummary`
  - `RegistryTenantScopeEnforcementService`
  - registry resource input mappers for skills, harnesses, instructions, and package manifests
- Resolver integration:
  - `RegistryResolution.scopeDecisions`
  - `RegistryResolution.scopeSummary`
  - resolver lifecycle/approval/eval/checksum/semver gates remain authoritative
- Approval queue and dashboard read-model summaries for missing-scope warnings and representative out-of-scope denials.
- Mutation scope checks in the registry service boundary. v1 checks metadata and can block explicitly denied representative sensitive cases, but it does not weaken existing registry gates.
- Read-only/evaluation API endpoints:
  - `GET /registry/scope/summary`
  - `POST /registry/scope/evaluate`
  - `GET /registry/scope/decisions`
  - `GET /readiness/registry/scope/summary`
- Dashboard registry panel rows for registry scope summary, approval queue scope, resolver scope status, and mutation scope status.
- Policy actions/resources:
  - `registry.scope.read`
  - `registry.scope.evaluate`
  - `registry.scope.enforce_future`
  - `registry.audit.read`
  - `registry.approval_queue.read`
  - `registry.mutation.scope_check`
  - `registry_scope` resource kind

## What v1 Does Not Implement

- No production tenant provisioning.
- No database row-level security.
- No production Auth/RBAC, sessions, JWTs, API keys, or service-account credential issuance.
- No external IdP, LLM, MCP, Git, Vault, registry, or provider calls.
- No active registry mutation through auto-improvement.
- No resolver override, registry gate bypass, or auto-apply.
- No production dashboard/readiness filtering claim.

## Registry Scope Decision Model

`RegistryScopeDecision` records:

- registry resource kind: skill, harness, instruction, package, eval result, approval queue, history, or audit;
- resource id plus optional tenant/team/project/repo/provider ids;
- decision: `in_scope`, `out_of_scope_warning`, `out_of_scope_denied`, `missing_scope_warning`, `missing_scope_denied`, or `not_applicable`;
- enforcement mode: `metadata_only`, `warning`, `deny_sensitive`, or `future_production`;
- reason plus request/correlation/actor/service-account metadata when available;
- sanitized metadata with explicit `productionEnforcement: false`, `policyDenyStillWins: true`, and `resolverGatesPreserved: true`.

## Resolver Integration

The registry resolver attaches scope decisions to selected refs only as metadata. Scope decisions do not make excluded candidates selectable. Pending approval, failed eval, non-active lifecycle, checksum mismatch, unsupported semver, dependency errors, and policy deny remain authoritative.

Tests assert that pending approval stays excluded and checksum mismatch stays blocked while scope metadata is present.

Registry Signed Package / Artifact Trust v1 attaches a separate trust decision layer for package digest, mock signature, and provenance metadata. Scope decisions remain about tenant/team/project/repo/provider visibility and mutation context; artifact trust decisions remain about package provenance and future signed-bundle readiness. Neither layer bypasses registry resolver gates, policy denies, checksum mismatch blockers, or governance apply gates.

## Registry Reads And Approval Queue

Dashboard/readiness and API scope summaries evaluate current skills, harnesses, instructions, package manifests, and approval queue entries. General reads produce metadata/warnings by default. Missing tenant or registry package scope never grants broader access; it is surfaced as `missing_scope_warning` unless an explicit sensitive representative check uses `deny_sensitive`.

## Mutation Scope Check Behavior

Registry mutations still use existing registry role and optional Policy-as-code authorization first. Scope checks run as metadata-side checks and can block only when the scope service returns a denied representative decision. A scope allow does not override policy deny or registry lifecycle/approval/eval/checksum gates.

## Policy/Auth Behavior

Default policy allows safe mock/readiness `registry.scope.read`, `registry.scope.evaluate`, and `registry.mutation.scope_check` when metadata-only safety flags are present. `registry.scope.enforce_future` is denied by default.

Mock AuthContext, RequestContext, and service-account actor metadata may be used for attribution, but they are not production auth claims.

## Dashboard And API Surfaces

The dashboard registry panel shows:

- total scope resources;
- in-scope/warning/denied/missing-scope counts;
- approval queue scope summary;
- resolver scope status with `resolverGatesPreserved: true`;
- mutation scope status with `policyDenyStillWins: true`;
- `productionEnforcement: false`.

API and readiness endpoints return sanitized metadata only. They do not mutate registry resources and do not expose secrets or env values.

## Eval Suite Execution Harness Relationship

Eval Suite Execution Harness v1 records eval suite/run/verdict metadata on registry targets and proposal eval requirements. Registry Tenant Scope Enforcement v1 can scope registry reads, resolver results, approval queues, audit/readiness queries, and mutation checks around that metadata, but it does not claim production tenant isolation for eval evidence.

## Production Enforcement Gaps

- Durable tenant/team/project/repo/provider/package grants.
- Production Auth/RBAC and trusted scope claims.
- Repository query filters and row-level security.
- Dashboard/readiness/audit filtering.
- Cache partitioning and tenant-scoped persistence.
- Signed package trust and production package visibility rules.

## Safety Constraints

- No external provider calls.
- No real LLM/MCP/Git/Vault calls.
- No secrets, env values, tokens, credential cache paths, or raw provider credentials.
- No active registry mutation through auto-improvement.
- No resolver gate weakening.
- No production tenant isolation claim.

## Known Limitations

- Enforcement is representative and metadata-first.
- General registry reads warn rather than filter.
- Denial is limited to explicit sensitive representative paths.
- Scope metadata is in-memory/mock and not a durable authorization grant.
