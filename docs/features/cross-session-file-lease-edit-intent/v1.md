# Cross-session File Lease / Edit Intent Graph v1

Status: `v1_implemented`

Cross-session File Lease / Edit Intent Graph v1 adds metadata-only file leases, edit intents, graph edges, and overlap assessments for concurrent LLM/agent sessions. It helps Aichestra warn, serialize, split, review, or block overlapping planned edits before merge queue time without locking real files or mutating source files.

Durable Collaboration Stores v1 now adds a safe repository/schema boundary for `FileLease`, `EditIntent`, and `EditOverlapAssessment` records. The default edit-intent service remains in-memory/mock-first; distributed locks, production tenant isolation, and production DB operations remain future work.

## Why This Exists

Separate branches and per-agent workspaces prevent shared working tree branch-switch interference, but they do not prevent two sessions from planning to edit the same file or a tightly related directory. Conflict Manager v1 can detect file overlap and dry-run merge outcomes later. Multi-session Agent Run Coordination v1 can compare declared target files. This v1 adds an explicit edit-intent graph so file ownership and planned edits are visible earlier and can be shown in API/dashboard/readiness metadata.

## What v1 Implements

- `FileLease`, `EditIntent`, `EditIntentGraphNode`, `EditIntentGraphEdge`, and `EditOverlapAssessment` models in `packages/runner/src/edit-intent.ts`.
- `EditIntentGraphService` with deterministic in-memory storage.
- File lease request and release metadata.
- Edit intent declaration metadata.
- Session-to-file, session-to-directory, session-to-branch, session-to-workspace, and session-to-session graph construction.
- Same-file, same-directory, same-branch, same-workspace, read/write, and broad-unknown overlap detection.
- Coordination recommendations: `allow`, `warn`, `split_files`, `serialize`, `require_review`, and `block`.
- Safe API endpoints under `/agents/*`.
- Dashboard Agent Runner panel visibility.
- DTO sanitization for absolute paths and secret-like metadata.

## What v1 Does Not Implement

- No OS-level file locks.
- No source file reads, writes, parsing, or mutation.
- No real Git worktree creation, checkout, switch, fetch, push, merge, rebase, force-push, or branch deletion.
- No vendor CLI execution.
- No LLM provider calls.
- No external provider calls.
- No production-ready durable lock manager or production DB operations.
- No distributed locks.
- No symbol-aware language server analysis.
- No merge execution change. Merge Queue Policy v2 consumes edit overlap summaries as metadata-only readiness evidence.
- No production tenant filtering or production auth behavior.

## File Lease Model

`FileLease` records:

- `id`
- `repoId`
- normalized `filePath`
- optional `filePathHash`
- `leaseKind`: `read`, `write_intent`, `exclusive_write_future`, or `review`
- owner session, agent run, task, actor, branch, and optional workspace lease ids
- `status`: `requested`, `active`, `warning_overlap`, `blocked_overlap`, `released`, or `expired`
- timestamps, optional expiry, and sanitized metadata

The lease is advisory metadata. `exclusive_write_future` is recorded for future enforcement design but is not globally enforced in v1.

## Edit Intent Model

`EditIntent` records:

- `repoId`, `sessionId`, optional `agentRunId`, optional `taskId`
- `branchName` and optional `workspaceLeaseId`
- `intentKind`: `modify`, `create`, `delete_future`, `rename_future`, `read_only`, `refactor`, `test_update`, or `docs_update`
- target `filePaths`
- `directoryScopes`
- optional future `declaredSymbols`
- confidence and lifecycle status
- sanitized metadata

Read-only intents can overlap write intents as low-severity warnings. Write-like intents on the same file serialize.

## Graph Model

Graph nodes represent:

- sessions
- files
- directories
- branches
- workspaces
- tasks

Graph edges represent:

- intended edits
- file leases
- overlaps
- same directory
- same branch
- same workspace
- future symbol overlap

Graph ids are deterministic from metadata. The graph is recomputed from in-memory records and does not inspect the filesystem.

## Overlap Assessment

The service evaluates active intents and leases per repo:

- same exact write target: high severity, `serialize`
- read-only plus write target: low severity, `warn`
- same directory: medium severity, `split_files`
- same directory with broad refactor: high severity, `require_review`
- same branch: high severity, `serialize`
- same workspace: critical severity, `block`
- missing target files or broad unknown scope: medium severity, `warn`

`FileLease.status` is updated to `active`, `warning_overlap`, or `blocked_overlap` based on the latest metadata assessment. Releasing a file lease removes it from active lease overlap evaluation.

## API Surface

Endpoints:

- `POST /agents/edit-intents`
- `GET /agents/edit-intents`
- `POST /agents/file-leases`
- `POST /agents/file-leases/:id/release`
- `GET /agents/file-leases`
- `GET /agents/edit-intent-graph`
- `GET /agents/edit-overlaps`
- `GET /agents/edit-overlaps/:id/recommendation`
- `GET /agents/edit-intent-summary`

These endpoints update in-memory metadata only. They do not lock files, mutate source files, run Git, call providers, or expose secrets/env values.

## Coordination, Branch, Workspace, And Merge Queue Interaction

Multi-session Agent Run Coordination v1 remains responsible for session grouping and existing same-file/same-directory/session warnings. The edit-intent graph is a more explicit file lease and graph surface that can consume the same session, branch, task, and workspace lease ids.

Agent Workspace Lifecycle v2 remains responsible for workspace lease lifecycle and cleanup decisions. Same workspace edit intent overlap is critical and recommends `block`, but this feature does not allocate or delete workspaces.

Real Git Adapter v2 and branch orchestration metadata remain separate. The edit-intent graph records branch names and can surface same-branch risks, but it does not create branches or call Git. Merge Queue Policy v2 consumes edit overlap summaries for readiness decisions and holds. Conflict Resolution Assistant v1 consumes the same metadata to classify conflicts and generate review-only resolution plans. v1 does not mutate merge queue execution state.

## Dashboard And Readiness

The Agent Runner dashboard read model now includes:

- `editIntents`
- `fileLeases`
- `editIntentGraph`
- `editOverlapAssessments`
- `editIntentSummary`

The rendered dashboard shows active intent/lease counts, graph node/edge counts, same-file warnings, same-directory warnings, same-workspace conflicts, and explicit `noFileLocks` / `noSourceMutation` status.

`GET /health` exposes safe Agent Runner metadata for edit intent graph status, active edit intent count, active file lease count, file locking disabled, and source mutation disabled.

## Safety Constraints

Default runtime/tests must not:

- lock real files;
- mutate source files;
- delete workspaces or branches;
- run remote Git;
- execute vendor CLI;
- call LLM providers or external providers;
- read credential caches;
- expose secrets/env values;
- weaken Git, Runner, Local Agent, Auth/RBAC, Policy, SecretRef, Tenant Scope, Dashboard, Observability, or Safety gates.

DTOs redact secret-like metadata and full absolute paths.

## Known Limitations

- In-memory storage only.
- No distributed lock or durable lease table.
- No production tenant filtering.
- No symbol-aware or semantic conflict analysis.
- No source AST parsing.
- No merge execution or provider mutation. Merge Queue Policy v2 consumes edit-intent overlap metadata for readiness decisions and holds, and Conflict Resolution Assistant v1 consumes it for review-only summaries/plans.
- No real branch/workspace allocation.
- Real Merge Execution Policy v1 consumes high/critical edit-overlap metadata as future execution blockers, but it still does not lock files, parse source, mutate source, or execute merges.
