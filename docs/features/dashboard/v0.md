# Dashboard API-backed Read Model Migration v0

## What Changed

Dashboard API-backed Read Model Migration v0 adds a read-only dashboard data layer:

- shared dashboard read-model DTO types in `packages/shared`;
- sanitized API read-model assembly in `apps/api/src/dashboard-read-model.ts`;
- `/dashboard/*` API endpoints for overview, tasks, Git, conflicts, registry, LLM, agents, policy, auth/RBAC, providers, security, Local Agent, MCP Gateway, production readiness, GitHub App hardening and controlled runtime status, observability, database operations, secret backend migration, staging deployment readiness, and audit sections;
- a web `DashboardDataProvider` abstraction with API and demo providers;
- dashboard rendering that consumes read models instead of directly constructing services;
- deterministic tests for API read models, provider mapping, fallback behavior, rendering, and no-secret exposure.

## Read Model Architecture

The API read-model assembler composes current service and repository state only. It does not create tasks, run workflows, evaluate new policy decisions, request secret leases, execute agent runs, create Local Agent fixture invocations, call GitHub, or call LLM providers.

The web layer consumes `DashboardReadModels` through:

- `ApiDashboardDataProvider`: fetches `/dashboard/*` endpoints.
- `DemoDashboardDataProvider`: builds deterministic fixture read models from the existing demo data path for static tests/offline rendering.

Runtime API mode can be enabled with:

```bash
AICHESTRA_DASHBOARD_DATA_SOURCE=api
AICHESTRA_DASHBOARD_API_BASE_URL=http://127.0.0.1:3000
```

If API mode is configured and the API is unavailable, demo fallback is used unless:

```bash
AICHESTRA_DASHBOARD_DISABLE_DEMO_FALLBACK=true
```

## Safety Rules

- No GitHub calls.
- No LLM provider calls.
- No vendor CLI execution.
- No runner command execution.
- No workflow execution.
- No secret lease issuance.
- No MCP real transport or tool invocation from dashboard read routes.
- No credential cache reads.
- No production DB connection, migration execution, backup/restore job, destructive retention job, or database URL exposure.
- No real secret backend connection, secret migration, rotation job, credential issuance, or env value exposure.
- No staging deployment, infrastructure manifest apply, production traffic, external provider call, remote MCP, vendor CLI execution, or staging env value exposure.
- No active CI workflow creation, CI job execution, deployment job, default remote integration test execution, artifact upload, raw CI log upload, or CI env value exposure.
- No GitHub App integration-test profile live execution, env value exposure, private-key read, token exchange, GitHub call, auto-merge, force-push, or branch deletion from dashboard read routes.
- No real auth provider connection, login/logout/session/JWT/password flow, service-account credential issuance, token/cookie/session/assertion exposure, or external IdP call.
- No GitHub App private-key read/signing, live installation token exchange, or token/private-key/webhook-secret exposure.
- No raw unredacted logs.
- No tokens, provider API keys, or credential values in API/web output.

`sanitizeDashboardValue` masks token-like strings, sensitive env assignments, and credential-cache paths before read models are returned or rendered.

## Fallback Strategy

Demo fallback remains for deterministic static tests and offline local preview. It is explicit in `DemoDashboardDataProvider` and should not be treated as production dashboard state.

API-backed mode is available through `ApiDashboardDataProvider`; it fetches all `/dashboard/*` sections and maps them to the shared read-model DTOs.

## Known Limitations

- Dashboard read models expose Auth/RBAC v0 mock actor context, but they are not production tenant/RBAC scoped.
- Auth/RBAC Production Readiness read models are planning-only; they do not implement real auth, sessions, IdP calls, tenant isolation, or service-account credential issuance.
- Empty API state can produce empty sections until workflows or API actions create records.
- Demo fallback still runs deterministic fixture workflows for static/offline tests.
- Phase 4 improvement/governance read models surface existing in-memory state only.
- No streaming UI or production analytics layer is implemented.
- Database operations read models are planning/readiness metadata only; they do not make persistent DB operations production-ready.
- Secret backend migration read models are planning/readiness metadata only; they do not implement production secret management.
- GitHub App controlled runtime read models are mock/status metadata only; they do not implement live installation discovery, private-key signing, or installation-token exchange.
- GitHub App integration-test read models are readiness metadata only; they do not run live tests, call GitHub, exchange installation tokens, or expose env values.
- Staging deployment read models are planning/readiness metadata only; they do not deploy staging or make staging production-ready.
- Staging CI/CD read models are planning/readiness metadata only; they do not create active workflows, run jobs, upload artifacts, run remote integrations by default, deploy staging, or make staging production-ready.
