# Branch Cleanup / Orphan Lease Recovery v1

Status: v1_implemented

Branch Cleanup / Orphan Lease Recovery v1 is implemented as a mock-first, metadata-only readiness layer for detecting and proposing safe cleanup of orphaned branch leases, workspace leases, worktree allocations, merge queue entries, agent sessions, and PR ownership handoffs. The default runtime never deletes real branches, real worktrees, real PRs, or any filesystem state, and never executes remote Git operations.

## What v1 Implements

- Core models (`packages/git-adapter/src/branch-cleanup.ts`):
  - `OrphanLeaseRecord` with `leaseKind` (`branch | workspace | worktree | merge_queue | pr_ownership | agent_session`), `detectedReason` (`expired | missing_owner | task_completed | task_failed | merge_completed | branch_missing_future | workspace_missing_future | stale_queue_entry | abandoned_session | inconsistent_state`), and severity bucket.
  - `CleanupRecommendation` with `recommendation` (`keep | mark_abandoned | release_metadata_lease | mark_cleanup_pending | require_manual_review | delete_branch_future | remove_worktree_future | close_pr_future | archive_record_future`), `destructive: "false" | "true_future"`, and `requiredApproval`.
  - `CleanupDecision` with decision values `approved_metadata_only | rejected | held | future_destructive_review | blocked_policy | executed_metadata_only`.
  - `RecoveryAction` with action kinds `relink_branch_lease | relink_workspace_lease | refresh_status | recreate_metadata_record | mark_failed | mark_abandoned | manual_reconciliation` and status `proposed | executed_metadata_only | blocked | future`.
  - `BranchCleanupSummary` exposing scan/orphan/recommendation/decision counts plus deterministic safety status (`destructiveCleanupEnabled: false`, `realBranchDeleted: false`, `realWorktreeRemoved: false`, `realPullRequestClosed: false`, `remoteGitCallsExecuted: false`, `filesystemDeletionsExecuted: false`).
- `BranchCleanupRecoveryService` with deterministic detection rules, `InMemoryBranchCleanupRecoveryRepository`, recommendation derivation, decision recording, metadata-only execution, and recovery proposal.
- New policy actions (`packages/policy/src/types.ts`) plus default rules (`packages/policy/src/default-rules.ts`):
  - allow `cleanup.scan`, `cleanup.recommend`, `cleanup.decide`, `cleanup.metadata_execute` under mock-first metadata-only context;
  - deny `cleanup.destructive_execute_future`, `branch.delete_future`, `worktree.remove_future`, `pr.close_future` by default.
- New `cleanup` `PolicyResourceKind`.
- Read-only API endpoints under `/git/cleanup/*`:
  - `GET /git/cleanup/orphans`
  - `GET /git/cleanup/recommendations`
  - `GET /git/cleanup/summary`
  - `GET /git/cleanup/recovery-actions`
  - `GET /git/cleanup/decisions`
  - `POST /git/cleanup/scan`
  - `POST /git/cleanup/recommendations/:id/decision`
  - `POST /git/cleanup/metadata-execute`
  - `POST /git/cleanup/orphans/:id/propose-recovery`
- `/health` metadata key `branchCleanup` exposing safety status and counts.
- Dashboard panel `/dashboard/git-cleanup` and `BranchCleanupReadModel` (summary, orphans, critical/high-severity orphans, recommendations, destructive future vs. metadata-only counts, decisions, recovery actions, policy status, and no-destructive status).
- Demo dashboard data provider entry and rendered HTML panel `Branch Cleanup / Orphan Lease Recovery`.
- Deterministic tests (`tests/branch-cleanup-orphan-lease-recovery-v1.test.ts`) covering all detection rules, recommendation/decision/execution pathways, policy deny/allow defaults, API endpoints, dashboard panel, and the no-secret/no-env guarantees.

## What v1 Does Not Implement

- Real branch deletion (`git branch -D`, `git push --delete`).
- Real worktree removal (`git worktree remove`, `rm -rf`).
- Real PR closure (no provider PR-close API call).
- Filesystem deletion outside fixture tests.
- Real fetch/push/merge/rebase/force-push.
- Real LLM calls or vendor CLI execution.
- Production tenant enforcement of cleanup scope (Tenant Scope Enforcement remains partial).
- PR Ownership Handoff Model v1 itself; cleanup hooks into the future model only.
- Durable cleanup record storage; records remain in-memory in v1.

## Orphan Lease Model

`OrphanLeaseRecord` captures the lease kind, related id, repo, branch name (if applicable), workspace lease id, sanitized worktree path label (never a raw user path), owner actor/service-account, the detected reason, severity, detection timestamp, and sanitized metadata. The detection rules are deterministic and described in `v1-plan.md`.

## Cleanup Recommendation Model

`CleanupRecommendation` derives from a single orphan record. Recommendations are non-destructive by default; the four destructive kinds (`delete_branch_future`, `remove_worktree_future`, `close_pr_future`, `archive_record_future`) carry `destructive: "true_future"` and `requiredApproval: release_manager`. Recommendations record safety checks like `no_real_branch_deletion`, `no_real_worktree_removal`, `no_real_pr_closure`, `no_filesystem_deletion`, `no_remote_git`, `no_secret_exposure`.

## Recovery Action Model

`RecoveryAction` records the proposed reconciliation step for an orphan record. Recovery actions remain `proposed` or `future` by default; only metadata-only execution can transition them to `executed_metadata_only` and only when the related cleanup recommendation is non-destructive.

## Detection Rules

Detection is deterministic over the data sources (`BranchCleanupDataSource`). The rules are:

- Expired branch leases (status `expired`, or `active` with `expiresAt` past).
- Branch ownership with missing owner (`active` ownership missing both `ownerActorId` and `ownerServiceAccountId`).
- Workspace lease still active for completed/failed task.
- Workspace `ready_for_merge` without merge queue entry.
- Stale merge queue entry referencing released/expired branch lease, or branch ownership marked `abandoned`/`merged`.
- Abandoned or expired agent sessions.
- Shared worktree referenced by multiple active leases or validated allocations (critical severity).
- Branch merged but workspace still active.
- Workspace `cleanup_blocked_dirty` / `cleanup_blocked_uncommitted`.
- Expired PR ownership handoff (status `expired`, or `pending` past `expiresAt`).

## Metadata-only Cleanup

`executeMetadataOnlyCleanup(decisionId, context)` is the single safe mutation entrypoint. It refuses to operate on decisions whose source recommendation is `destructive: "true_future"`. Successful execution flips the decision to `executed_metadata_only` and updates the associated recovery action to `executed_metadata_only`, while emitting metadata-only audit records. It never calls the Git provider, the filesystem, or any remote API.

## Destructive Cleanup (Future)

Any recommendation flagged `destructive: "true_future"` requires `requiredApproval: platform_owner | release_manager | security_reviewer` and routes to `future_destructive_review`. v1 records the metadata but never executes; `branch.delete_future`, `worktree.remove_future`, `pr.close_future`, and `cleanup.destructive_execute_future` are denied by default.

## Policy / Auth Behavior

Default static policy:

- `cleanup.scan` â€” allow for viewer+ when `metadataOnly && !destructiveCleanupEnabled && !realBranchDeletion && !realWorktreeRemoval`.
- `cleanup.recommend` â€” allow for developer+ when `metadataOnly && !destructiveCleanupEnabled`.
- `cleanup.decide` â€” allow for reviewer+ under the same context.
- `cleanup.metadata_execute` â€” allow for reviewer+ when no destructive flag and no real branch/worktree/PR mutation.
- `cleanup.destructive_execute_future`, `branch.delete_future`, `worktree.remove_future`, `pr.close_future` â€” deny by default.

Policy deny always wins. Auth/RBAC actor metadata, service-account metadata, RequestContext, and Tenant Scope metadata flow through into both audit records and recovery action metadata.

## API and Dashboard Surfaces

`/git/cleanup/*` endpoints are read-only or metadata-only mutation, and respond `405 Method Not Allowed` for other methods. `/dashboard/git-cleanup` returns a `BranchCleanupReadModel` with summary, orphan groupings, recommendation groupings, decisions, recovery actions, policy status, and no-destructive status. Dashboard sanitization redacts secrets, env values, and credential cache paths via existing helpers. `/health` exposes `branchCleanup.{status, scans, orphanRecords, recommendations, decisions, recoveryActions, destructiveFutureRecommendations, metadataOnlyRecommendations, destructiveCleanupEnabled: false, realBranchDeleted: false, realWorktreeRemoved: false, realPullRequestClosed: false, remoteGitCallsExecuted: false, filesystemDeletionsExecuted: false, noSecretsExposed: true, envValuesExposed: false}`.

## No-destructive-default Guarantee

By default, Branch Cleanup / Orphan Lease Recovery v1 records `destructiveCleanupEnabled: false`, `realBranchDeleted: false`, `realWorktreeRemoved: false`, `realPullRequestClosed: false`, `remoteGitCallsExecuted: false`, `filesystemDeletionsExecuted: false`, `noSecretsExposed: true`. Destructive recommendations remain future-only; metadata-only execution refuses them.

## Known Limitations

- Records remain in-memory in v1. Durable cleanup record storage is future work.
- Worktree allocation and PR ownership handoff snapshots are pluggable â€” when those features land, their snapshot adapters can be wired without modifying the cleanup service.
- Production tenant enforcement and scope filtering remain future work.
- Cleanup automation is intentionally absent; reviewers approve metadata-only cleanup and destructive cleanup remains future-only.
