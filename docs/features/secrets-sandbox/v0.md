# Secrets and Sandbox Design v0

## What v0 Implements

Secrets and Sandbox Design v0 adds a mock-first security boundary for secret metadata, secret lease requests, sandbox profiles, sandbox sessions, network egress policy, redaction policy, security audit events, API visibility, dashboard visibility, and deterministic tests.

It does not implement real secret retrieval, production secret injection, cloud secret managers, container/VM sandboxes, network enforcement, vendor credential cache access, or real provider calls.

## Secret Models

`packages/security` defines:

- `SecretRef`: metadata-only secret reference. It stores provider, name, scope, status, and metadata, but never raw secret material.
- SecretRef-backed Provider Credentials v1 extends `SecretRef` with `secretKind` and optional `envKey` references for GitHub, LLM, and provider API credentials. `envKey` is a reference only; it is never a secret value.
- `SecretScope`: policy scope for allowed resource kinds, actions, provider ids, runner kinds, TTL, and approval requirement.
- `SecretLease`: request/approval lifecycle metadata for future secret access. It never stores the secret value.
- `SecretAccessDecision`: deterministic allow/deny/require-approval decision linked to policy decisions.

The default seed includes a mock metadata reference and disabled future cloud-secret placeholders only.

## SecretManager Interface

`SecurityControlService` implements the v0 `SecretManager` interface:

- `getManagerKind()`
- `validateSecretRef(secretRef)`
- `createSecretRef(secretRef)`
- `updateSecretRefStatus(secretRefId, status)`
- `resolveCredential(request)`
- `resolveCredentialForInternalUse(request)`
- `requestLease(request)`
- `revokeLease(leaseId)`
- `getSafeEnvironment(leaseId)`
- `getSecretMetadata(secretRefId)`
- `recordSecretAudit(event)`

Default behavior denies broad secret reads and returns no raw environment secret through public DTOs. SecretRef-backed Provider Credentials v1 adds an explicit env-only provider for controlled GitHub/LLM integration tests; future `VaultSecretManagerFuture`, `AwsSecretsManagerFuture`, `GcpSecretManagerFuture`, and `AzureKeyVaultFuture` placeholders throw clear not-implemented errors and do not connect.

Secret Backend Migration Planning v0 documents the path from env fallback to Vault/cloud/custom backends and exposes read-only readiness metadata. It does not activate those future placeholders, read backend credentials, rotate secrets, or inject real secrets into runners, MCP tools, Local Agent, Git, LLM, or provider processes.

## Sandbox Models

`SandboxProfile` describes allowed capabilities such as network, file write, shell execution, remote Git, secrets, commands, paths, runtime, output limits, and cleanup policy.

`SandboxSession` records a requested/active/completed/cleaned-up sandbox session. v0 creates metadata records only and does not run a real sandbox runtime.

Default profiles:

- `sandbox_default_deny`: denies network, secrets, remote Git, arbitrary shell execution, and file write.
- `sandbox_local_temp_fixture`: permits only safe fixture/temp workspace command execution metadata with network/secrets/remote Git denied.
- future container, Firecracker, and Kubernetes profiles are disabled placeholders.

## Network Egress Policy

`NetworkEgressPolicy` models default allow/deny, allowed/denied hosts, ports, localhost/private-network flags, and metadata.

The default policy is `network_default_deny`, which denies all external network egress. v0 records decisions but does not implement OS-level network enforcement.

## Redaction Policy

`RedactionPolicy` controls masking of bearer tokens, API-key-like strings, environment dumps, provider tokens, and credential cache paths. It also limits stored preview bytes.

`redactWithPolicy` masks sensitive strings before API/dashboard/audit storage where v0 controls the path.

## Policy-as-code Integration

Policy-as-code v0 now includes actions and resources for:

- `secret.metadata.read`
- `secret.lease.request`
- `secret.lease.issue`
- `secret.lease.revoke`
- `sandbox.profile.use`
- `sandbox.session.create`
- `network.egress`
- `runner.secret.inject`
- `provider.credential.resolve`
- `local_agent.secret.forward`
- `audit.secret.view`

Default rules deny secret reads, runner secret injection, network egress, Local Agent secret forwarding, and secret audit viewing. Provider credential resolution is allowed only for active env-backed SecretRefs when the env secret provider is explicitly enabled and policy, Git/LLM gates, and allowlists pass. Safe local sandbox profile/session use is allowed only when network, secrets, and remote Git are disabled.

Production Auth/RBAC Planning v0 adds actor context for secret and sandbox attribution. SecretRef-backed Provider Credentials v1 now checks `AuthorizationService` before env-backed credential resolution when available. `AuthorizationService` does not grant raw secret access; `security_admin` can inspect audit/security metadata, while direct `secret.read`, credential-cache access, runner secret injection, and Local Agent secret forwarding remain denied by Policy-as-code.

Production Auth/RBAC v1 Planning adds read-only planning for tenant-scoped SecretRefs, security admin roles, service-account actors, request context propagation, and mock actor deprecation. It does not issue sessions, call IdPs, issue service-account credentials, grant raw secret access, or make secret issuance production-ready.

## Runner Integration

`AgentRunnerService` can receive `SecurityControlService`. When controlled local command execution is enabled for an explicit local workspace, it records a safe local sandbox session before the run and stores sandbox decision/session ids in run metadata. This does not enable secrets, network, remote Git, or production sandboxing.

## Enterprise Provider Integration

Enterprise Provider Abstraction remains blocked by default. Provider catalog entries may reference `secretRef` metadata, but v0 does not resolve raw provider credentials. Local CLI providers now route through Local Agent Protocol v1 and must not read vendor credential caches.

## Real Git Adapter v2 Integration

Real Git Adapter v2 can resolve GitHub API credentials and GitHub webhook secrets through SecretRef-backed Provider Credentials v1. Preferred config uses `AICHESTRA_GITHUB_TOKEN_SECRET_REF` and `AICHESTRA_GITHUB_WEBHOOK_SECRET_REF` plus the explicit env provider and env-key allowlist. Legacy `AICHESTRA_GITHUB_TOKEN` and `AICHESTRA_GITHUB_WEBHOOK_SECRET` remain supported when no SecretRef is configured. Token and webhook secret values are not stored in `SecretRef`, repos, pull request records, webhook read models, audit events, health responses, dashboard data, runner metadata, or Local Agent metadata.

## LLM Gateway v1 Integration

LLM Gateway v1 can resolve OpenAI-compatible API keys through SecretRef-backed Provider Credentials v1. Preferred config uses `AICHESTRA_LLM_API_KEY_SECRET_REF` plus the explicit env provider and env-key allowlist. Legacy `AICHESTRA_LLM_API_KEY` remains supported when no SecretRef is configured. The key is not stored in `SecretRef`, model catalog records, virtual model keys, usage ledger metadata, LLM audit events, health responses, dashboard read models, runner metadata, or Local Agent metadata.

## MCP Gateway v0 Integration

MCP Gateway v0 references `SecretRef`, network, sandbox, and redaction concepts but does not issue SecretLeases or forward secrets to tools. Any MCP tool declaring `requiredSecretRefs` is blocked by default. Network, write, deploy, local execution, real MCP transport, and Local Agent MCP forwarding remain disabled. MCP input/output/audit previews are redacted and size-limited before API/dashboard exposure.

## Local Agent Protocol Integration

Local Agent Protocol v1 uses Secrets and Sandbox v0 as metadata-only guardrails:

- invocation envelopes reference `SandboxProfile`, `NetworkEgressPolicy`, `RedactionPolicy`, and `SecretScope` ids instead of raw secrets;
- `sandbox_default_deny` is the default safe profile for mock dispatch;
- fixture daemon channels and compatibility checks remain metadata-only;
- network egress remains denied by default;
- no `SecretLease` is issued by default;
- `local_agent.secret.forward` remains denied by policy;
- stdout/stderr stream previews and normalized events are redacted before protocol audit/API/dashboard storage;
- sandbox and secret access decisions are recorded as metadata only.

## Test Strategy

`tests/secrets-sandbox-design-v0.test.ts` covers:

- secret refs contain no raw values
- invalid raw-secret and credential-cache references are rejected
- mock lease requests are denied without approval
- lease revoke works
- safe environment returns no raw secret by default
- default sandbox denies network/secrets/remote Git
- safe local fixture profile remains constrained
- future sandbox profiles are disabled placeholders
- network egress is denied
- redaction masks bearer tokens, API keys, env dumps, and credential cache paths
- policy decisions deny secret/network/runner secret operations
- runner integration records sandbox metadata
- provider boundary remains blocked
- security API, health, dashboard, and repository behavior
- SecretRef-backed provider credential resolution, env provider gates, Git/LLM integration, credential audit, and no-secret API/dashboard exposure are covered in `tests/secretref-provider-credentials-v1.test.ts`.

## Known Limitations

- Secret backends are in-memory and metadata-only.
- No real Vault/cloud secret manager integration exists.
- No production secret injection exists.
- No real sandbox runtime exists.
- Network egress is modeled and policy-checked, not enforced by OS/container controls.
- Real Local Agent daemon/network transport and vendor CLI execution remain future work.
- Security repositories are in-memory; Postgres tables are schema skeleton only.
- Auth/RBAC v0 provides mock actor context only; it is not production auth for secret issuance.
- Auth/RBAC v1 planning is readiness metadata only; real IdP, tenant isolation, session handling, and service-account credential issuance remain future work.
