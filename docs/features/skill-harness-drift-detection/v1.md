# Skill / Harness Drift Detection v1

Status: v1_implemented

Skill / Harness Drift Detection v1 is implemented as an advisory, mock-first, deterministic registry-level metadata layer that surfaces drift signals against Skill, Harness, Instruction, RegistryPackage, and Compatibility Profile targets. Drift Detection v1 never executes real eval suites, real canaries, or real LLM calls. It never mutates active registry entries, never auto-applies improvement candidates, never weakens Registry resolver gates, Governance, Policy, Auth/RBAC, Tenant Scope, SecretRef, Dashboard sanitization, Observability redaction, Staging/CI gates, or Safety guarantees. It never calls external providers, runs remote Git operations, executes vendor CLI, reads credential caches, or exposes secrets/env values. Default runtime stays mock-first and the apply gate is denied by default.

## What v1 Implements

- Models (`packages/registry/src/drift-detection.ts`):
  - `RegistryDriftTargetKind ∈ {skill | harness | instruction | registry_package | compatibility_profile}`
  - `RegistryDriftSignalKind ∈ {failure_rate_increase | token_cost_increase | runtime_increase | eval_status_decline | compatibility_warning_increase | manual_override_increase | rollback_involvement | conflict_involvement | stale_instruction | policy_denial_increase | provider_model_mismatch | unknown}`
  - `RegistryDriftSeverity ∈ {info | low | medium | high | critical}`
  - `RegistryDriftSignal`, `RegistryDriftBaseline`, `RegistryDriftAssessment`, `RegistryDriftRecommendation`, `RegistryDriftSummary`
  - `RegistryDriftAssessmentStatus ∈ {no_drift | watch | degraded | critical | insufficient_data}`
  - `RegistryDriftRecommendationKind ∈ {monitor | create_improvement_candidate | require_eval | require_canary | review_instruction | review_harness | review_provider_compatibility | deprecate_future | rollback_review}`
  - Deterministic seed data: `defaultRegistryDriftSignals` (8 signals across the seeded skills/harnesses/instructions) and `defaultRegistryDriftBaselines` (8 baselines).
- Service: `RegistryDriftDetectionService` with `listSignals`, `listBaselines`, `listAssessments`, `listRecommendations`, `collectSignals`, `buildBaseline`, `assessTarget`, `assessAll`, `linkToImprovementCandidate`, `getSummary`, `computeScoreForSignals`.
- New policy actions and default rules (`packages/policy/src/types.ts`, `packages/policy/src/default-rules.ts`):
  - `registry.drift.read`, `registry.drift.assess`, `registry.drift.recommend` — allow for the respective roles under the mock metadata-only environment (`metadataOnly && !registryMutationExecuted && !autoApplyEnabled && !evalExecuted && !canaryExecuted && !externalCallExecuted`).
  - `registry.drift.create_candidate_future`, `registry.drift.auto_apply_future` — deny by default.
  - New `registry_drift` `PolicyResourceKind`.
- Read-only API endpoints under `/registry/drift/*`:
  - `GET /registry/drift/signals`
  - `GET /registry/drift/baselines`
  - `GET /registry/drift/assessments`
  - `GET /registry/drift/recommendations`
  - `GET /registry/drift/summary`
  - `POST /registry/drift/assess`
  - `POST /registry/drift/assess-all`
- Read-only readiness endpoint: `GET /readiness/registry/drift/summary`.
- `/health` metadata key `registryDrift` exposing planning status, signal/baseline/assessment/recommendation counts, severity flags, and safety flags (`applyAllowed`, `registryMutationExecuted`, `evalExecuted`, `canaryExecuted`, `autoImprovementApplied`, `externalCallExecuted`, `noSecretsExposed`, `envValuesExposed`).
- Dashboard read model and panel: `GET /dashboard/registry-drift` and the "Skill / Harness Drift Detection" panel in the rendered HTML.
- Dashboard/Readiness Tenant Scope plan entries for `registry_drift` (`packages/deployment-readiness/src/dashboard-tenant-scope.ts`).
- Deterministic tests (`tests/skill-harness-drift-detection-v1.test.ts`) covering deterministic signal seed, baseline assembly, assessments across skill/harness/instruction targets, severity scoring with weighted signal multiplier, recommendation production, governance follow-up surface, policy deny short-circuit, policy default deny for create_candidate/auto_apply, dashboard panel rendering, API safety, no-secret/no-mutation guarantees.

## What v1 Does Not Implement

- No real LLM, MCP, vendor CLI, GitHub, or any external-provider calls.
- No real eval execution. `require_eval` recommendations are advisory only.
- No real canary execution. `require_canary` recommendations are advisory only.
- No registry mutation. Drift assessments never modify active Skill, Harness, Instruction, or RegistryPackage entries.
- No auto-improvement application. `create_improvement_candidate` recommendations remain draft-only; `linkToImprovementCandidate` produces a draft reference and policy denies `registry.drift.create_candidate_future` by default.
- No auto-apply. Policy denies `registry.drift.auto_apply_future` by default.
- No production tenant enforcement. Scope mismatch is metadata-only.
- No durable storage. Signals/baselines are deterministic seed data.
- No real telemetry ingestion. Signals come from seeded mock data; future external sources are gated behind `RegistryDriftSignalSource.future_external`.
- No drift signing or attestation. Future trust hooks remain separate.

## Scoring Model

`RegistryDriftDetectionService.computeScoreForSignals` is deterministic:

```
score = clamp(0, 100, Σ(weight[signal.kind] × multiplier[signal.severity]))
```

`weight` defaults: `failure_rate_increase: 35`, `eval_status_decline: 35`, `rollback_involvement: 20`, `policy_denial_increase: 25`, `compatibility_warning_increase: 15`, `token_cost_increase: 12`, `runtime_increase: 10`, `stale_instruction: 8`, `manual_override_increase: 12`, `conflict_involvement: 12`, `provider_model_mismatch: 10`, `unknown: 5`. `multiplier`: `info × 0.5`, `low × 0.8`, `medium × 1.0`, `high × 1.3`, `critical × 1.6`.

Severity is derived from the integer-clamped score: `[0..20] info`, `[21..40] low`, `[41..60] medium`, `[61..80] high`, `[81..100] critical`. Assessment status follows the score: `[0..20] no_drift` (or `insufficient_data` if no signals matched), `[21..40] watch`, `[41..80] degraded`, `[81..100] critical`, or `insufficient_data` when no signals are matched.

## Recommendation Model

Recommendations are advisory metadata only. `applyAllowed` is the literal `false`. The recommendation kind is derived from the matched signal kind and target kind. Recommendations always carry `metadata.gatePreserved: true`, `metadata.applyAllowed: false`, `metadata.evalExecuted: false`, `metadata.canaryExecuted: false`, `metadata.externalCallExecuted: false`.

`linkToImprovementCandidate` returns a draft-only candidate reference with `applyAllowed: false` and `draftOnly: true`. When `registry.drift.create_candidate_future` policy denies the request (the default), the result records `metadata.policyDenied: true` and the candidate reference becomes `draft_only_blocked`. No registry entries are mutated and no auto-apply is triggered.

## Resolver / Auto-Improvement Integration

The drift service reads through the registry data source (skills/harnesses/instructions) but never mutates resolver state, registry entries, or compatibility profiles. Auto-improvement integration is forward-only: `linkToImprovementCandidate` produces a draft-only proposal reference that future Auto-Improvement v2 can consume, but v1 does not invoke it. The resolver's `selectableRegistryEntry` semantics and lifecycle/approval/eval/checksum gates remain authoritative.

## Eval Suite Execution Harness Relationship

Eval Suite Execution Harness v1 can consume drift-follow-up metadata through the `drift-followup-suite`. Drift Detection remains advisory and does not execute evals itself. The eval harness records deterministic mock eval runs only; it does not run real telemetry, canaries, providers, MCP tools, vendor CLIs, or auto-apply actions.

## Policy / Auth Behavior

Default static policy:

- `registry.drift.read` — allow for viewer+ in metadata-only context.
- `registry.drift.assess` — allow for developer+ in metadata-only context where `resolverGatesPreserved: true`.
- `registry.drift.recommend` — allow for reviewer+ in metadata-only context.
- `registry.drift.create_candidate_future` — deny by default.
- `registry.drift.auto_apply_future` — deny by default.

When the policy evaluator returns `deny`, `assessTarget` short-circuits to an `insufficient_data` assessment with `metadata.policyDenied: true` and no recommendations are recorded.

## API and Dashboard Surfaces

- `/registry/drift/*` endpoints are read-only (`GET`) or evaluation-only (`POST /assess`, `POST /assess-all`); other methods return `method_not_allowed`. No endpoint mutates registry entries or executes eval/canary.
- `/readiness/registry/drift/summary` is read-only.
- `/dashboard/registry-drift` returns a `RegistryDriftReadModel` with summary, signal/baseline/assessment/recommendation arrays, critical/degraded/watch/insufficient_data buckets, governance follow-up list, and `noAutoApplyStatus`. Dashboard sanitization redacts secret-like patterns and credential cache paths via existing helpers.
- `/health.registryDrift` exposes counts and explicit `applyAllowed: false`, `registryMutationExecuted: false`, `evalExecuted: false`, `canaryExecuted: false`, `autoImprovementApplied: false`, `externalCallExecuted: false`, `noSecretsExposed: true`, `envValuesExposed: false`.

## No-Auto-Apply Guarantee

Drift assessments and recommendations are advisory metadata only. The service never:
- mutates Skill, Harness, Instruction, RegistryPackage, or Compatibility Profile entries,
- executes evals or canaries,
- applies improvement candidates,
- weakens Registry resolver / Approval / Governance / Lifecycle gates,
- bypasses Policy, Auth/RBAC, Tenant Scope, SecretRef, Observability redaction, Staging/CI gates.

`requiredActions` and `governanceFollowUps` contain human-actionable suggestions like `review_failure_signals`, `rerun_eval`, `review_compatibility_matrix`, `refresh_instruction_metadata`, but the service neither performs nor schedules those actions.

## Known Limitations

- Signals/baselines are deterministic seed data; durable storage and per-tenant customization are future work.
- Real telemetry, ledger, observability, and governance-audit-derived signals remain future-only (`RegistryDriftSignalSource.future_external`).
- Drift recommendations do not feed an active auto-improvement loop; `linkToImprovementCandidate` is draft-only and gated by `registry.drift.create_candidate_future` deny-by-default.
- Production tenant enforcement, drift signing, and drift attestation remain separate planned tasks.
- Eval/canary recommendations are advisory; no eval/canary execution occurs in v1.
