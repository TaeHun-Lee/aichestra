Plain status: Audit Query Scope Enforcement: v1_implemented_partial.

Audit Query Scope Enforcement v1 adds a mock-safe, read-only audit query decision and redaction layer for Aichestra's existing observability read models. It evaluates requested audit detail level, roles, scope metadata, and RequestContext attribution, then returns deterministic allow/redact/deny metadata. It does not implement production audit storage security.

## What v1 Implements

- `AuditQueryScopeRequest`
- `AuditQueryScopeDecision`
- `AuditQueryRedactionPlan`
- `AuditQueryScopeDecisionSummary`
- `AuditQueryScopeEnforcementSummary`
- `AuditQueryScopeEnforcementService`
- Read-only/check-only API surfaces:
  - `GET /readiness/audit-scope/summary`
  - `GET /readiness/audit-scope/redaction-plans`
  - `POST /observability/audit/query-scope/check`
- Scope-decision metadata on `GET /observability/audit/events`.
- Dashboard Observability metadata for audit query scope status, allowed detail level, redaction plan count, missing-scope warnings, raw-payload-forbidden status, and production-storage-enforcement false.
- External Observability Export v1 consumes the same no-raw-payload and no-secret posture for export envelope safety checks; export endpoints remain check-only and disabled by default.
- Deterministic tests covering role behavior, redaction, API/dashboard safety, and no-secret/no-env guarantees.

## What v1 Does Not Implement

v1 does not implement production Auth/RBAC, real tenant provisioning, database row-level security, production audit storage backend, real external SIEM/export, external observability backend delivery, external identity-provider calls, sessions, JWTs, API keys, service-account credentials, durable audit-query grants, production query filtering, raw payload access, or production audit query security.

## Audit Query Scope Request Model

`AuditQueryScopeRequest` records:

- actor/principal attribution;
- roles;
- tenant/team/project/repo/provider scope ids;
- resource kinds;
- audit sources;
- requested detail level: `summary`, `metadata`, `detail`, or `raw_payload_forbidden`;
- request/correlation ids;
- sanitized metadata.

The service can derive these fields from safe RequestContext/AuthContext-like metadata. It does not parse Authorization headers, cookies, sessions, JWTs, or production tokens.

## Decision Model

`AuditQueryScopeDecision` returns:

- `allow_summary`
- `allow_metadata`
- `redact_detail`
- `deny_detail`
- `deny_query`
- `warning_missing_scope`

Each decision includes:

- reason;
- matched, missing, and mismatched scopes;
- allowed detail level;
- redacted fields;
- request/correlation metadata;
- `rawPayloadAllowed: false`;
- `productionStorageEnforcement: false`.

## Redaction Plan

`AuditQueryRedactionPlan` defines deterministic redaction fields for:

- summary output;
- metadata output;
- detail output;
- raw-payload-forbidden requests.

Raw payloads, Authorization/cookie/session/token fields, credential/cache fields, private keys, env values, prompts, tool input/output, provider request/response, and webhook payload names are redacted or omitted depending on the allowed detail level.

## Role Behavior

- `viewer`: summary only; metadata/detail/raw payload are redacted or denied.
- `developer`: metadata requires project, repo, or provider scope; detail is reduced to metadata/summary.
- `security_admin`: operational metadata/detail visibility with secret-like/raw fields still redacted.
- `audit_reader`: metadata/detail require tenant/team/project plus audit-query source/resource metadata.
- `platform_admin`: operational metadata/detail visibility with redaction; raw payload remains forbidden.
- service accounts: no bypass; missing audit-query scope or human review role denies detail.

These decisions are representative v1 checks, not production authorization.

## Dashboard And API Behavior

`GET /observability/audit/events` now includes a `scopeDecisionSummary`. When the decision allows only summary, event detail metadata is replaced with summary-safe redaction markers. When raw payload is requested, v1 returns a denied/empty query result.

`/dashboard/observability` shows:

- audit query scope implementation status;
- allowed detail level;
- redacted detail count;
- missing-scope warnings;
- redaction plans;
- raw payload forbidden true;
- production storage enforcement false;
- no-secret/no-env status.

## Production Gaps

Production audit query enforcement still requires:

- trusted production Auth/RBAC claims;
- durable tenant/team/project/repo/provider/audit-query grants;
- durable audit storage with scoped query predicates;
- row- or query-layer filtering;
- cache partitioning by role/scope/detail level;
- external export governance;
- External Observability Export Integration-Test Profile v1 before any backend-specific exporter validation;
- retention/legal-hold integration;
- security review.

## No Raw Payload Guarantee

`raw_payload_forbidden` requests are always denied in v1. Common audit envelopes still never store raw provider payloads, raw prompts, raw webhook payloads, tokens, cookies, sessions, credential cache contents, provider credentials, or env values.

## Known Limitations

- Scope checks are deterministic metadata comparisons.
- Existing audit repositories are still in-memory by default.
- Dashboard metadata is descriptive and does not prove production authorization.
- No durable query audit store or tenant-indexed search exists.
- Production tenant isolation remains false.
