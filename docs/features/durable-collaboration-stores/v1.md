# Durable Collaboration Stores v1

Status: `v1_implemented`

Default runtime: in-memory

Optional Postgres: schema/repository support where implemented, selected only by the existing storage provider gate

Production DB readiness: not production-ready

## What v1 Implements

Durable Collaboration Stores v1 adds a safe persistence boundary for collaboration-control records that are required before production concurrency control can survive process restarts.

Implemented surfaces:

- durable collaboration store inventory
- provider-neutral repository interfaces in `packages/db`
- in-memory repository implementations for all v1 repository groups
- optional Postgres repository implementations behind `AICHESTRA_STORAGE_PROVIDER=postgres`
- safe Postgres schema skeletons in `infra/migrations/0001_initial_aichestra_schema.sql`
- repository factory wiring
- read-only readiness endpoints:
  - `GET /readiness/collaboration-stores/inventory`
  - `GET /readiness/collaboration-stores/repositories`
  - `GET /readiness/collaboration-stores/schema`
  - `GET /readiness/collaboration-stores/summary`
- dashboard panel and API surface:
  - `GET /dashboard/collaboration-stores`
  - Durable Collaboration Stores panel in the web dashboard
- `/health.durableCollaborationStores` metadata
- deterministic tests for in-memory contracts, schema coverage, API/dashboard safety, and optional Postgres skip behavior

## What v1 Does Not Implement

v1 does not implement production-ready database operations. It does not run migrations automatically, enforce row-level security, enable production tenant isolation, execute real Git operations, mutate user workspaces, delete branches or worktrees, call external providers, execute real LLM calls, execute vendor CLIs, read credential caches, or store credentials.

Feature services remain mock-first and in-memory by default. The new `packages/db` boundary gives services a durable persistence target, but service-by-service migration to use durable repositories as the source of truth remains a follow-up.

## Repository Architecture

`packages/db` now exposes durable collaboration repository groups:

- `branchOrchestration`
- `agentSessionCoordination`
- `agentWorkspace`
- `agentWorktreeAllocation`
- `editIntent`
- `mergeQueuePolicy`
- `conflictResolution`
- `prOwnership`
- `cleanupRecovery`

Each group supports:

- `create`
- `getById`
- `list`
- `updateStatus`
- `updateMetadata`
- `appendEvent`
- `listEvents`
- `summarize`

All records are metadata-only. Metadata is sanitized on create/update/event append and before return.

## In-Memory Behavior

The default provider is still `in_memory`. In-memory durable repositories are deterministic and safe for the default MVP runtime and tests. They do not call Git, mutate workspaces, call providers, execute commands, or read environment values.

`durableCollaborationStoreConfigured` is `false` for the default in-memory runtime.

## Optional Postgres Behavior

Optional Postgres durable repositories are created by the existing Postgres repository factory. They are selected only when the Postgres storage provider is explicitly configured through existing Persistent DB v1 gates.

Optional Postgres tests are skipped unless `AICHESTRA_TEST_DATABASE_URL` is configured. API, dashboard, health, and readiness responses never expose the DB URL.

## Safety Guarantees

v1 preserves these guarantees:

- default runtime remains in-memory
- Postgres is opt-in only
- no remote Git operations
- no branch deletion
- no worktree deletion
- no user workspace mutation
- no real LLM calls
- no vendor CLI execution
- no external provider calls
- no credential-cache reads
- no secret or env value exposure
- no DB URL exposure
- no raw payload storage

## Known Limitations

- Existing feature services are not fully migrated to use durable repositories as their authoritative store.
- Postgres implementation is a safe repository boundary, not production operations readiness.
- No automatic migration runner is enabled by default.
- Production tenant isolation, row-level security, retention jobs, backup/restore, and pooling remain out of scope.
