# Local Agent Protocol v0

## What v0 Implements

Local Agent Protocol v0 adds a mock-first coordination layer for future user-machine Local Agents.

It implements provider-neutral Local Agent registration, sessions, capabilities, invocation envelopes, consent requests and decisions, invocation lifecycle states, normalized stdout/stderr/system events, in-memory repositories, mock transport, API endpoints, health metadata, dashboard visibility, provider audit linkage, policy checks, and security/redaction integration.

Local Agent Protocol v1 now builds on this baseline with mock signed channels, a controlled fixture daemon simulation, capability advertisements, CLI compatibility checks, consent read models, stream events, and deterministic timeout/cancel/disconnect behavior. See `docs/features/local-agent-protocol/v1.md`.

It does not implement a real daemon, WebSocket, gRPC, HTTP tunnel, vendor CLI execution, PTY automation, real provider calls, credential-cache reads, credential exchange, real secret injection, production sandboxing, or network enforcement.

## Cloud vs Local Agent Boundary

Aichestra Cloud records protocol metadata and decisions. It does not run local CLI providers directly.

The future Aichestra Local Agent is the user-machine component that may eventually broker local CLI tools after consent, policy, sandbox, network, secret, redaction, and audit gates are satisfied. v0 only models that boundary and exercises it with `MockLocalAgentTransport`.

## Why Cloud Cannot Run `local_cli` Directly

`local_cli` providers rely on user-machine context and provider-owned local sessions. Aichestra Cloud must not:

- execute Claude Code, Codex CLI, Gemini CLI, or vendor CLIs;
- read or upload `~/.codex/auth.json`, `~/.claude`, Google credential caches, OS keychains, or session files;
- convert local CLI OAuth/session state into a backend credential;
- use PTY automation or danger/full-access modes.

Direct Cloud local CLI execution is represented by policy/audit as blocked.

## Domain Models

`packages/llm-gateway/src/local-agent-protocol.ts` defines:

- `LocalAgentRegistration`
- `LocalAgentCapability`
- `LocalAgentSession`
- `LocalAgentInvocationEnvelope`
- `LocalAgentInvocation`
- `LocalAgentConsentRequest`
- `LocalAgentConsentDecision`
- `LocalAgentNormalizedEvent`
- `LocalAgentProtocolAuditEvent`

No model stores raw secrets, provider API keys, bearer tokens, OAuth tokens, or credential-cache content.

## Invocation Envelope

`LocalAgentInvocationEnvelope` stores provider id, Local Agent id, workspace ref, optional task/run refs, optional instruction hash and prompt ref, consent level, sandbox/network/redaction policy refs, secret scope ids, timeout, mock signature status, and sanitized metadata.

Signature status is metadata-only in v0:

- `not_required_mock`
- `valid_mock`
- `invalid`
- `missing`

Invalid or missing signatures block dispatch.

## Consent

Supported consent levels:

- `read_only`
- `workspace_write`
- `shell_execution`
- `network_or_secret_access`
- `danger_full_access`

Default behavior:

- `read_only` can dispatch only after policy and consent are satisfied.
- `workspace_write`, `shell_execution`, `network_or_secret_access`, and `danger_full_access` fail closed by default.
- Denied or expired consent blocks invocation.

## Mock Transport

`MockLocalAgentTransport` is in-memory only. It does not use network sockets and does not execute processes.

Dispatch returns deterministic mock events:

- system progress
- stdout message
- stderr progress

Stderr progress is not treated as failure. Completion state is determined by explicit completion/exit code.

## Lifecycle States

`LocalAgentInvocationState`:

- `requested`
- `awaiting_consent`
- `consent_denied`
- `policy_blocked`
- `dispatched`
- `running`
- `completed`
- `failed`
- `cancelled`
- `timed_out`
- `local_agent_unavailable`

Illegal transitions are rejected. Completed/failed invocations cannot be dispatched or completed again. Running/dispatched invocations can be cancelled.

## Policy Integration

Policy-as-code v0 now includes Local Agent protocol actions:

- `local_agent.register`
- `local_agent.invoke`
- `local_agent.consent.request`
- `local_agent.consent.approve`
- `local_agent.cancel`
- `local_agent.revoke`
- `local_agent.event.receive`
- `local_cli.invoke`
- `local_cli.danger_full_access`

Defaults:

- allow mock Local Agent registration/revocation/cancel for system/mock admin actors;
- require consent for Local Agent invocation;
- allow read-only invocation only with connected mock agent and approved consent;
- deny revoked/disconnected agent invocation;
- deny direct local CLI execution;
- deny credential cache read/upload;
- deny local CLI shell/file/network/danger access.

## Secrets and Sandbox Integration

Protocol dispatch references `SandboxProfile`, `NetworkEgressPolicy`, `RedactionPolicy`, and secret scope ids.

v0 behavior:

- sandbox profile is required for local CLI providers;
- default-deny sandbox metadata may be recorded;
- network egress remains denied by default;
- no secret lease is issued by default;
- Local Agent secret forwarding is denied;
- redaction policy is required before event storage;
- stdout/stderr payloads are redacted and preview-limited before storage.

## Enterprise Provider Integration

`LocalCliProviderAdapter` now uses `LocalAgentProtocolService` when configured.

Outcomes:

- no connected Local Agent: `local_agent_required`
- connected agent with missing consent: `awaiting_consent`
- approved read-only mock invocation: deterministic `completed` result from mock transport
- direct Cloud local CLI execution: blocked by policy/audit

No vendor CLI is executed and no credential file is read.

## Event Normalization

`LocalAgentNormalizedEvent` stores:

- invocation id
- source: `stdout`, `stderr`, or `system`
- type: token/message/progress/warning/error/final/metadata
- sanitized payload
- redaction flag
- timestamp

Stdout and stderr remain separate. Structured parser helper `parseLocalAgentStructuredOutput` returns deterministic parse errors for malformed JSON/JSONL.

## Audit Events

Local Agent protocol audit events include registration, heartbeat, revocation, invocation request, policy block, awaiting consent, consent request/approval/denial, dispatch, unavailable agent, event receipt, completion, cancellation, output redaction, credential-cache denial, and direct local CLI execution blocks.

Audit metadata is sanitized and excludes secrets, tokens, API keys, credential-cache content, and raw prompts.

## Dashboard

The dashboard shows Local Agent Protocol status, registered agents, capabilities, pending consent, recent invocation state, normalized event previews, redaction status, audit events, blocked direct local CLI execution, and blocked credential-cache access.

It is not a desktop agent UI.

## Test Strategy

`tests/local-agent-protocol-v0.test.ts` covers registration, capabilities, heartbeat, revoke, disconnected/revoked dispatch denial, invocation envelopes, consent approval/denial/expiry, danger/shell/network/secret/default-deny behavior, mock dispatch, event redaction, cancellation, illegal transitions, policy integration, provider integration, event parser behavior, API endpoints, health, audit, and no-secret exposure.

Existing provider and dashboard tests cover integration regressions.

## Known Limitations

- Runtime repositories are in-memory only.
- Postgres tables are schema skeleton only.
- No real daemon or transport exists.
- No vendor CLI execution exists.
- No production auth/RBAC exists.
- No production sandbox or network enforcement exists.
- No real secret lease or Local Agent secret forwarding exists.
