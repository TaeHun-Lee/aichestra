# LLM Gateway v1

## What v1 Adds Over v0

LLM Gateway v1 enables one controlled real provider path: an OpenAI-compatible HTTP chat-completions provider. The same boundary can be used with LiteLLM-compatible deployments that expose the OpenAI API surface.

The default runtime remains mock-first. `MockLLMProvider` is still the default provider, and default tests do not call real LLM providers.

## Selected Provider Path

v1 implements only:

- provider kind: `openai_compatible`;
- operation: chat completion;
- response normalization into `LLMCompletionResult`;
- token/cost normalization from provider usage when present;
- deterministic error mapping;
- redaction before audit/dashboard/API storage.

It does not implement broad multi-provider support, BYOK, OAuth, device-code, WIF, IAM, Local CLI provider execution, streaming, tool calls, or vendor CLI invocation.

## Remote LLM Safety Gates

Remote completion is impossible unless all required gates pass:

- `AICHESTRA_LLM_PROVIDER=openai_compatible`
- `AICHESTRA_ENABLE_REMOTE_LLM=true`
- `AICHESTRA_ALLOW_REMOTE_LLM_COMPLETION=true`
- `AICHESTRA_LLM_BASE_URL`
- `AICHESTRA_LLM_API_KEY_SECRET_REF` with the env SecretRef provider enabled, or legacy `AICHESTRA_LLM_API_KEY`
- model catalog status is `active`
- virtual model key allows `openai_compatible` and the selected model id
- budget check allows the request
- policy allows provider credential resolution, provider invocation, remote completion, model use, and completion
- model allowlist passes when `AICHESTRA_LLM_ALLOWED_MODELS` is configured

`AICHESTRA_LLM_DEFAULT_MODEL` maps `openai-compatible/default` to the provider-side model name. `AICHESTRA_LLM_ALLOWED_MODELS` may include either Aichestra model ids or provider-side model names.

## OpenAI-Compatible Provider Boundary

HTTP behavior is isolated inside `OpenAICompatibleLLMProvider` and its `OpenAICompatibleHttpClient` boundary in `packages/llm-gateway/src/providers.ts`.

Service/API/dashboard layers do not call `fetch` or provider SDKs directly. Tests use mocked HTTP clients. The real fetch client is only used when the remote provider is selected and all gateway/policy/budget gates have passed.

## Config Variables

```bash
AICHESTRA_LLM_PROVIDER=mock | openai_compatible
AICHESTRA_ENABLE_REMOTE_LLM=false
AICHESTRA_ALLOW_REMOTE_LLM_COMPLETION=false
AICHESTRA_LLM_BASE_URL=
AICHESTRA_LLM_API_KEY_SECRET_REF=
AICHESTRA_LLM_API_KEY=
AICHESTRA_ENABLE_ENV_SECRET_PROVIDER=false
AICHESTRA_ALLOWED_SECRET_ENV_KEYS=
AICHESTRA_LLM_ALLOWED_MODELS=
AICHESTRA_LLM_DEFAULT_MODEL=
AICHESTRA_LLM_INTEGRATION_TESTS=false
```

Health, config, dashboard, and audit expose only booleans/counts such as `apiKeyConfigured`, `baseUrlConfigured`, and `allowedModelCount`. They never return the API key or base credential material.

SecretRef-backed Provider Credentials v1 is now the preferred credential path. `AICHESTRA_LLM_API_KEY` remains a legacy env fallback when no SecretRef is configured. The env SecretRef provider is explicit, allowlist-aware, and never enumerates environment variables.

## Policy Integration

Policy-as-code v0 now allows OpenAI-compatible remote completion only when v1 gates pass. It still denies remote completion by default.

Policy actions used by the gateway:

- `llm.model.use`
- `provider.credential.resolve`
- `provider.invoke`
- `llm.remote_completion`
- `llm.completion`

Policy denial occurs before any HTTP call and is audited.

## Budget and Usage Ledger Integration

The existing virtual model key and per-task budget checks remain authoritative. A remote call records usage only after successful provider completion.

Usage ledger entries include:

- `taskId`
- `taskRunId`
- provider kind
- provider id, when supplied
- model id
- input tokens
- output tokens
- estimated cost
- latency
- gateway request id

Blocked and failed calls create LLM audit events but do not create successful usage entries.

## Audit and Redaction

v1 adds remote-specific audit events:

- `llm_remote_completion_requested`
- `llm_remote_completion_blocked`
- `llm_remote_completion_completed`
- `llm_remote_completion_failed`
- `llm_budget_blocked`
- `llm_policy_blocked`
- `llm_provider_error`
- `llm_output_redacted`

Existing mock success/block events remain for compatibility.

Audit metadata is sanitized. It must not include API keys, bearer tokens, raw credentials, provider credential cache content, or unredacted provider output.

## API Endpoints

Existing LLM endpoints now support v1 metadata:

- `GET /llm/config`
- `GET /llm/providers`
- `GET /llm/models`
- `POST /llm/completions`
- `GET /llm/usage`
- `GET /llm/audit`

`POST /llm/completions` uses the mock provider by default. Remote calls return blocked results unless every gate, budget check, model allowlist, and policy decision passes.

## Health and Dashboard

`GET /health` reports:

- provider kind
- remote LLM enabled
- remote completion enabled
- base URL configured
- API key configured
- allowed model count
- default model configured
- integration-test gate

The dashboard LLM read model shows remote gate state, selected/default model metadata, allowed model summary, usage, audit, and blocked examples. It does not expose secrets.

## Integration Test Setup

Real remote LLM tests are skipped unless all are set:

```bash
AICHESTRA_LLM_INTEGRATION_TESTS=true
AICHESTRA_LLM_PROVIDER=openai_compatible
AICHESTRA_ENABLE_REMOTE_LLM=true
AICHESTRA_ALLOW_REMOTE_LLM_COMPLETION=true
AICHESTRA_LLM_API_KEY=...
AICHESTRA_LLM_BASE_URL=https://...
AICHESTRA_LLM_DEFAULT_MODEL=...
# or
AICHESTRA_LLM_ALLOWED_MODELS=...
```

Default tests use mocked HTTP clients and do not call real providers.

## Known Limitations

- Not production-ready LLM integration.
- No BYOK.
- SecretRef-backed credentials use the explicit env provider only; Vault/cloud secret manager integration is not implemented.
- No OAuth/device-code/WIF/IAM.
- No broad multi-provider routing.
- No streaming.
- No tool-call support.
- No production auth/RBAC scoping.
- Runtime LLM repositories remain in-memory by default.
