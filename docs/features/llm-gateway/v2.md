# LLM Gateway v2

## What v2 Adds Over v1

LLM Gateway v2 adds a safe multi-provider routing control plane on top of the v1 OpenAI-compatible path. The default runtime remains mock-first.

v2 implements provider-aware `LLMRoute` records, bounded `LLMFallbackPolicy` records, in-memory `LLMRoutingDecision` records for every routed request, provider health read models, model capability and prompt-class matching, Auth/RBAC, Policy-as-code, SecretRef, budget, usage ledger, audit, redaction, API visibility, and dashboard visibility.

v2 does not make broad real provider calls. The only remote-capable provider remains the v1 OpenAI-compatible HTTP boundary, and it is still disabled unless every explicit gate passes.

## Routing Model

`LLMRoutingRequest` captures task/run attribution, actor/principal metadata, request id, correlation id, source, requested model, requested capabilities, prompt class, priority, fallback limit, optional budget limit, and optional provider/model scope metadata where supplied by the scope model.

`LLMRoute` maps a provider id/kind to a model id with priority, enabled status, capabilities, prompt classes, token and estimated-cost hints, remote and SecretRef requirements, and fallback eligibility.

`LLMRoutingDecision` records the selected route or blocked/no-route outcome with policy, budget, credential decision ids, and sanitized request/correlation metadata where available. Decisions never contain API keys, bearer tokens, credential values, raw prompts, or raw provider output.

## Route Selection Flow

The router loads route candidates, filters disabled routes and models, applies provider/model allowlists and denylists, matches capabilities and prompt classes, enforces `AICHESTRA_LLM_ROUTING_MODE`, checks RequestContext/AuthContext, Auth/RBAC and Policy-as-code, checks SecretRef credentials for remote routes, checks budget and virtual model key policy, and executes the selected provider adapter.

Mock routes are selected by default. `local_cli` routes are represented as Local Agent required and are not directly executable.

## Fallback Safety

Fallback is disabled by default:

```bash
AICHESTRA_ENABLE_LLM_FALLBACK=false
AICHESTRA_LLM_MAX_FALLBACK_ATTEMPTS=0
```

Fallback requires routing config, fallback policy, budget, and policy allow decisions. It is bounded, deterministic in tests, and does not bypass Auth/RBAC, Policy-as-code, SecretRef, model allowlist, provider allowlist, or budget gates. Policy, credential, and budget denials stop fallback by default.

## Provider Catalog Integration

v2 keeps `MockLLMProvider` and the v1 `OpenAICompatibleLLMProvider`.

It adds disabled or non-executing provider skeletons for Anthropic-compatible, Gemini-compatible, Bedrock-compatible, Vertex-compatible, Azure-compatible, LiteLLM-compatible, and Local CLI bridge provider kinds. Skeleton providers return deterministic `provider_not_implemented`, `provider_unavailable`, or `local_agent_required` results and do not read credentials or make network calls. Local CLI Provider Templates v1 adds metadata-only Claude Code, OpenAI Codex CLI, Gemini CLI, Aider, and custom local CLI templates; these are not executable LLM providers.

## Config Variables

```bash
AICHESTRA_LLM_ROUTING_MODE=mock_only
AICHESTRA_ENABLE_LLM_FALLBACK=false
AICHESTRA_LLM_MAX_FALLBACK_ATTEMPTS=0
AICHESTRA_ALLOWED_LLM_PROVIDER_KINDS=
AICHESTRA_ALLOWED_LLM_PROVIDER_IDS=
AICHESTRA_DENIED_LLM_PROVIDER_IDS=
AICHESTRA_ALLOWED_LLM_MODELS=
AICHESTRA_DENIED_LLM_MODELS=
```

OpenAI-compatible remote completion still requires the v1 gates:

```bash
AICHESTRA_LLM_PROVIDER=openai_compatible
AICHESTRA_ENABLE_REMOTE_LLM=true
AICHESTRA_ALLOW_REMOTE_LLM_COMPLETION=true
AICHESTRA_LLM_BASE_URL=...
AICHESTRA_LLM_API_KEY_SECRET_REF=...
AICHESTRA_ENABLE_ENV_SECRET_PROVIDER=true
AICHESTRA_ALLOWED_SECRET_ENV_KEYS=AICHESTRA_LLM_API_KEY
```

- `GET /llm/routes`
- `POST /llm/routes`
- `PATCH /llm/routes/:id/status`
- `GET /llm/fallback-policies`
- `POST /llm/fallback-policies`
- `POST /llm/route`
- `POST /llm/completions`
- `GET /llm/routing/decisions`
- `GET /llm/providers/health`
- `GET /llm/usage`
- `GET /llm/audit`

`POST /llm/completions` now routes through the router. Default completions stay mock. Remote completions remain blocked unless all v1 and v2 gates pass.

## Dashboard and Health

`GET /health` reports routing mode, fallback status, max fallback attempts, provider counts, remote provider counts, and remote LLM gate state without exposing secrets.

The dashboard LLM read model includes routing mode, route table, provider health, fallback policy, recent routing decisions, fallback attempts, provider/model usage attribution, blocked route examples, and local CLI routes that require Local Agent.

LLM Gateway integration-test profile v1 adds read-only readiness and dashboard surfaces for optional OpenAI-compatible live tests:

- `GET /readiness/llm-integration/profile`
- `GET /readiness/llm-integration/test-cases`
- `GET /readiness/llm-integration/safety-checks`
- `GET /readiness/llm-integration/summary`
- `GET /dashboard/llm-integration`

These surfaces expose gate counts, model allowlist counts, budget status, fallback safety, SecretRef presence, and no-secret/no-env/no-raw-response booleans only. They do not call a provider, resolve API keys, return base URL values, enable streaming/tool calls, use vendor CLI, or read credential caches.

Secret Backend Migration Planning v0 adds read-only planning for migrating `llm_api_key` SecretRefs from env fallback to a selected real backend, plus lease TTL, rotation, env fallback deprecation, and no-secret/no-env dashboard visibility. It does not implement BYOK, OAuth, WIF/IAM, provider credential cache reads, or real secret backend calls.

Vault-backed Secret Backend v1 allows LLM API key and provider API key SecretRefs to use `provider: vault` behind explicit Vault gates and path allowlists. This does not change remote LLM safety: OpenAI-compatible completion still requires all LLM v1/v2 remote gates, model allowlists, budget checks, Auth/RBAC, Policy-as-code, and credential resolution checks. Default runtime/tests still do not call LLM providers or Vault.

Production Auth/RBAC v1 Planning adds read-only planning for future LLM gateway service-account attribution, tenant/model/provider scoping, request context propagation, and mock actor deprecation through `/readiness/auth/*`, `/dashboard/auth-production`, and safe `/health` metadata. RequestContext Propagation v1 now carries selected request/correlation metadata into LLM route selection, completion policy checks, routing decisions, usage metadata, and audit. Service Account Actor Boundary v1 adds `llm_gateway_service` and `llm_router_service` fallback attribution without issuing credentials. Tenant/Repo/Provider Scope Model v1 adds additive `ProviderScope` and `ModelScope` metadata to migrated LLM routing decisions and audit records. It does not implement real IdP calls, sessions, JWTs, tenant enforcement, tenant budgets, or service-account credential issuance.

## Persistence

Runtime route, fallback policy, routing decision, and provider health read models are in-memory in v2. Future durable tables are documented in `docs/foundations/persistent-storage-schema-v0.md`.

## Known Limitations

- Not production-ready LLM integration.
- No BYOK.
- No OAuth/device-code/WIF/IAM.
- No real Anthropic, Gemini, Bedrock, Vertex, Azure, or LiteLLM calls.
- No Local CLI execution.
- Local CLI templates remain `template_only`, Local Agent required, direct execution disabled, PTY unsupported, credential-cache reads denied, and secret forwarding denied.
- No streaming or tool-call routing.
- No persistent route administration store.
- OpenAI-compatible remains the only remote-capable provider and still requires explicit v1 gates.
- Vault-backed LLM credentials are supported only as a gated provider boundary; production Vault rollout and rotation remain future work.

## MCP Gateway Integration

MCP Gateway v0 now owns mock-first tool catalog and invocation governance. LLM Gateway v2 does not auto-execute model-generated tool calls; future tool-use context must go through MCP Gateway, Auth/RBAC, Policy-as-code, SecretRef/Secrets/Sandbox, redaction, and audit boundaries.
