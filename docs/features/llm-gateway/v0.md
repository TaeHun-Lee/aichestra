# LLM Gateway v0

## What v0 Implements

LLM Gateway v0 adds a mock-first central model control plane:

- Provider-neutral `LLMProvider` interface.
- `LLMGatewayService` for routing, budget checks, usage ledger recording, and audit events.
- Deterministic `MockLLMProvider` as the default provider.
- `OpenAICompatibleLLMProvider` skeleton that blocks remote calls by default and performs no network calls.
- In-memory model catalog with deterministic seed models.
- Virtual model key skeletons as internal policy objects, not provider credentials.
- Per-task and virtual-key budget checks.
- Usage ledger integration for successful mock completions.
- API, health, dashboard, and tests.

This is not real LLM provider integration.

## LLMProvider Interface

Defined in `packages/llm-gateway/src/types.ts`:

- `getProviderKind`
- `validateConnection`
- `listModels`
- `createCompletion`
- `estimateUsage`, optional
- `close`, optional

Provider-neutral models include `LLMModel`, `LLMCompletionRequest`, `LLMCompletionResult`, `LLMUsageAttribution`, `LLMAuditEvent`, `BudgetDecision`, and `VirtualModelKey`.

## LLMGateway Service

`LLMGatewayService` in `packages/llm-gateway/src/gateway.ts` owns:

- deterministic route selection
- budget checks
- virtual key checks
- completion routing
- usage ledger recording
- LLM audit event recording
- legacy `complete()` compatibility for the current mock agent runner

The worker now uses the central gateway service as its default LLM gateway while preserving existing usage ledger attribution.

## MockLLMProvider

`MockLLMProvider`:

- returns deterministic completions
- estimates token count and cost deterministically
- supports provider unavailable simulation
- supports model disabled simulation
- uses no secrets
- makes no external calls

## OpenAI-Compatible Skeleton

`OpenAICompatibleLLMProvider`:

- is disabled by default
- performs no network calls
- has no SDK dependency
- returns `blocked_remote_llm_disabled` when remote LLM is disabled
- returns `remote_llm_completion_disabled` when completion is not explicitly allowed
- returns `openai_compatible_not_implemented` even if future config is present

## Model Catalog

Seed models:

- `mock-model`
- `mock-small@1.0`
- `mock-coder@1.0`
- `mock-conflict-resolver@1.0`
- `mock-registry-reviewer@1.0`
- `openai-compatible/default`, originally disabled in v0 and now active in v1 only behind explicit remote gates

Disabled and deprecated models are not selected by default.

## Virtual Key Skeleton

`VirtualModelKey` is an internal policy object:

- owner scope
- allowed provider kinds
- allowed model ids
- budget/rate metadata
- active/disabled lifecycle

It is not a provider API key and does not store OpenAI, Anthropic, Gemini, Bedrock, or other provider secrets.

## Budget Policy

Gateway budget checks enforce:

- active virtual key
- allowed provider
- allowed model
- per-task budget limit
- virtual key per-task budget limit
- deterministic estimated cost

Blocked calls create audit events and do not create usage ledger entries.

## Usage Ledger Integration

Successful `routeCompletion` calls create usage ledger entries with:

- `taskId`
- `taskRunId`
- provider kind
- model id
- token counts
- estimated cost
- `source = llm_gateway`
- gateway request id

The legacy worker flow still records exactly one usage entry per task run.

## Safety Gates

Default:

- `AICHESTRA_LLM_PROVIDER=mock`
- `AICHESTRA_ENABLE_REMOTE_LLM=false`
- `AICHESTRA_ALLOW_REMOTE_LLM_COMPLETION=false`

Optional future config placeholders:

- `AICHESTRA_LLM_PROVIDER=openai_compatible`
- `AICHESTRA_LLM_BASE_URL`
- `AICHESTRA_LLM_API_KEY`

The API key env var is only a config placeholder in v0. It is not stored, logged, returned, or used to call a remote provider.

## Known Limitations

- No real provider calls.
- No streaming.
- No BYOK.
- No production secret manager.
- No persistent LLM model catalog repository yet.
- No persistent LLM audit/request repository yet.
- No real billing or rate limiter.
- OpenAI-compatible provider is a skeleton only.

## Enterprise Provider Abstraction Follow-up

Enterprise LLM Provider Abstraction v0 extends this package with provider catalog/auth/credential/token/adapter skeletons for Claude, Codex, Gemini, Vertex, Bedrock, Foundry, and local CLI provider boundaries.

LLM Gateway still defaults to `MockLLMProvider`. Provider catalog entries may be referenced for provider id and billing-mode attribution, but no catalog entry enables a real call in v0. Local CLI entries route through Local Agent Protocol v1, require mock channel/compatibility/consent gates, and never read local credential caches.
