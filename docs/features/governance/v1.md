# Phase 4 Governance v1

## What v1 adds over Auto-improvement v0

Phase 4 Auto-improvement v0 can produce deterministic draft proposals and draft registry changes. Governance v1 adds the review and gate-evaluation layer around those proposals while keeping the system mock-only and draft-only.

Implemented additions:

- Proposal review queue.
- Governance decision records.
- Local/mock proposal eval run attachment.
- Canary readiness evaluation without rollout execution.
- Apply gate evaluation without apply behavior.
- Improvement governance audit events.
- Dashboard visibility for governance state.
- Registry/Governance RequestContext Migration v1 attribution for migrated decision, eval, canary, apply-gate, and audit paths.

## Proposal review queue

`ProposalReviewQueueItem` is a read model for reviewable proposals. Draft, `awaiting_review`, and `eval_required` proposals appear by default. Rejected, archived, and applied proposals are excluded unless explicitly requested.

The queue includes proposal identity, target information, blocking reasons, required evals, approval and canary requirements, the latest decision, and a deterministic recommended action.

## Governance decision model

`ProposalGovernanceDecision` records reviewer intent:

- `approve`
- `reject`
- `request_changes`
- `mark_eval_required`
- `mark_canary_required`
- `archive`

Decisions update proposal status only. They do not apply draft registry changes and do not mutate active `SkillPackage`, `HarnessDefinition`, or `InstructionArtifact` records.

## Eval run attachment

`ProposalEvalRun` stores manually supplied or mock/local eval metadata. Governance v1 does not execute eval suites. A passed eval run can satisfy an eval requirement. A failed eval run blocks proposal readiness.

## Canary readiness

`CanaryReadiness` checks whether a proposal has a ready canary plan when the safety policy requires canary review. The check does not execute rollout and does not change active registry entries.

## Apply gate

`ProposalApplyGate` explains why a proposal cannot be applied. Governance v1 always includes `active_apply_not_implemented`, so `canApply` remains `false`.

Default blocking reasons may include:

- `active_apply_not_implemented`
- `human_approval_required`
- `eval_pass_required`
- `canary_required`
- `auto_apply_disabled`

## Readiness changes

`ProposalReadiness` now considers:

- Latest governance decision.
- Required eval ids.
- Attached eval run results.
- Canary readiness.
- Safety policy.
- Draft registry change status.

Readiness is deterministic and explanatory only.

## Audit events

Governance v1 records improvement governance audit events for:

- `proposal_decision_recorded`
- `proposal_approved`
- `proposal_rejected`
- `proposal_changes_requested`
- `proposal_eval_attached`
- `proposal_canary_readiness_checked`
- `proposal_apply_gate_checked`
- `proposal_apply_blocked`

These are separate from registry mutation audit logs because Governance v1 does not mutate registry entries.

After Registry/Governance RequestContext Migration v1, migrated governance audit events may include request id, correlation id, auth mode, source, principal id, actor kind, service account id, and policy decision id where available. This is attribution metadata only and does not enable production auth.

## API endpoints

- `GET /improvement/proposal-review-queue`
- `POST /improvement/proposals/:id/decisions`
- `GET /improvement/proposals/:id/decisions`
- `POST /improvement/proposals/:id/eval-runs`
- `GET /improvement/proposals/:id/eval-runs`
- `GET /improvement/proposals/:id/canary-readiness`
- `GET /improvement/proposals/:id/apply-gate`
- `POST /improvement/proposals/:id/apply` returns forbidden/not implemented
- `GET /improvement/governance-audit`

## Dashboard changes

The dashboard shows:

- Proposal review queue count and recommended action.
- Governance decisions.
- Proposal eval runs.
- Canary readiness blockers.
- Apply gate blockers.
- Governance audit event count.

The UI states that draft changes are not active, eval and canary are not executed automatically, auto-apply is disabled, and apply is not implemented.

## Safety constraints

- No real LLM calls.
- No embeddings.
- No external services.
- No real eval execution.
- No canary rollout execution.
- No active registry mutation.
- No proposal auto-approval.
- No apply behavior.

## Tests

Governance v1 tests cover:

- Proposal review queue inclusion and exclusion.
- Governance decisions and audit events.
- Eval run attachment.
- Canary readiness checks.
- Apply gate blockers.
- API endpoints.
- Dashboard data assumptions.
- Regression that active registry entries remain unchanged.

## Known limitations

- Governance auth still uses deterministic mock actors or mock service-account attribution; production auth and tenant scoping remain future work.
- Eval results are attached metadata only.
- Canary readiness checks do not execute rollout.
- Apply remains intentionally unavailable.
- Governance audit is in-memory for the current milestone.
