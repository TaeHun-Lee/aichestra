# PR Ownership / Handoff Model v1

## Status

PR Ownership / Handoff Model v1 is implemented as a mock-first, metadata-only ownership and handoff layer for PRs, branches, agent runs, tasks, merge queue entries, and conflict resolution plans.

It never calls GitHub APIs, assigns remote reviewers, updates remote PRs, pushes branches, merges PRs, closes PRs, deletes branches, executes LLM/provider/vendor calls, mutates user workspaces, reads credential caches, or exposes secrets/env values.

Durable Collaboration Stores v1 now adds a safe repository/schema boundary for `PrOwnershipRecord`, `PrHandoffRequest`, and `PrHandoffDecision` metadata. The PR ownership service remains local/mock-first and does not use durable storage to call GitHub or assign remote reviewers.

  - conflict resolution ownership linkage metadata

- `/dashboard/git` PR Ownership / Handoff visibility.

## What v1 Does Not Implement

- GitHub assignee or reviewer assignment
- remote PR labels, state, comments, close, or update operations
- branch push, merge, rebase, force-push, or deletion
- auto-merge
- human notification delivery
- production-ready distributed ownership locks
- production Auth/RBAC or tenant-enforced ownership storage
- agent execution handoff
- automatic merge queue hold/release

## Ownership Model

`PrOwnershipRecord` records the responsible actor for a PR/branch scope:

- repo id, branch name, optional PR id/number
- optional `BranchLease`, `WorkspaceLease`, `MergeQueueEntry`, and Conflict Resolution Plan ids
- task, task run, and agent run ids
- owner actor/principal/kind/team metadata
- reviewer actor/team metadata
- status, timestamps, and sanitized metadata

Owner kind supports `human`, `agent`, `service_account`, `team`, and `unknown`. Status supports active, review requested, handoff requested, handed off, blocked, abandoned, merged, and future closed states.

## Handoff Model

`PrHandoffRequest` records local handoff intent from one actor to another actor, team, or service-account metadata target.

Supported handoff kinds:

- `human_to_human`
- `agent_to_human`
- `human_to_agent_future`
- `service_to_human`
- `team_handoff`

Request states are `requested`, `accepted`, `rejected`, `expired`, `cancelled`, `blocked_policy`, and `blocked_missing_context`.

`PrHandoffDecision` records `accept`, `reject`, `hold`, `request_more_info`, `expired`, or `blocked`. Accepted handoffs transfer owner metadata locally only. Rejected, expired, blocked, held, or more-info decisions preserve the existing owner. `human_to_agent_future` acceptance remains blocked in v1.

## Policy Behavior

Default static policy allows metadata-only ownership/handoff reads and updates for safe mock roles when:

- `metadataOnly` is true
- `remotePrUpdate` is false
- remote reviewer assignment is false
- auto-merge and branch deletion are false
- secrets/env values are not exposed

`pr_reviewer.assign_future` and `pr_remote_update_future` are denied by default. Policy deny wins. Service accounts provide attribution only and do not bypass human review or handoff requirements.

## Merge Queue Interaction

The service can evaluate merge queue ownership readiness:

- queue entry with an active/review-requested/handoff-requested/handed-off ownership record: `owner_present`
- queue entry without such a record: `missing_owner`

This is metadata evidence for dashboard/reviewer and future merge queue policy use. v1 does not mark entries ready, release holds, execute merges, or update PRs.

## Conflict Resolution Interaction

Ownership records may link to Conflict Resolution Assistant plan ids. The service exposes owner/reviewer metadata for a plan so a conflict review can identify the responsible actor/team.

It does not mark conflict plans reviewed, apply patches, release holds, or execute validation.

## API and Dashboard Surfaces

Implemented endpoints:

- `POST /git/pr-ownership`
- `GET /git/pr-ownership`
- `GET /git/pr-ownership/:id`
- `POST /git/pr-ownership/:id/review-request`
- `POST /git/pr-ownership/:id/handoff`
- `POST /git/pr-handoffs/:id/decision`
- `GET /git/pr-handoffs`
- `GET /git/pr-ownership/summary`
- `GET /readiness/pr-ownership/summary`

`/dashboard/git` shows ownership records, reviewers, handoff requests, handoff decisions, merge queue owner readiness, and safety flags:

- remote PR update disabled
- remote reviewer assignment disabled
- GitHub API calls disabled
- auto-merge disabled

`/health` reports counts and safe booleans only.

## Safety Constraints

- metadata only
- no real remote Git operation
- no GitHub API call
- no remote PR update
- no real reviewer assignment
- no PR merge, close, or branch deletion
- no auto-merge
- no LLM/provider/vendor CLI call
- no workspace mutation
- no credential cache read
- no secrets/env values
- policy deny remains authoritative
- existing Git, Merge Queue, Branch Lease, Workspace Lifecycle, Multi-session Coordination, Auth/RBAC, Policy, Tenant Scope, Dashboard, Observability, Registry, Governance, and Safety gates remain intact

## Known Limitations

- In-memory runtime state only.
- No production tenant-filtered ownership store.
- No notification or inbox delivery.
- No remote PR assignment or provider comment updates.
- Human-to-agent handoff is modeled but acceptance is blocked as future work.
- Merge queue ownership readiness is advisory metadata in v1.
- Real Merge Execution Policy v1 consumes PR owner/reviewer readiness as future execution precondition evidence. Missing owner metadata blocks future manual readiness, but still does not update remote PRs or execute merges.
