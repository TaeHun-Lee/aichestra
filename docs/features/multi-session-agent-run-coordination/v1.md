# Multi-session Agent Run Coordination v1

Status: v1_implemented.

## Summary

Multi-session Agent Run Coordination v1 adds mock-first session coordination for multiple concurrent agent/LLM sessions aimed at the same repository, base branch, task, source scope, branch, workspace, or file set. It is metadata-only: it does not execute agents, call LLM providers, run vendor CLIs, create branches, create worktrees, switch branches, merge, rebase, push, delete workspaces, or read credential caches.

The implementation lives in `packages/runner/src/coordination.ts`, with DTOs in `packages/runner/src/dto.ts`, API routes under `/agents/sessions` and `/agents/coordination/*`, dashboard read-model visibility, and deterministic tests in `tests/multi-session-agent-run-coordination-v1.test.ts`.

## Why This Exists

One user can run LLM sessions A, B, and C against the same repo, and multiple users can run sessions against that same repo at the same time. If those sessions share a working directory, branch switches and file edits can interfere. If they reuse a branch or target the same file, review and merge work can conflict even when each run looks valid in isolation.

v1 gives Aichestra an early coordination layer so future runner execution can assign separate branches and workspaces and surface conflict risk before destructive or provider-backed work happens.

## Session Model

`AgentSession` records:

- `id`, `userId`, `actorId`;
- optional `taskId` and `taskRunId`;
- required `agentRunId`, `repoId`, `baseBranch`;
- optional `providerId`, `modelId`, `branchLeaseId`, `workspaceLeaseId`;
- optional `branchName`;
- `targetFiles` and `sourceScope`;
- status: `requested`, `assigned`, `running`, `paused`, `waiting_on_conflict`, `ready_for_review`, `ready_for_merge`, `completed`, `failed`, or `abandoned`;
- timestamps and sanitized metadata.

The model is intentionally separate from `Task`, `TaskRun`, `AgentRun`, `BranchLease`, and `AgentWorkspaceLease`. Coordination can exist before all of those related records are available.

## Coordination Group Model

`AgentRunCoordinationGroup` groups active sessions by repo and base branch, while retaining task/user metadata when the group is single-task or single-user. A group has:

- `coordinationMode`: `isolated_by_workspace`, `serialized`, `allow_parallel_with_warnings`, or `blocked`;
- `status`: `healthy`, `overlap_warning`, `conflict_risk`, `blocked`, or `requires_human_review`;
- metadata containing active session ids, branch/workspace lease ids, target source scopes, and recommended action.

## Overlap Detection

v1 uses deterministic metadata-only overlap detection from session target files, source scopes, branch names, workspace lease ids, and base branches:

- same exact file: high severity, `serialize`;
- same directory: medium severity, `split_files`;
- same branch: high severity, `serialize`;
- same workspace: critical severity, `block`;
- missing target files: medium severity, `warn`;
- base branch drift: low severity, `warn`;
- distinct branches/workspaces/files/directories: allowed.

Future overlap modes such as symbol-aware detection, semantic conflict prediction, test-impact graph analysis, and edit intent graphs are explicitly out of scope for v1.

Cross-session File Lease / Edit Intent Graph v1 now implements the first edit-intent graph slice as a separate runner-owned metadata service. Multi-session coordination still owns session grouping and high-level overlap status; the edit-intent graph owns explicit file leases, edit intent records, graph nodes/edges, overlap assessments, and no-file-lock/no-source-mutation visibility.

Multi-user / Multi-session Branch Orchestrator v2 now implements the branch allocation and ownership slice as a separate Git-adapter-owned metadata service. Multi-session coordination still detects session overlap and same-branch risk; the branch orchestrator owns safe branch name generation, active branch ownership records, branch collision decisions, workspace lease blockers at branch allocation time, and base branch drift metadata.

## Concurrency Policy

`AgentConcurrencyPolicy` seeds these default policies:

- require isolated workspace metadata;
- require unique branch metadata;
- warn on unknown overlap;
- serialize same-file overlap;
- block same-workspace overlap;
- keep symbol-aware overlap as future metadata only.

These policies are coordination hints and blockers. They do not grant runner execution, remote Git, remote LLM, secret access, workspace mutation, or policy bypass.

## Branch And Workspace Leases

`AgentRunCoordinationService` can link `branchLeaseId` to existing `BranchLease` metadata and `workspaceLeaseId` to Agent Workspace Lifecycle v2 `AgentWorkspaceLease` metadata when those lookups are available.

Branch lease interaction:

- sessions can inherit task/run/base branch/target files from a branch lease;
- active sessions using the same `branchName` produce a high severity overlap;
- v1 does not create, switch, push, merge, rebase, or delete branches.
- Branch Orchestrator v2 can allocate or link branch ownership metadata that sessions may reference later; coordination remains advisory and does not own branch naming policy.

Workspace lease interaction:

- sessions can link to `AgentWorkspaceLease.id`;
- active sessions using the same `workspaceLeaseId` produce a critical same-workspace blocker;
- missing workspace leases remain visible as metadata gaps for future allocation;
- v1 does not create real worktrees or delete workspaces.

## Merge Queue Interaction

v1 does not change merge queue execution. Merge Queue Policy v2 now consumes session coordination metadata as readiness evidence:

- same-workspace blockers produce merge readiness holds;
- same-file and same-branch overlap recommends serialization or review before queueing and can contribute warning/hold evidence;
- `ready_for_review` and `ready_for_merge` are session states only and do not merge anything;
- `BranchLease`, `ConflictRisk`, `MergeQueueEntry`, and `MergeSimulationResult` remain separate domain concepts.
- Merge Queue Policy v2 records decisions and holds as metadata only; it does not execute merges, push, rebase, update provider PRs, or mutate workspaces.

## API And Dashboard Surfaces

API endpoints:

- `POST /agents/sessions`;
- `GET /agents/sessions`;
- `POST /agents/sessions/:id/target-files`;
- `POST /agents/sessions/:id/branch-lease`;
- `POST /agents/sessions/:id/workspace-lease`;
- `POST /agents/sessions/:id/ready-for-review`;
- `POST /agents/sessions/:id/ready-for-merge`;
- `GET /agents/coordination/groups`;
- `GET /agents/coordination/groups/:id/recommendation`;
- `GET /agents/coordination/overlaps`;
- `GET /agents/coordination/summary`;
- `GET /agents/coordination/policies`;
- `GET /agents/coordination/audit`;
- `POST /agents/coordination/evaluate`.

Dashboard read models expose active sessions, groups, overlaps, concurrency policies, audit events, branch/workspace assignment status, blockers, recommended actions, merge readiness counts, and no-destructive-action safety flags.

Cross-session File Lease / Edit Intent Graph v1 adds companion `/agents/edit-intents`, `/agents/file-leases`, `/agents/edit-intent-graph`, `/agents/edit-overlaps`, and `/agents/edit-intent-summary` endpoints plus dashboard rows for file leases and edit overlap summaries. These endpoints are metadata-only and do not execute agents or mutate files.

Multi-user / Multi-session Branch Orchestrator v2 adds companion `/git/branches/orchestrate`, `/git/branches/orchestration*`, `/git/branches/ownership`, and `/git/branches/drift` endpoints plus `/dashboard/git` branch orchestrator rows. These endpoints are metadata-only and do not create branches, switch branches, mutate workspaces, or call providers.

## Safety Constraints

v1 preserves the default safe runtime:

- no real LLM calls;
- no vendor CLI execution;
- no remote Git operations;
- no branch checkout/switch/merge/rebase/push/delete;
- no real worktree creation;
- no destructive workspace cleanup;
- no credential-cache reads;
- no secret or env value exposure;
- no weakening of Auth/RBAC, Policy, Git, LLM, Runner, Local Agent, Registry, Governance, Dashboard, Observability, Tenant Scope, or Secrets/Sandbox gates.

Audit metadata is sanitized, and summary/readiness metadata explicitly marks `noDestructiveAction: true`, `remoteGitOperation: false`, `agentExecution: false`, and `secretsExposed: false`.

## Known Limitations

- In-memory repository only.
- No distributed locks.
- No durable coordination table.
- No symbol-aware overlap detection.
- No semantic merge prediction.
- No test-impact graph.
- No merge execution or provider mutation. Merge Queue Policy v2 consumes coordination metadata for readiness decisions and holds, but enforcement remains metadata-only.
- No real workspace allocation or branch creation.
- Branch Orchestrator v2 owns branch allocation metadata separately; coordination does not provide a distributed branch lock.
- Edit-intent graph data is separate and in-memory; coordination does not consume it for durable distributed locking or merge execution.
