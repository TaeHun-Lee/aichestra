# Merge Queue Policy v2

## Status

Merge Queue Policy v2 is implemented as a mock-first, metadata-only readiness layer.

It does not execute merges. It does not call remote Git providers, fetch, push, rebase, force-push, delete branches, update PRs, run vendor CLIs, call LLM providers, mutate user workspaces, or expose secrets/env values.

## What v2 Implements

- Core merge queue policy models:
  - `MergeQueuePolicy`
  - `MergeReadinessDecision`
  - `MergeQueuePriorityRule`
  - `MergeQueueHold`
- `MergeQueuePolicyService` for deterministic evaluation, ranking, holds, hold release metadata, decision listing, hold listing, and queue summaries.
- Policy-as-code actions:
  - `merge_queue.read`
  - `merge_queue.evaluate`
  - `merge_queue.hold`
  - `merge_queue.release_hold`
  - `merge_queue.merge_execute_future`
- A default active mock policy that denies future merge execution.
- API metadata endpoints under `/git/merge-queue/*`.
- Dashboard conflict panel visibility for policy decisions, holds, blockers, warnings, priority order, and disabled merge execution.
- Tests for ready, hold, blocked/review, ranking, policy-deny, API safety, dashboard rendering, and no merge-execution enablement.

## Policy Model

`MergeQueuePolicy` defines:

- `id`, `name`, and `status` (`active_mock`, `disabled`, or `future`)
- required checks
- blocking conditions
- warning conditions
- priority rules
- retry rules
- sanitized metadata

The default policy is `active_mock`. Its metadata explicitly records:

- `mergeExecutionEnabled: false`
- `remoteGitOperation: false`
- `autoMergeEnabled: false`
- `forcePushEnabled: false`
- `rebaseEnabled: false`
- `branchDeletionEnabled: false`
- `externalProviderCalls: false`
- `secretsExposed: false`
- `envValuesExposed: false`

## Readiness Decisions

`MergeReadinessDecision` records the current policy view of a queue entry:

- `ready`
- `hold`
- `blocked`
- `warning`
- `needs_human_review`
- `future_manual_merge`

Each decision includes reasons, blocking reasons, warnings, required actions, priority, request/correlation/actor/service-account metadata, timestamp, and sanitized evidence metadata.

Readiness is conservative:

- policy deny blocks.
- expired or missing branch lease blocks.
- dry-run text conflict blocks.
- dry-run failure requires human review.
- high/critical conflict risk creates a hold.
- missing validation or approval creates a hold.
- workspace not ready creates a hold.
- same-file or same-workspace overlap creates a hold.
- ready requires safe evidence and still requires the future/manual merge gate because execution is disabled.

## Priority Rules

Priority is deterministic. Ranking considers:

- readiness class
- existing queue priority
- lower conflict score first
- smaller changed-file footprint first
- human priority metadata
- release blocker metadata
- stable queue-entry tie-breaking

The ranking only orders metadata. It never starts a merge.

## Hold Rules

`MergeQueueHold` supports:

- `conflict_risk`
- `dry_run_failed`
- `validation_missing`
- `approval_missing`
- `workspace_not_ready`
- `branch_lease_expired`
- `edit_overlap`
- `policy_denied`
- `human_review_required`

Releasing a hold updates hold metadata only. It does not mark an entry ready; the service must re-evaluate current evidence.

## Inputs

The service consumes evidence where available:

- existing merge queue entry metadata
- branch lease status
- highest conflict risk for the lease
- latest dry-run merge simulation
- Agent Workspace Lifecycle v2 workspace lease snapshots
- Multi-session Agent Run Coordination v1 overlap snapshots
- Cross-session File Lease / Edit Intent Graph v1 overlap assessments
- validation/approval request metadata
- human priority/release blocker metadata
- Policy-as-code decisions

Workspace paths remain redacted at API/dashboard boundaries. Secret-like metadata is sanitized.

## API Surface

Implemented metadata endpoints:

- `GET /git/merge-queue/policy`
- `GET /git/merge-queue/decisions`
- `GET /git/merge-queue/holds`
- `GET /git/merge-queue/summary`
- `POST /git/merge-queue/:entryId/evaluate`
- `POST /git/merge-queue/:entryId/hold`
- `POST /git/merge-queue/:entryId/release-hold`
- `POST /git/merge-queue/:entryId/release-hold/:holdId`

These endpoints return policy metadata only. They do not merge, rebase, fetch, push, delete branches, call GitHub, update remote PRs, run agents, call LLMs, or expose secrets/env values.

## Dashboard Surface

The dashboard conflict panel now shows:

- queue entries
- policy status
- readiness decisions
- holds
- blocking reasons
- warnings
- required actions
- priority order
- conflict risk and dry-run context
- disabled merge execution and disabled auto-merge status

The Git panel merge queue linkage includes the policy decision and explicit `mergeExecutionEnabled: false`.

## Policy/Auth Behavior

Default static policy allows mock/readiness metadata reads and evaluation for safe roles when:

- metadata-only mode is true
- real merge execution is false
- remote Git operation is false
- auto-merge is false

`merge_queue.merge_execute_future` is denied by default. Policy deny wins over readiness evidence.

## No-auto-merge Guarantee

Merge Queue Policy v2 does not implement auto-merge or production merge execution. It only computes readiness and holds. Any future execution path must be a separate gated task with explicit policy, Auth/RBAC, tenant scope, Git provider, branch lease, workspace lifecycle, observability, and safety checks.

## Known Limitations

- Decisions and holds are in-memory metadata in the default runtime.
- Validation and approval evidence is supplied as request/readiness metadata; there is no durable validation result repository yet.
- Ranking is deterministic but not dependency-graph aware beyond metadata hooks.
- Hold release is metadata-only and requires re-evaluation.
- Production tenant enforcement and durable merge queue storage remain future work.
