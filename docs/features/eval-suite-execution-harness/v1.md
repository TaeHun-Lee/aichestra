# Eval Suite Execution Harness v1

Eval Suite Execution Harness v1 adds a mock-first deterministic evaluation boundary for registry candidates and governance proposals. It executes local metadata checks only and records reviewable eval evidence without calling real providers or enabling canary/apply workflows.

Status:

- Eval Suite Execution Harness: `v1_implemented`
- Eval execution: mock deterministic only
- External eval: not implemented
- Auto-apply: disabled

## What v1 Implements

- `RegistryEvalSuite`, `RegistryEvalCase`, `RegistryEvalRun`, `RegistryEvalCaseResult`, and `RegistryEvalVerdict` models.
- `EvalSuiteExecutionService` in `packages/registry`, with deterministic suite listing, run requests, mock execution, case results, verdicts, eval result attachment, proposal eval-run attachment, and summary metadata.
- Seed suites for skill required metadata, harness safety, instruction compatibility, registry package artifact trust, compatibility matrix follow-up, drift follow-up, and a blocked future provider-live placeholder.
- API endpoints for `/registry/eval-suites`, `/registry/eval-runs`, and `/readiness/registry/evals/summary`.
- Dashboard Registry panel visibility for suite counts, run verdicts, external-eval disabled state, canary/apply relation, and no-secret status.
- Policy actions for eval suite read/run/attach and denied future external/override paths.
- Governance/proposal readiness linkage through `ProposalEvalRun` metadata when an eval result is explicitly attached to a proposal requirement.

## What v1 Does Not Implement

- Real external eval suites.
- Real LLM, MCP, vendor CLI, GitHub, artifact registry, Vault, Kubernetes, Temporal, OPA, Cedar, cloud, or external auth calls.
- Provider output grading, embeddings, benchmark harnesses, or remote fixture downloads.
- Canary execution.
- Apply execution or auto-apply.
- Active registry entry mutation through auto-improvement.
- Production eval scheduling, queueing, or durable eval result storage.

## Eval Suite Model

`RegistryEvalSuite` defines suite id/name, `suiteKind`, supported target kinds, status, required inputs, forbidden capabilities, timeout metadata, and sanitized metadata. v1 can run only `mock_deterministic`, `fixture_static`, `compatibility`, and `drift_followup` suites. `provider_live_future` and `external_future` suites return blocked/future metadata.

## Eval Case Model

`RegistryEvalCase` defines case id, suite id, `caseKind`, input, expected output, expected verdict, and metadata. v1 cases inspect registry/readiness metadata such as required fields, lifecycle/approval/eval/checksum gates, artifact trust checksum metadata, compatibility gate status, and drift follow-up evidence.

## Eval Run And Verdict Model

`RegistryEvalRun` records suite/target/proposal linkage, status, actor/service-account attribution, request/correlation ids, and metadata. `RegistryEvalCaseResult` records per-case verdicts and bounded summaries. `RegistryEvalVerdict` aggregates pass/fail/warning/skipped counts and marks eval evidence as required for future apply/canary decisions.

## Deterministic Mock Execution

Mock execution is local and in-memory. It never calls providers, executes tools, runs vendor CLIs, downloads fixtures, or evaluates model output. A run can become:

- `passed` when all cases pass.
- `failed` when registry gate or required metadata checks fail.
- `warning` for advisory drift or pending-gate conditions.
- `skipped`/`future_external` for future provider suites.
- `blocked_policy` or `blocked_missing_target` for denied or missing evidence.

## Registry Integration

Eval run attachment can create local `RegistryEvalResult` metadata for Skill, Harness, or Instruction targets. Attachment does not change active registry content unless an explicit future workflow updates eval status through existing registry gates. Resolver lifecycle, approval, eval, checksum, semver, policy, tenant scope, artifact trust, and governance gates remain authoritative.

## Governance / Proposal Readiness Integration

Eval run attachment can also create `ProposalEvalRun` metadata when a proposal id and eval requirement id are supplied. Proposal readiness can then see passed/failed eval evidence. This does not approve proposals, execute canaries, or enable apply.

## Canary / Apply Gate Relationship

v1 records eval evidence only. Canary readiness and apply gates remain independent blockers. Failed eval metadata keeps readiness blocked. Passed eval metadata can satisfy eval requirements, but human approval, canary evidence, apply policy, and future execution gates still decide the next state. `autoApplyEnabled` remains false.

## Policy / Auth Behavior

New or verified policy actions:

- `registry.eval_suite.read`
- `registry.eval_suite.run_mock`
- `registry.eval_suite.attach_result`
- `registry.eval_suite.run_external_future`
- `governance.eval.require`
- `governance.eval.override_future`

Read, mock run, attach, and require actions are allowed in safe mock/readiness contexts when policy permits. External/future runs and overrides are denied by default. Policy deny wins.

## API / Dashboard Surfaces

API endpoints:

- `GET /registry/eval-suites`
- `GET /registry/eval-suites/:id/cases`
- `POST /registry/eval-runs`
- `POST /registry/eval-runs/:id/execute-mock`
- `GET /registry/eval-runs`
- `GET /registry/eval-runs/:id`
- `GET /registry/eval-runs/:id/results`
- `GET /registry/eval-runs/:id/verdict`
- `POST /registry/eval-runs/:id/attach`
- `GET /readiness/registry/evals/summary`

Dashboard:

- `/dashboard/registry` shows the Eval Suite Execution Harness panel.
- `/health.registryEvalSuites` shows status, suite/case/run counts, external eval disabled, provider-call false, canary false, auto-apply false, and no-secret/no-env status.

## Safety Guarantees

- No real external eval suite execution.
- No real LLM/MCP/provider/vendor CLI calls.
- No canary execution.
- No auto-apply.
- No active registry mutation through auto-improvement.
- No secret or env value exposure.
- Registry resolver and governance gates are preserved.

## Known Limitations

- Run/case/verdict repositories are in-memory only.
- Eval cases inspect metadata, not real model outputs or code execution results.
- No scheduled eval queue exists.
- No live provider profile exists.
- Proposal target ids from governance metadata may differ from registry object ids; v1 records link metadata but does not attempt production-grade target identity reconciliation.
