# MCP Gateway v0

## What v0 Implements

MCP Gateway v0 adds a mock-first tool governance layer. It defines provider-neutral MCP server/tool catalog models, deterministic mock tool invocation, disabled real transport skeletons, in-memory repositories, Auth/RBAC and Policy-as-code checks, Secrets/Sandbox redaction and network-deny integration, API endpoints, health metadata, dashboard read models, and deterministic tests.

It does not call real MCP servers and is not production MCP integration.

## Server And Tool Catalog

`packages/mcp-gateway` owns:

- `MCPServerCatalogEntry`
- `MCPToolDefinition`
- `MCPToolInvocationRequest`
- `MCPToolInvocationResult`
- `MCPToolAuditEvent`
- `MCPGateway`
- `MockMCPGateway`
- `DisabledRealMCPTransportGateway`

Seeded mock servers:

- `mock-github-mcp`
- `mock-docs-search-mcp`
- `mock-jira-mcp`
- `mock-db-schema-mcp`
- `mock-ci-mcp`

Seeded active low-risk tools are read-only and deterministic. Secret-requiring tools are blocked by default. High-risk write tools, critical DB/deploy tools, and future real transport entries are disabled or denied.

## Invocation Flow

`MockMCPGateway.invokeTool`:

1. Resolves the server/tool catalog record.
2. Rejects unknown, disabled, deprecated, or non-mock transport entries.
3. Resolves `AuthContext`.
4. Checks `AuthorizationService`.
5. Checks Policy-as-code for MCP invocation actions.
6. Blocks secret-requiring, network, write, deploy, and local-execution tools.
7. Runs only deterministic mock read-only output.
8. Redacts and size-limits previews.
9. Records invocation and MCP audit metadata.

No raw secrets, provider tokens, API keys, or credential-cache content are stored.

## Auth/RBAC Integration

Auth/RBAC adds MCP permissions for:

- `mcp.server.list`
- `mcp.tool.list`
- `mcp.tool.invoke.low_risk`
- future high/critical invocation boundaries
- `mcp.audit.read`

Viewer can list MCP metadata. Developer can invoke allowed low-risk read-only mock tools. Service account runner can invoke scoped low-risk tools when the request scope matches. No role bypasses high-risk or critical deny-by-default policy.

Production Auth/RBAC v1 Planning adds read-only future service-account, tenant/scope, request context, and permission matrix planning for MCP Gateway actors through `/readiness/auth/*` and `/dashboard/auth-production`. It does not enable real MCP transport, IdP calls, sessions, tenant enforcement, or service-account credential issuance.

## Policy Integration

Policy-as-code adds:

- `mcp.server.list`
- `mcp.tool.list`
- `mcp.tool.invoke`
- `mcp.tool.invoke.low_risk`
- `mcp.tool.invoke.high_risk`
- `mcp.tool.invoke.critical`
- `mcp.tool.secret.resolve`
- `mcp.tool.network_access`
- `mcp.tool.write`
- `mcp.tool.deploy`

Default rules allow listing and low-risk mock read-only invocation under safe environment flags. They deny high-risk, critical, secret, network, write, deploy, and legacy `mcp.tool.call` paths by default.

## SecretRef, Secrets, Sandbox, And Redaction

MCP tools that declare `requiredSecretRefs` are blocked in v0 and receive no SecretLease. No raw secret is forwarded to mock tools. Network egress remains denied by default. Tools that imply local execution, process spawning, or real transport remain unavailable.

Secret Backend Migration Planning v0 models future `future_mcp_tool_secret` credential migration, lease TTL, rotation, and readiness blockers. It does not issue MCP tool secrets, connect to real secret backends, or enable real MCP transport.

Input/output/audit previews are passed through the existing redaction boundary before persistence or API/dashboard exposure.

## LLM, Runner, And Local Agent Integration Points

LLM Gateway v2 can expose MCP catalog metadata in future, but v0 does not auto-execute model-generated tool calls. Runner metadata can reference allowed MCP tools in future, but runners do not execute tools directly. Local Agent MCP transport is a disabled future placeholder.

## API Endpoints

- `GET /mcp/config`
- `GET /mcp/servers`
- `GET /mcp/servers/:id`
- `GET /mcp/servers/:id/tools`
- `GET /mcp/tools/:id`
- `POST /mcp/tools/:id/invoke`
- `GET /mcp/invocations`
- `GET /mcp/invocations/:id`
- `GET /mcp/audit`

Responses are DTO-based and never expose secrets or raw unredacted output.

## Health And Dashboard

`GET /health` reports MCP gateway kind, real MCP transport enabled false, server count, active tool count, high/critical enabled count, external call status, and secret/network forwarding status.

`/dashboard/mcp` shows gateway config, server catalog, tool catalog, risk summary, recent invocations, blocked examples, integration safety status, and recent audit events.

## Persistence

Runtime repositories are in-memory in v0:

- `InMemoryMCPServerCatalogRepository`
- `InMemoryMCPToolDefinitionRepository`
- `InMemoryMCPToolInvocationRepository`
- `InMemoryMCPToolAuditRepository`

Future Postgres table skeletons are documented in `docs/foundations/persistent-storage-schema-v0.md`.

## Tests

`tests/mcp-gateway-v0.test.ts` covers:

- domain model/repository creation;
- low-risk deterministic mock invocation;
- unknown, disabled, high-risk, critical, secret, and real transport blocks;
- Policy-as-code and Auth/RBAC behavior;
- scoped service-account invocation;
- API endpoints;
- health/dashboard visibility;
- no-secret exposure.

## Known Limitations

- No real MCP server transport.
- No stdio/http/sse MCP calls.
- No GitHub/Jira/Slack/DB/search real integrations.
- No tool-generated workflow execution.
- No model-generated automatic tool execution.
- No SecretLease issuance to tools.
- Runtime repositories remain in-memory.
- Production auth, tenant scoping, and real connector governance remain future work.
