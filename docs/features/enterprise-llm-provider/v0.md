# Enterprise LLM Provider Abstraction v0

## What v0 Implements

Enterprise LLM Provider Abstraction v0 incorporates the closed enterprise provider design into Aichestra's mock-first architecture.

It adds provider catalog, auth, credential, token, adapter, Local Agent boundary, parser, redaction, audit, API, dashboard, policy, and schema skeletons. It does not make real provider calls.

## ProviderKind Model

`packages/llm-gateway/src/enterprise-providers.ts` defines:

- `cloud_api`
- `oauth_api`
- `workload_identity_api`
- `cloud_iam`
- `local_cli`
- `pty_interactive_fallback`

These are enterprise provider transport/auth classes, separate from the older LLM model provider kinds such as `mock` and `openai_compatible`.

## ProviderAuth Model

Supported auth union:

- `api_key`: env key or secret reference only; no raw key value.
- `oauth_user`: future OAuth user path.
- `device_code`: future device-code path.
- `workload_identity`: future WIF path.
- `cloud_iam`: future cloud IAM path.
- `external_cli_session`: local CLI session boundary with `credentialAccess = never_read_tokens`.

Validation rejects raw tokens, credential cache paths, local CLI auth other than `external_cli_session`, PTY default enablement, and shared user CLI sessions.

## ProviderCatalog

Seeded disabled/skeleton entries:

- `anthropic-api-key`
- `anthropic-wif`
- `claude-code-local`
- `openai-api-key`
- `codex-cli-local`
- `gemini-api-key`
- `gemini-cli-local`
- `vertex-gemini-cloud`
- `bedrock-anthropic-cloud`
- `azure-foundry-cloud`

Entries record vendor, kind, auth type, supported models, billing mode, capabilities, policy notes, metadata, and lifecycle status. No provider is enabled for real calls.

## CredentialManager

`StaticCredentialManager`:

- returns credential references only;
- returns no token for local CLI providers;
- records credential access audit;
- denies credential cache read attempts;
- keeps direct provider-adapter secret resolution blocked; API-key catalog entries may reference SecretRef ids for the separate `SecurityControlService` credential boundary.

It never reads `~/.codex/auth.json`, `~/.claude`, Google ADC cache files, keychains, or vendor credential stores.

## TokenResolver

`MockTokenResolver`:

- returns empty headers/env for local CLI providers;
- returns not-implemented for real token headers, refresh, OAuth, WIF, and IAM exchange;
- returns no raw provider secrets.

## ProviderAdapter

Skeleton adapters:

- `CloudApiProviderAdapter`
- `OAuthApiProviderAdapter`
- `WorkloadIdentityProviderAdapter`
- `CloudIamProviderAdapter`
- `LocalCliProviderAdapter`
- `PtyInteractiveFallbackProviderAdapter`

Cloud/OAuth/WIF/IAM adapters return blocked results. LLM Gateway v1 can use the `openai-api-key` catalog entry as sanitized provider metadata for the controlled `openai_compatible` HTTP path, but ProviderAdapter cloud invocation remains blocked. LLM Gateway v2 adds provider-aware routing read models and disabled skeleton routes for Anthropic, Gemini, Bedrock, Vertex, Azure, LiteLLM, and Local CLI provider kinds. Those skeleton routes do not invoke ProviderAdapter cloud calls or vendor CLIs. Local CLI now routes through Local Agent Protocol v1 when a compatible fixture Local Agent, established mock channel, and approved consent exist; otherwise it returns `local_agent_required`, `local_agent_unavailable`, `channel_required`, `awaiting_consent`, `consent_denied`, or `provider_template_incompatible`. PTY returns `disabled_by_policy`.

## Local CLI Provider Contract

Local CLI templates are metadata only. Local CLI Provider Templates v1 expands this into first-class template, compatibility, parser, security constraint, and readiness metadata:

- Claude headless JSON
- Claude stream JSON
- Codex headless
- Codex JSONL
- Gemini JSON
- Aider diff summary
- Custom local CLI raw

The v1 read model exposes `claude-code-template-v1`, `codex-cli-template-v1`, `gemini-cli-template-v1`, `aider-template-v1`, and `custom-local-cli-template-v1`. All v1 templates remain `template_only`, Local Agent required, execution disabled, PTY unsupported, credential-cache reads denied, and secret forwarding denied.

Templates use argv arrays, not shell strings. They are not executed in v0.

Rules:

- Aichestra Cloud does not execute vendor CLIs directly.
- Aichestra must not read local vendor credential caches.
- Local CLI requires Aichestra Local Agent Protocol v1 coordination and cannot be executed directly by Aichestra Cloud.
- Danger/full-access modes are denied by default.

## Aichestra Cloud vs Aichestra Local Agent

Local Agent Runner v1 is Aichestra's mock/local task runner.

Aichestra Local Agent Protocol v1 is the mock-first user-machine coordination layer for future vendor local CLI providers. It defines registration, heartbeat, capabilities, mock sessions, mock signed channels, fixture daemon simulation, invocation envelopes, consent, compatibility checks, normalized streams/events, audit, and in-memory transport:

- `LocalAgentRegistration`
- `LocalAgentCapability`
- `LocalAgentSession`
- `LocalAgentInvocationEnvelope`
- `LocalAgentInvocation`
- `LocalAgentConsentRequest`
- `LocalAgentConsentDecision`
- `LocalAgentNormalizedEvent`
- `LocalAgentChannel`
- `LocalAgentHandshake`
- `LocalAgentCapabilityAdvertisement`
- `LocalCliCompatibilityEntry`
- `LocalAgentInvocationStream`
- `LocalAgentStreamEvent`

No daemon, socket protocol, WebSocket/gRPC/HTTP tunnel, PTY automation, or vendor CLI execution is implemented.

## Parser Strategy

Provider parser utilities support:

- `raw`
- `json`
- `jsonl`
- `ndjson`

Malformed JSON returns deterministic parse errors. Stderr progress logs do not automatically mean failure. Exit code plus parser status determines success/failure.

## Redaction Strategy

`redactSecretText` masks:

- bearer tokens;
- API key-like strings;
- `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `AICHESTRA_LLM_API_KEY`, `LLM_API_KEY`, `GITHUB_TOKEN`, and Google credential env dumps;
- `~/.codex/auth.json`;
- `~/.claude`;
- Google ADC cache references.

Provider audit metadata also redacts secret, token, key, prompt, and credential fields.

## Known Limitations

- No real provider API calls through Enterprise ProviderAdapter skeletons. LLM Gateway v1/v2 has a separate controlled OpenAI-compatible HTTP provider path behind explicit gates.
- No real OAuth/device-code/WIF/IAM exchange.
- SecretRef-backed provider credentials are available through `packages/security` for cloud API style credentials, including Auth/RBAC and Policy checks before env reads. Provider adapter skeletons remain blocked and no production secret manager is implemented.
- No real Aichestra Local Agent daemon or network transport.
- No vendor CLI execution.
- No PTY automation.
- Provider catalog and provider audit are in-memory at runtime.
- No production provider governance or legal/terms validation.
