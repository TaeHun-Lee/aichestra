# Real Merge Execution Policy v1

Status: `v1_implemented`

Merge execution: `disabled`

Auto-merge: `disabled`

Remote push: `disabled`

## What v1 Implements

Real Merge Execution Policy v1 defines a mock-first policy/readiness boundary for any future real merge path. It adds:

- `RealMergeExecutionPolicy` with `mergeExecutionEnabled: false`, `autoMergeEnabled: false`, and `remotePushEnabled: false`;
- `RealMergeExecutionRequest` metadata for repo, base/source branch, merge queue entry, branch lease, workspace lease, PR ownership, conflict resolution plan, dry-run merge, request/correlation, actor/principal, and sanitized metadata;
- `RealMergeExecutionDecision` metadata with blocking reasons, warnings, required actions, and hard false side-effect flags;
- `RealMergePrecondition` checklist metadata for branch lease, workspace, worktree, merge queue, dry-run, conflict resolution, validation, approval, PR ownership, tenant scope, policy, auth, observability, and rollback;
- `RealMergeForbiddenOperation` metadata for auto-merge, remote push, force-push, rebase, branch deletion, worktree deletion, policy bypass, approval bypass, and dry-run bypass;
- `RealMergePostExecutionEvidenceTemplate` for future implementation planning only;
- deterministic in-memory/mock `RealMergeExecutionPolicyService`;
- Policy-as-code actions/resources for safe metadata read/evaluate/request and denied future execution/override/post-evidence actions;
- read-only API, readiness, health, and dashboard visibility.

## What v1 Does Not Implement

v1 does not execute merges. It does not run `git merge`, provider merge APIs, remote push, fetch, rebase, force-push, checkout/switch, branch deletion, worktree deletion, PR updates, reviewer assignment, auto-merge, rollback/revert, validation commands, LLM calls, vendor CLI calls, or external provider calls.

`future_execution_allowed` is modeled for type completeness but is not returned by the v1 service. The highest successful state is `ready_for_manual_future`, which means only that modeled metadata preconditions passed and that a later explicit execution profile could be considered.

## Execution Policy Model

`RealMergeExecutionPolicy` records:

- policy id/name/status;
- `mergeExecutionEnabled: false`;
- `autoMergeEnabled: false`;
- `remotePushEnabled: false`;
- required evidence, approvals, and checks;
- forbidden operations;
- allowed future mode labels;
- safe metadata.

The default policy status is `policy_defined`, not production-ready.

## Decision Model

`RealMergeExecutionDecision.decision` may be:

- `denied_default`;
- `blocked_missing_approval`;
- `blocked_missing_dry_run`;
- `blocked_conflict_risk`;
- `blocked_workspace_not_ready`;
- `blocked_branch_lease_invalid`;
- `blocked_scope_mismatch`;
- `blocked_policy_denied`;
- `blocked_no_owner`;
- `ready_for_manual_future`.

Every decision records:

- `mergeExecutionPerformed: false`;
- `autoMergePerformed: false`;
- `remotePushPerformed: false`.

## Preconditions

Future merge execution would require, at minimum:

- active scope-matching `BranchLease`;
- ready merge queue entry;
- clean dry-run merge;
- low or accepted conflict risk;
- no unresolved high/critical edit-intent overlap;
- no unresolved conflict resolution plan;
- PR owner/reviewer metadata;
- validation green or explicitly not required;
- approval/signoff present;
- tenant/repo scope match;
- Policy-as-code allows metadata request while denying v1 execution;
- rollback/revert plan metadata;
- request/correlation audit metadata.

v1 only records these as precondition statuses. It does not enforce them by executing Git.

## Forbidden Operations

The service exposes the forbidden operation list through API/dashboard/readiness:

- `auto_merge`;
- `remote_push`;
- `force_push`;
- `rebase`;
- `branch_delete`;
- `worktree_delete`;
- `bypass_policy`;
- `bypass_approval`;
- `bypass_dry_run`.

## API / Readiness / Dashboard

API endpoints:

- `GET /git/merge-execution/policy`;
- `GET /git/merge-execution/forbidden-operations`;
- `GET /git/merge-execution/post-evidence-template`;
- `POST /git/merge-execution/evaluate`;
- `GET /git/merge-execution/decisions`;
- `GET /git/merge-execution/summary`.

Readiness:

- `GET /readiness/merge-execution/summary`.

Dashboard:

- `/dashboard/conflicts` includes a Real Merge Execution Policy panel with enabled flags, latest decisions, blocking reasons, preconditions, forbidden operations, and post-evidence template status.

All surfaces expose booleans/counts/sanitized metadata only. They do not expose secrets/env values or raw credentials.

## Relationship With Merge Queue Policy

Merge Queue Policy v2 remains a readiness/hold/ranking layer and continues to deny `merge_queue.merge_execute_future`. Real Merge Execution Policy v1 consumes merge queue entry/readiness metadata as precondition evidence. A ready queue entry is necessary future evidence but never sufficient to execute a merge.

## Relationship With Conflict Resolution Assistant

Conflict Resolution Assistant v1 plans remain review-only with `applyAllowed: false`. Real Merge Execution Policy v1 can treat draft, review-required, or blocked conflict plans as blockers, and reviewed no-apply plans as metadata evidence. It does not apply conflict plans or release holds.

## Relationship With PR Ownership / Handoff

PR Ownership / Handoff v1 supplies owner/reviewer readiness evidence. Missing owner metadata blocks future manual readiness. No remote PR assignment, reviewer assignment, PR update, PR close, or merge is performed.

## Relationship With Worktree / Workspace Lifecycle

Agent Workspace Lifecycle v2 and Agent Worktree Allocation v1 provide optional readiness metadata. A linked workspace that is not ready blocks future manual readiness. Production `git worktree` allocation and destructive cleanup remain disabled.

## Relationship With Tenant / Repo Scope

Tenant/Repo/Provider Scope metadata and Tenant Scope Enforcement v1 are consumed as representative metadata. Scope mismatch blocks evaluation. v1 does not claim production tenant isolation.

## Post-Execution Evidence Template

The post-execution evidence template defines future fields such as merge request id, decision id, future merge commit SHA, future merged-by actor, validation result ids, audit requirements, and rollback checkpoint id. It is a template only and is not evidence of executed merge.

## Safety Guarantees

- No real merge.
- No auto-merge.
- No remote push/fetch/rebase/force-push/branch delete.
- No remote PR update.
- No workspace mutation.
- No GitHub API calls by default.
- No LLM/provider/vendor CLI calls.
- No credential cache reads.
- No secrets/env values in runtime surfaces or tests.
- Policy deny wins.

## Known Limitations

- In-memory decisions/preconditions are not durable.
- Approval/signoff is metadata-only.
- Tenant enforcement is representative scaffolding, not production isolation.
- No live merge integration profile exists.
- Rollback evidence is modeled but not executable.
