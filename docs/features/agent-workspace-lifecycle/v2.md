# Agent Workspace Lifecycle v2

Status: `v2_implemented`

Agent Workspace Lifecycle v2 adds a mock-first workspace lease model for per-agent-run isolation. It makes workspace ownership, lifecycle state, branch lease linkage, and cleanup eligibility explicit without enabling production worktree creation or destructive cleanup.

Multi-session Agent Run Coordination v1 now consumes `AgentWorkspaceLease.id` as optional `workspaceLeaseId` metadata. Shared active workspace leases are surfaced as critical coordination blockers, while workspace lifecycle remains responsible only for workspace lease lifecycle state and cleanup eligibility.

Multi-user / Multi-session Branch Orchestrator v2 now consumes optional `workspaceLeaseId` metadata on branch ownership records. Shared active workspace leases produce branch orchestration blockers even when branch names differ, because branch-only separation does not protect a shared checkout. Workspace lifecycle still owns only lease lifecycle and cleanup decision metadata.

Cross-session File Lease / Edit Intent Graph v1 now consumes optional `workspaceLeaseId` metadata for edit intent overlap assessment. Same workspace edit-intent overlap is a critical metadata blocker, but workspace lifecycle remains the only owner of workspace lease lifecycle and cleanup decision state.

## Why Per-Agent Workspaces Are Required

A single physical repository working directory is unsafe for concurrent LLM sessions. If LLM B switches branches in a shared working tree, LLM A and LLM C are affected immediately. Aichestra now models the intended isolation chain as:

```text
Task / AgentRun
  -> BranchLease
  -> WorkspaceLease
  -> isolated worktree path
  -> lifecycle state
  -> cleanup eligibility
```

`BranchLease` remains the Git/conflict-scope lease. `AgentWorkspaceLease` is the runner/workspace lifecycle lease.

## What v2 Implements

- `AgentWorkspaceLease`, `AgentWorkspaceLifecycleEvent`, and `AgentWorkspaceCleanupDecision` domain models in `packages/runner`.
- An in-memory `AgentWorkspaceLifecycleService` with deterministic mock behavior.
- Fixture-only allocation through `LocalAgentWorkspaceManager` validation.
- One active workspace lease per `AgentRun` by default.
- Shared path detection with `isolationStatus = shared_forbidden`.
- `branchLeaseId` linkage when supplied by Git/conflict flows.
- `workspaceLeaseId` in `AgentRun` result metadata and runner audit metadata.
- Safe lifecycle API endpoints under `/agents/workspaces/*`.
- Dashboard/readiness visibility for active leases, workspace kind, lifecycle status, isolation status, cleanup decisions, branch lease linkage, and no-destructive-cleanup status.
- Linkage for Multi-session Agent Run Coordination v1 through `workspaceLeaseId` metadata.
- Linkage for Multi-user / Multi-session Branch Orchestrator v2 through `workspaceLeaseId` metadata.
- DTO sanitization for local workspace paths and secret-like metadata keys.

## What v2 Does Not Implement

- No real `git worktree add` or `git worktree remove`.
- No production workspace cleanup.
- No branch checkout, switch, merge, rebase, force-push, branch deletion, or remote push.
- No remote Git calls.
- No provider or vendor CLI execution.
- No LLM provider calls.
- No secret/env value exposure.
- No persistent Postgres workspace lifecycle repository.
- No production tenant isolation or production deployment behavior.

## WorkspaceLease Model

`AgentWorkspaceLease` records:

- `id`
- `taskId`
- optional `taskRunId`
- `agentRunId`
- `repoId`
- optional `branchLeaseId`
- `branchName`
- `baseBranch`
- `workspaceKind`
- `workspacePath`
- `status`
- `isolationStatus`
- timestamps and optional expiry
- owner actor/service-account attribution
- sanitized metadata

Workspace kinds:

- `fixture`: explicit fixture/temp path validated under the configured workspace root.
- `git_worktree_future`: future local Git worktree allocation, modeled only.
- `clone_future`: future clone-per-agent allocation, modeled only.
- `remote_workspace_future`: future remote workspace allocation, modeled only.

## Cleanup Decisions

Cleanup is decision-only by default. The service records:

- `keep`
- `cleanup_allowed`
- `cleanup_blocked_dirty`
- `cleanup_blocked_uncommitted`
- `cleanup_blocked_unmerged`
- `cleanup_blocked_policy`
- `future_manual_review`

Cleanup decisions include changed files, uncommitted-change status, merge status, optional policy decision id, reason, and sanitized metadata. `recordCleanupCompleted` does not delete files; it only records metadata after a safe fixture decision.

## Fixture-Only Safety Constraints

- Fixture allocation must validate under the configured workspace root.
- Repository root and filesystem root are rejected unless an explicit future fixture exception exists.
- Shared active workspace paths are rejected or flagged as `shared_forbidden`.
- Non-fixture paths are never deleted.
- Destructive cleanup is disabled in default runtime.
- Full local paths are redacted in workspace lifecycle DTOs.
- Secret-like metadata keys are redacted before API/dashboard output.

## Future Git Worktree Strategy

Future real worktree allocation should require:

- `repoId`
- `baseBranch`
- `branchName`
- workspace root allowlist
- branch lease
- policy allow decision
- no path traversal
- no shared root
- cleanup policy

The future allocator should live behind a Git/workspace boundary and should not run in default runtime/tests. This v2 milestone only models the future allocation kinds and the safety gates needed before implementation.

## Runner Integration

`AgentRunnerService` now ensures each saved run has a `workspaceLeaseId`:

- fixture/local runs can allocate a fixture lease and mark it active when path validation passes;
- mock/default runs create a `git_worktree_future` lease as metadata only;
- branch lease ids from run metadata are copied onto the workspace lease;
- runner audit metadata includes `workspaceLeaseId`.

Actual runner execution behavior remains unchanged. `MockAgentRunner` remains the default, and `LocalAgentRunner` remains disabled by default.

## Git Branch Lease Integration

Git and Conflict Manager branch leases remain separate. Workspace lifecycle v2 accepts a `branchLeaseId` to link a per-agent workspace to the branch lease used for conflict and merge-queue decisions. No Git provider operation is invoked by the workspace lifecycle service.

## Multi-session Coordination Integration

Multi-session Agent Run Coordination v1 accepts `workspaceLeaseId` and `branchLeaseId` on `AgentSession`. When two active sessions share a workspace lease id, the coordination service records a `same_workspace` critical blocker and moves affected sessions to `waiting_on_conflict`. When sessions share a branch name or target the same files/directories, coordination records high or medium overlap warnings. Workspace lifecycle v2 does not enforce these policies itself and does not execute real workspace or Git operations.

## Branch Orchestrator Integration

Multi-user / Multi-session Branch Orchestrator v2 accepts `workspaceLeaseId` on branch orchestration requests and ownership records. When an active branch ownership record already references the same workspace lease id for a different agent run, the orchestrator records `blocked_same_workspace` metadata. Workspace lifecycle v2 does not allocate branches and does not enforce branch ownership; it only exposes the workspace lease id and lifecycle state for the branch orchestrator to reference.

## Edit Intent Graph Integration

Cross-session File Lease / Edit Intent Graph v1 accepts optional `workspaceLeaseId` on `FileLease` and `EditIntent`. It uses that metadata to mark same-workspace edit overlap as `critical` with recommendation `block`. This remains advisory metadata: no real file lock, workspace mutation, cleanup, or Git operation is performed.

## API and Dashboard Surfaces

API endpoints:

- `GET /agents/workspaces`
- `GET /agents/workspaces/:id`
- `GET /agents/workspaces/:id/events`
- `POST /agents/workspaces/request`
- `POST /agents/workspaces/:id/activate`
- `POST /agents/workspaces/:id/freeze`
- `POST /agents/workspaces/:id/ready-for-merge`
- `POST /agents/workspaces/:id/merge-completed`
- `POST /agents/workspaces/:id/cleanup/check`

Mutation endpoints update in-memory lifecycle metadata only. They do not delete workspaces and do not run Git commands.

Dashboard Agent Runner read models now include:

- `workspaceLeases`
- `workspaceEvents`
- `cleanupDecisions`
- `workspaceLifecycle` summary
- `coordinationSessions`, `coordinationGroups`, `sessionOverlaps`, `concurrencyPolicies`, `coordinationAuditEvents`, and `coordinationSummary` from Multi-session Agent Run Coordination v1
- `editIntents`, `fileLeases`, `editIntentGraph`, `editOverlapAssessments`, and `editIntentSummary` from Cross-session File Lease / Edit Intent Graph v1

The panel shows active leases, workspace kind, branch name, branch lease linkage, isolation status, lifecycle state, cleanup decision, merge readiness metadata, and explicit no-destructive-cleanup status.

## Safety Constraints

Default runtime/tests must not:

- run real Git worktree operations outside fixture-only tests;
- delete user workspaces;
- run remote Git;
- expose secrets/env values;
- execute vendor CLI;
- call LLM providers;
- weaken Git, Runner, Local Agent, Auth/RBAC, Policy, SecretRef, Tenant Scope, Dashboard, Observability, or Safety gates.

## Known Limitations

- Workspace lifecycle repositories are in-memory only.
- Cleanup checks do not inspect live Git status; callers pass deterministic dirty/unmerged metadata.
- Future worktree/clone/remote workspace allocation is modeled only.
- Branch lease linkage is metadata-only and does not allocate branches or worktrees.
- Multi-session coordination consumes workspace lease ids as metadata only and does not provide distributed locking or durable workspace reservations yet.
- Branch orchestrator consumes workspace lease ids as metadata only and does not allocate worktrees, create branches, or provide distributed branch ownership locks yet.
- Edit intent graph consumes workspace lease ids as metadata only and does not provide OS file locks or durable edit reservations yet.
- Merge Queue Policy v2 consumes workspace lease status as metadata-only readiness evidence and may hold entries when workspaces are not ready, but it does not allocate, delete, or mutate workspaces.
- Dashboard/readiness surfaces are not production tenant-filtered.
