# Multi-user / Multi-session Branch Orchestrator v2

Status: `v2_implemented`

## Summary

Multi-user / Multi-session Branch Orchestrator v2 adds mock-first branch allocation and ownership metadata for concurrent human users and agent sessions working against the same repository. It assigns deterministic safe branch names, links branch ownership to `BranchLease` and optional `WorkspaceLease` metadata, detects active branch collisions, models base branch drift, and exposes API/dashboard/readiness visibility.

This implementation is metadata-only. It does not create Git branches, check out or switch branches, create worktrees, push, fetch, merge, rebase, force-push, delete branches, call GitHub, execute agents, call LLM providers, run vendor CLIs, read credential caches, or expose secrets/env values.

Durable Collaboration Stores v1 now adds a safe repository/schema boundary for `BranchOrchestrationRequest`, `BranchOrchestrationDecision`, and `BranchOwnershipRecord` metadata. The branch orchestrator remains mock-first and does not treat durable storage as permission to create or mutate Git refs.

## Why Branch-only Separation Is Insufficient

Concurrent sessions need both branch and workspace separation. Unique branches keep review and merge ownership clear, but they do not protect a shared working directory. If LLM A and LLM B share one checkout, either session can still affect the other by switching the local branch or editing files in place. v2 therefore models the intended chain:

```text
User / Session / Task / AgentRun
  -> Repo
  -> Base branch
  -> Unique branch
  -> BranchLease
  -> WorkspaceLease, if available
  -> MergeQueueEntry metadata
```

Branch ownership is coordinated in `packages/git-adapter`; workspace lifecycle remains in `packages/runner`.

## Branch Orchestration Model

`BranchOrchestrationRequest` records who is asking for a branch and why:

- user, actor, task, task run, agent run, and optional session ids;
- repo id and base branch;
- optional requested branch name;
- branch purpose: `agent_work`, `conflict_resolution`, `review_fixup`, `merge_candidate`, or `experiment`;
- target files and source scope;
- sanitized metadata.

`BranchOrchestrationDecision` records the result:

- `allocated`;
- `reused_existing_lease`;
- `blocked_collision`;
- `blocked_policy`;
- `blocked_same_workspace`;
- `warning_base_branch_drift`;
- `future_manual_review`.

Decisions include branch name, optional branch/workspace lease ids, reason, warnings, and sanitized metadata.

## Naming Policy

The default naming policy is:

```text
aichestra/{repoSlug}/{taskId}/{agentRunId}
```

The implementation sanitizes unsafe characters, lowercases generated segments, rejects empty branch names, enforces the `aichestra/` prefix, rejects path traversal-like segments, rejects reserved targets such as `main`, `master`, `develop`, `release`, and `head`, and rejects destructive/reserved segments such as `force`, `delete`, `branch-d`, and `rm`.

The policy is deterministic in tests and does not include timestamps by default.

## Ownership Model

`BranchOwnershipRecord` tracks active ownership:

- branch name, repo, and base branch;
- user, actor, task, task run, agent run, and optional session ids;
- branch lease id;
- optional workspace lease id;
- status: `active`, `frozen`, `ready_for_review`, `ready_for_merge`, `merged`, `abandoned`, or `expired`;
- optional expiry and sanitized metadata.

One active branch name cannot be allocated to a different active session or task/run owner without a collision decision. A matching logical owner may reuse its existing lease metadata.

## Branch Lease Integration

When a safe `branchLeaseCreator` is available and the request has task and task run ids, the orchestrator creates or links a `BranchLease` record as metadata. This is the existing Conflict Manager branch lease model; it remains separate from branch orchestration ownership.

The orchestrator never calls a Git provider to create the branch. `BranchLease` linkage exists so conflict risk, changed-file metadata, merge queue policy, session coordination, and dashboard panels can agree on ownership.

## Workspace Lease Integration

If Agent Workspace Lifecycle v2 is available, the orchestrator can link `workspaceLeaseId` to branch ownership records. Active ownership records sharing a workspace lease id are blocked as `blocked_same_workspace` because a shared checkout can still interfere even with unique branch names.

Agent Worktree Allocation v1 can consume the orchestrator's branch lease and workspace lease metadata to validate a fixture-only or dry-run worktree allocation request. The allocation service owns path/root collision checks; the branch orchestrator continues to own branch naming and branch ownership metadata.

If workspace lifecycle metadata is not available, the field remains optional and future-facing. v2 does not allocate, inspect, delete, or mutate workspaces.

## Multi-session Coordination Integration

Multi-session Agent Run Coordination v1 remains responsible for session grouping and overlap warnings. Branch Orchestrator v2 can consume active session metadata to detect same active branch use and can publish branch ownership metadata for dashboards and future coordination consumers.

The two services are intentionally separate:

- coordination groups sessions, files, scopes, branches, and workspaces;
- branch orchestration allocates branch names and records branch ownership.

Neither service executes agents or mutates Git/workspaces.

## Merge Queue Interaction

The orchestrator records merge queue linkage metadata when a merge queue lookup is available:

- linked queue entry count;
- queue statuses;
- recommendations;
- ready-for-merge entry count.

Merge Queue Policy v2 consumes branch ownership linkage, same-branch blockers, same-workspace blockers, and readiness metadata as evidence for policy decisions and holds. This remains metadata-only and does not change Git adapter execution.

`markReadyForReview` and `markReadyForMerge` update branch ownership metadata only. They do not merge, rebase, push, change a provider PR, or mutate queue execution state.

## Base Branch Drift

`BaseBranchDriftStatus` is modeled from caller-supplied metadata:

- `current`;
- `behind_base`;
- `base_changed`;
- `unknown`;
- `future_check_required`.

Recommendations are metadata only: `continue`, `refresh_needed`, `rebase_future`, `merge_base_future`, or `manual_review`. v2 does not run `git fetch`, compute real merge bases, rebase, or merge.

## API and Dashboard Surfaces

API endpoints:

- `POST /git/branches/orchestrate`;
- `GET /git/branches/orchestration`;
- `GET /git/branches/orchestration/:id`;
- `GET /git/branches/orchestration/summary`;
- `GET /git/branches/orchestration/policies`;
- `GET /git/branches/orchestration/audit`;
- `GET /git/branches/ownership`;
- `GET /git/branches/drift`.

Dashboard `/dashboard/git` now includes:

- branch orchestration requests and decisions;
- active branch ownership records;
- branch naming policy;
- collision and same-workspace blockers;
- workspace linkage;
- base branch drift statuses;
- merge readiness counts;
- branch orchestrator audit events;
- no-destructive-Git, no-remote-Git, no-workspace-mutation, and no-secret status.

Conflict Resolution Assistant v1 consumes branch ownership, branch lease, workspace lease, merge queue, dry-run, conflict risk, and edit-overlap metadata to produce review-only conflict summaries and plans. Branch Orchestrator v2 remains metadata-only and does not execute any assistant recommendation.

## Safety Constraints

Default runtime/tests must not:

- run real remote Git;
- create, switch, merge, rebase, fetch, push, force-push, or delete branches;
- create or delete worktrees;
- mutate user workspaces;
- execute agents;
- call LLM providers or vendor CLIs;
- call external providers;
- read credential caches;
- expose secrets/env values;
- weaken Auth/RBAC, Policy, SecretRef, Tenant Scope, Dashboard, Observability, Runner, Local Agent, Registry, Governance, Staging, CI/CD, or Secrets/Sandbox gates.

Audit and dashboard metadata is sanitized before it is returned.

## Known Limitations

- In-memory repository only.
- No distributed branch ownership lock.
- No real provider branch existence check.
- No real merge-base or drift calculation.
- No real branch creation or workspace allocation.
- No production tenant-filtered branch ownership store.
- No merge execution or provider mutation. Merge Queue Policy v2 consumes branch ownership metadata for readiness decisions and holds, and Conflict Resolution Assistant v1 consumes it for review-only summaries/plans.
- PR Ownership / Handoff Model v1 consumes branch/PR/lease identifiers as local owner/reviewer and handoff metadata, but it does not create remote branches, assign remote reviewers, or update provider PRs.
- Real Merge Execution Policy v1 consumes branch lease/source branch ownership metadata as future precondition evidence, but it still does not create branches, check out branches, push, merge, or delete branches.
- Same-symbol and semantic conflict prediction remain future work.
