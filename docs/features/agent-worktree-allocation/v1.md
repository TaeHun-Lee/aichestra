# Agent Worktree Allocation v1

Status: `v1_implemented`

## Summary

Agent Worktree Allocation v1 adds a mock-first, fixture-only boundary for assigning one isolated working directory path per agent run before future Git worktree execution is considered. It validates workspace roots, requested paths, branch lease linkage, branch name safety, and shared path collisions, then returns reviewable allocation metadata.

The implementation lives in `packages/runner/src/worktree-allocation.ts`, with DTO exports in `packages/runner/src/dto.ts`, API routes under `/agents/worktrees/*`, readiness routes under `/readiness/agent-worktrees/*`, dashboard visibility under `/dashboard/agents`, and deterministic tests in `tests/agent-worktree-allocation-v1.test.ts`.

Durable Collaboration Stores v1 now adds a safe repository/schema boundary for `AgentWorktreeAllocationRequest` and `AgentWorktreeAllocationResult` metadata. The allocation service remains mock-first and fixture-only; durable storage does not enable real `git worktree` commands or workspace mutation.

## What v1 Implements

- `AgentWorktreeAllocationRequest`, `AgentWorktreeAllocationResult`, and `AgentWorktreeSafetyCheck` models.
- `AgentWorktreeAllocationService` with deterministic in-memory request/result/safety-check storage.
- Dry-run allocation that validates root, path, branch lease, branch name, and collision metadata without filesystem mutation.
- Fixture allocation that can link to Agent Workspace Lifecycle v2 fixture workspace lease metadata.
- Production `git_worktree_future` mode blocked as `future_git_worktree_required`.
- Workspace root allowlist metadata using future `AICHESTRA_WORKSPACE_ROOT_ALLOWLIST` and disabled default `AICHESTRA_ENABLE_AGENT_WORKTREE_ALLOCATION=false`.
- API, dashboard, `/health`, and readiness summaries that expose counts and booleans only.
- Secret/env-like metadata redaction and local path redaction in DTOs.

## What v1 Does Not Implement

- No real `git worktree add` or `git worktree remove`.
- No checkout, switch, reset, clean, fetch, push, merge, rebase, force-push, branch deletion, or provider PR mutation.
- No production worktree allocation.
- No destructive cleanup or deletion of real user directories.
- No source file mutation.
- No LLM provider, vendor CLI, hosted Git provider, MCP, Vault, kubectl, or credential-cache access.
- No production tenant isolation enforcement or production auth.

## Why Worktree Allocation Is Required

Branch leases and safe branch naming are necessary but not sufficient. Two sessions can use different branches and still interfere if they share one local checkout. v1 makes the intended isolation chain explicit:

```text
AgentRun
  -> BranchLease
  -> AgentWorkspaceLease
  -> AgentWorktreeAllocationResult
  -> isolated requested path metadata
```

## Request Model

`AgentWorktreeAllocationRequest` records:

- `repoId`, `baseBranch`, and `branchName`;
- `branchLeaseId` and optional `workspaceLeaseId`;
- optional `baseRepoPath`;
- `requestedPath` and `workspaceRoot`, redacted in DTOs;
- `agentRunId`, optional `taskId`, optional `userId`;
- allocation mode: `fixture_only`, `dry_run`, or `git_worktree_future`;
- sanitized metadata and creation timestamp.

## Result Model

`AgentWorktreeAllocationResult` records:

- decision: `allocated_fixture`, `dry_run_valid`, `blocked_root_not_allowed`, `blocked_path_collision`, `blocked_branch_missing`, `blocked_policy`, or `future_git_worktree_required`;
- sanitized worktree path;
- branch, branch lease, and optional workspace lease linkage;
- safe metadata flags including `realGitWorktreeExecuted: false`, `remoteGitOperation: false`, `destructiveCleanupExecuted: false`, `sourceMutation: false`, and `productionWorktreeAllocation: false`.

## Safety Checks

`AgentWorktreeSafetyCheck` uses these check kinds:

- `workspace_root_allowed`
- `path_within_root`
- `path_not_shared`
- `branch_lease_present`
- `branch_name_safe`
- `no_remote_git`
- `cleanup_safe`
- `fixture_only`

Failures block the allocation result. `git_worktree_future` produces a future/blocked decision and never runs Git.

## Workspace Root Allowlist

v1 treats workspace roots as readiness metadata. `AICHESTRA_WORKSPACE_ROOT_ALLOWLIST` can list future allowed roots, and `AICHESTRA_AGENT_WORKSPACE_ROOT` can be reused by local fixture tests. API, dashboard, and health output expose only whether an allowlist exists and its count, never the root values.

Requested paths must resolve within the requested root and must not use traversal. Active or reserved paths are blocked as shared path collisions.

## Workspace Lifecycle Integration

Fixture allocation can call Agent Workspace Lifecycle v2 to create a fixture `AgentWorkspaceLease` metadata record. That lease remains lifecycle metadata only: cleanup decisions do not delete user directories, and the worktree allocation result records destructive cleanup as not executed.

## Branch Orchestrator Integration

v1 validates that a supplied branch lease exists, is active, and matches repo, branch, and base branch metadata when a lookup is available. Multi-user / Multi-session Branch Orchestrator v2 can continue to own branch allocation and branch ownership records while worktree allocation owns path/root safety metadata.

## API And Dashboard Surfaces

API endpoints:

- `POST /agents/worktrees/dry-run`
- `POST /agents/worktrees/fixture-allocate`
- `GET /agents/worktrees/allocations`
- `GET /agents/worktrees/safety-checks`
- `GET /agents/worktrees/summary`

Readiness endpoints:

- `GET /readiness/agent-worktrees/summary`
- `GET /readiness/agent-worktrees/safety-checks`

Dashboard `/dashboard/agents` shows allocation status, enabled false, fixture-only true, allowlist count, path collision warnings, branch/workspace lease linkage, production worktree allocation false, no real Git worktree execution, no destructive cleanup, and no full local paths.

## Safety Constraints

- Default runtime remains safe and mock-first.
- Real worktree allocation is disabled even if future env gates are present.
- Policy deny remains authoritative.
- Auth/RBAC, Tenant Scope, Runner, Local Agent, Branch Lease, Workspace Lifecycle, Dashboard, Observability, Registry, Governance, and Safety gates are not bypassed.
- No secrets, env values, tokens, credential cache paths, or sensitive absolute paths are exposed.

## Known Limitations

- In-memory request/result/safety-check storage only.
- No durable distributed path reservation.
- No real worktree creation or cleanup.
- No live Git integration-test profile.
- No production tenant-scoped allocation ownership.
- Real Merge Execution Policy v1 consumes worktree allocation metadata as optional future precondition evidence, but production `git worktree` allocation and deletion remain disabled.
