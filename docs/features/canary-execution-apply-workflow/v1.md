# Canary Execution Harness + Apply Workflow v1

Status: v1_implemented (canary execution harness), v1_implemented_metadata_only (apply workflow)

Canary Execution Harness + Apply Workflow v1 is a mock-first, deterministic, advisory layer that lets governance reason about whether a draft registry change is safe to apply. It executes only deterministic mock canaries, never calls external providers, never runs real evals or canaries, never mutates active Skill/Harness/Instruction/RegistryPackage records, never auto-applies, and never weakens Registry, Governance, Policy, Auth/RBAC, Tenant Scope, SecretRef, Dashboard, Observability, Staging, CI/CD, or Safety gates.

## What v1 Implements

### Models (`packages/registry/src/canary-apply.ts`)

- `RegistryCanaryPlan` / `RegistryCanaryRun` / `RegistryCanaryResult` / `RegistryCanaryVerdict` capturing target, canary kind, eval/approval prerequisites, run lifecycle, deterministic metric outputs (`pass_rate`, `failure_rate`, `token_cost`, `runtime`, `policy_denial_rate`, `rollback_signal`, `user_feedback_future`), and overall verdict (`passed | failed | warning | skipped | blocked`).
- `RegistryApplyWorkflow` / `RegistryApplyGateDecision` / `RegistryRollbackPlan` linking proposal + draft change with status (`draft | eval_required | canary_required | approval_required | ready_for_manual_apply_future | blocked | applied_metadata_only | applied_future`), apply mode (`metadata_only | manual_future | automatic_forbidden`), and rollback metadata.
- `RegistryCanaryApplySummary` exposing counts and the literal-`false` safety constants `autoApplyEnabled`, `activeRegistryMutationAllowed`, `applyPerformed`, `activeRegistryMutated`, `externalCanaryExecuted`, `realEvalExecuted`, `realProviderCallExecuted`.

### Services

- `CanaryExecutionService` with `createCanaryPlan`, `requestCanaryRun`, `executeMockCanaryRun`, `getCanaryRun`, `listCanaryRuns`, `listCanaryResults`, `listCanaryVerdicts`, `getCanaryVerdictForRun`, `getRunsMatchingApprovedVerdicts`. Mock-deterministic only.
- `ApplyWorkflowService` with `createApplyWorkflow`, `createRollbackPlan`, `evaluateApplyGate`, `recordMetadataOnlyApplyDecision`, `listWorkflows`, `getWorkflow`, `listGateDecisions`, `listRollbackPlans`. Metadata only.
- `createCanaryAndApplyServices(input)` returns both services wired together plus a deterministic `getSummary()` accessor used by `/health`, `/readiness`, and the dashboard read model.

### Policy

- New `PolicyResourceKind` values `registry_canary` and `registry_apply_workflow`.
- Eight new `PolicyAction` values: `registry.canary.plan`, `registry.canary.run_mock`, `registry.canary.run_external_future`, `registry.apply_workflow.create`, `registry.apply_gate.evaluate`, `registry.apply.metadata_record`, `registry.apply.execute_future`, `registry.apply.auto_apply_future`.
- Default rules in `packages/policy/src/default-rules.ts`:
  - `registry.canary.plan` / `registry.canary.run_mock` allowed in metadata-only mock context (developer+).
  - `registry.apply_workflow.create` / `registry.apply_gate.evaluate` allowed in metadata-only mock context (reviewer+).
  - `registry.apply.metadata_record` allowed in metadata-only mock context (security_admin+).
  - `registry.canary.run_external_future`, `registry.apply.execute_future`, `registry.apply.auto_apply_future` denied by default.

## What v1 Does Not Implement

- No real canary execution. `task_subset_future`, `tenant_subset_future`, `provider_subset_future`, `external_future` plans always block.
- No real apply execution. `registry.apply.execute_future` deny-by-default.
- No auto-apply. `registry.apply.auto_apply_future` deny-by-default.
- No active Skill/Harness/Instruction/RegistryPackage mutation through auto-improvement.
- No real eval. Eval requirements are forward-compatible references (passing eval runs from Eval Suite Execution Harness v1 are accepted, but failed/missing block the gate).
- No real rollback. `registry_history_revert_future` and `package_version_revert_future` remain future-only.
- No external provider / MCP / LLM / vendor CLI / remote Git / credential-cache calls.
- No production tenant enforcement. Tenant scope mismatches remain partial enforcement metadata.

## Apply Workflow State Diagram

```
draft
  └─ eval_required ──┐
       │             │
       └─ canary_required ──┐
            │                │
            └─ approval_required ──┐
                 │                  │
                 └─ ready_for_manual_apply_future ──┐
                      │                              │
                      └─ applied_metadata_only ───┐
                           (recordMetadataOnlyApplyDecision)
       │  failed_eval / failed_canary / missing rollback / policy_denied / automatic_forbidden → blocked
       │  applied_future = future only, never reachable in v1
```

## Manual Approval Requirements

`evaluateApplyGate` requires:

1. Policy allow on `registry.apply_gate.evaluate`.
2. `applyMode !== "automatic_forbidden"`.
3. All `requiredEvalRunIds` present and `status === "pass"`.
4. All `requiredCanaryRunIds` present and their verdict is not `failed` / `blocked`.
5. All `requiredApprovalIds` present and `status === "approved"` (or non-empty).
6. A `RegistryRollbackPlan` with `status === "planned"` and `rollbackKind ∈ {metadata_only, manual_review}`.

Only when all six conditions hold does the workflow advance to `ready_for_manual_apply_future`. `recordMetadataOnlyApplyDecision` then records an audit-only `metadata_only_recorded` decision and moves the workflow to `applied_metadata_only`. Even at that point `applyPerformed` stays `false`, `activeRegistryMutated` stays `false`, and the registry resolver gates remain authoritative.

## Rollback Requirements

`RegistryRollbackPlan` v1 supports `rollbackKind ∈ {metadata_only, manual_review}` in active status. `registry_history_revert_future` and `package_version_revert_future` are reserved for future work and currently block. `requiredEvidence` always includes `draft_change_reference`, `governance_approval_reference`, and `registry_history_checkpoint`.

## Integration With Eval Suite Execution Harness v1

The API construction site uses `EvalSuiteExecutionService.listEvalRuns()` as the canary data source. A run with status `passed` translates to `pass`; everything else maps to `missing` (so failed/incomplete eval prerequisites block the canary). No new eval execution is triggered from the canary or apply workflow; references only.

## Integration With Governance / Auto-improvement

- The apply workflow references existing `ProposalReadiness`, `ProposalApplyGate`, and `DraftRegistryChange` IDs without mutating them.
- `recordMetadataOnlyApplyDecision` produces a metadata-only audit row; it never invokes the active registry mutation services.
- Auto-improvement remains draft/proposal-only — `improvement.apply` policy is unchanged; the new `registry.apply.auto_apply_future` action is denied by default to make the future apply-execution policy explicit.

## Policy / Auth Behavior

| Action | Default rule |
| --- | --- |
| `registry.canary.plan` | allow (developer+, mock metadata-only env) |
| `registry.canary.run_mock` | allow (developer+, mock metadata-only env) |
| `registry.canary.run_external_future` | deny |
| `registry.apply_workflow.create` | allow (reviewer+, mock metadata-only env) |
| `registry.apply_gate.evaluate` | allow (reviewer+, mock metadata-only env) |
| `registry.apply.metadata_record` | allow (security_admin+, mock metadata-only env) |
| `registry.apply.execute_future` | deny |
| `registry.apply.auto_apply_future` | deny |

Policy deny short-circuits the services: a denied plan creation produces `status: "blocked"`, a denied run produces `status: "blocked_policy"`, a denied execute produces a `blocked` verdict, a denied gate or metadata record produces `blocked_policy_denied`.

## No-Auto-Apply and No-Active-Mutation Guarantees

- `RegistryApplyWorkflow.autoApplyEnabled` and `.activeRegistryMutationAllowed` are the literal `false` type — changing them is a compile-time error.
- `RegistryApplyGateDecision.applyPerformed` and `.activeRegistryMutated` are similarly typed `false`.
- `RegistryCanaryApplySummary.applyPerformed`, `.activeRegistryMutated`, `.externalCanaryExecuted`, `.realEvalExecuted`, `.realProviderCallExecuted` are all literal-`false`.
- The default policy denies `registry.apply.execute_future` and `registry.apply.auto_apply_future`.
- The dashboard panel and `/health.registryCanaryApply` payload surface these flags so reviewers can see the safety posture at a glance.

## Known Limitations

- Mock metric thresholds (`DEFAULT_THRESHOLDS`) are deterministic seeds, not tuned against production traffic.
- Approvals are reference-only; the service does not generate or own governance approval state.
- Eval references only consider `passed` runs as `pass`; partial/warning runs are treated as missing in v1.
- Rollback plans are metadata-only; future versions must add real version revert pathways behind explicit governance.
- No production tenant enforcement; scope mismatches stay metadata.
