1. 요약 & 위험 5선

설계와 구현의 큰 골격(도메인 모델, 상태 머신 정의, 어댑터 경계, 정책/거버넌스 모델, mock-only 경계)은 잘 일치합니다. "real integration"으로 표시된 영역들 모두 mock-only 경계를 깨지 않으며, GitHub/네트워크/실제 시크릿 호출은 0건으로 확인됩니다. Phase 3 패키징/버저닝과 Phase 4 거버넌스/auto-apply 게이트도 enum/액션 단위까지 거의 매핑됩니다.

그러나 워크플로우/리졸버/권한 통합 지점에 치명적 누락이 있습니다. "배치는 갖춰져 있는데 결선이 빠진" 패턴이 반복됩니다 — 권한기 클래스는 import만 되어 있고 호출되지 않으며, 리졸버 게이트는 한쪽 경로에만 적용되어 있고, 상태 머신 가드는 Task에만 있고 TaskRun/MergeQueueEntry에는 없습니다.

가장 위험한 5가지

  1. Critical Instruction 리졸버의 게이트 누락 — 거부/평가실패/체크섬 불일치 instruction이 워크플로우에 주입될 수 있음. packages/registry/...selectableRegistryEntry는 4개 게이트(active/approval/eval/checksum)를 모두 적용하지만, 워커가 실제 사용하는 packages/core/src/instructions/resolver.ts:21–31는 status·agent만 검사.
  2. Critical Registry 변이 라우트에 권한기 미연결 — PolicyBackedRegistryMutationAuthorizer가 apps/api/src/main.ts:109에 import만 되고 어디에도 instantiate/적용되지 않음. Approval/rollback/deprecate 모두 무인증 + 권한 체크 없이 실행, actorId는 body 또는 "mock-admin" 디폴트.
  3. Critical 정책 차단된 재실행 경로의 잘못된 상태 전이로 런타임 throw — apps/worker/src/workflows/run-agent-task-workflow.ts:131–135에서 completed/failed 태스크 재실행 시 queued → planned 시도(전이 표에 없음) → Invalid task status transition 예외.
  4. High POST /merge-simulations에 임의 파일시스템 경로 허용 — repoPath가 LocalGitDryRunMergeSimulator로 직접 흐르며, allowlist 없음. 네트워크 노출 시 호스트의 임의 git 워크트리에서 git diff/git merge-tree 실행 가능.
  5. High Skill/Harness/Instruction (name, version) 유니크 부재 + active+approved 버전 불변성 부재 — Prisma @@unique 없음, in-memory 가드 없음. RegistryService.updateSkill은 active+approved 엔트리의 핵심 필드를 직접 수정 가능 → 버전 핀의 결정성과 변조 방지가 동시에 깨짐.
치명도 정리: 즉시 장애를 일으키는 코드 결함은 위 #3 한 건(재실행 시 throw). 나머지(#1, #2, #4, #5)는 "현재 데모 환경에서는 동작하지만 운영 노출 시 즉시 보안/정합성 사고로 직결되는 구조적 결함"입니다.

잘된 점 (참고)

  • mock-only 경계는 모든 패키지에서 깨끗하게 유지됨 (packages/adapters/src/git/에 fetch/Octokit/https:// 0건, packages/security/src/에 process.env/fs.readFile/spawn 0건).
  • Phase 4 정책 액션 enum 55개 전부 코드에 존재, 기본 fail-closed 정책 정상 동작 (packages/policy/src/engine.ts:99–112).
  • Local Agent Protocol v0는 MockLocalAgentTransport만 존재 (소켓/HTTP 0건).
  • Conflict Manager v0 점수표 7개 카테고리·임계치 일치, v1 시뮬레이터는 mode: "local_git_merge_tree" 명시 요청에서만 활성화.

4. 이슈 상세 (24건)

#1 — Instruction 리졸버의 approval / eval / checksum 게이트 누락

Critical 분류: 잘못된 구현 / 보안 · 확신도: 높음
관련 설계
docs/foundations/instruction-layer.md:8–16 — 활성·승인 완료(또는 not_required)·평가 통과(또는 not_required)·체크섬 일치, 4개 필터 동시 적용.
현재 구현
packages/registry/src/index.ts의 selectableRegistryEntry는 4게이트 적용. 그러나 워커가 실제 호출하는 packages/core/src/instructions/resolver.ts:21–31의 resolveInstructionArtifacts는 status === "active" + appliesToAgents.includes(agent)만 검사. apps/worker/src/workflows/run-agent-task-workflow.ts:177–184에서 직접 호출.
이격
코어 리졸버에 approvalStatus !== "rejected" / evalStatus !== "failed" / checksumStatus !== "mismatch" 체크 없음.
영향
거부/평가실패/위변조된 instruction이 에이전트 실행에 주입될 수 있음 → 거버넌스·체크섬 모델 무효화.
재현/확인
{ status: "active", approvalStatus: "rejected" } 시드 추가 → assembleInstructionSet([artifact], "backend") 호출 → 결과에 포함됨을 확인.
권장 수정
코어 리졸버에 4게이트 추가, 또는 입력 타입을 "이미 게이트 통과한 ref"로 좁힘.
추가 테스트
D 표 1 / 2 / 3 (rejected, failed-eval, checksum mismatch 제외 검증).

#2 — Registry 변이 라우트에 권한기 미연결

Critical 분류: 보안 / 미구현 · 확신도: 높음
관련 설계
Phase 3 v2 — MockRegistryMutationAuthorizer(5 role × 10 permission), PolicyBackedRegistryMutationAuthorizer로 감싼 mutation boundary.
현재 구현
apps/api/src/main.ts:109에 import만, instantiate 없음. apps/worker/src/workflows/run-agent-task-workflow.ts:63–75의 createRegistryService 호출도 authorizer 미공급. 모든 mutation 라우트가 registryService를 직접 호출, actor를 body 또는 defaultActor("mock-admin")로 채움.
이격
권한 체크 정의는 있지만 결선 없음. mock-admin은 모든 권한 보유.
영향
누구든 PATCH /registry/skills/:id/approval로 미승인 skill 승인 가능. 감사 로그 actor 위조 가능. Phase 4 거버넌스 결정도 동일 문제.
권장 수정
API 부트스트랩에서 authorizer instantiate → createRegistryService에 전달. 단기적으로 actor를 body 대신 trusted 헤더에서 추출.
추가 테스트
D 표 7 / 8.

#3 — 재실행 정책 차단 경로의 잘못된 상태 전이 (런타임 throw)

Critical 분류: 잘못된 구현 / 런타임 오류 · 확신도: 높음
관련 설계
docs/foundations/task-state-machine.md — completed → queued/failed → queued 허용. queued → branch_created | running | failed | cancelled. planned → policy_blocked만 차단 진입.
현재 구현
apps/worker/src/workflows/run-agent-task-workflow.ts:131–135 — 재실행 시 queued로 보낸 뒤 정책 차단되면 planned 시도.
이격
queued의 허용 외부 전이에 planned 없음 → Error: Invalid task status transition: queued -> planned throw.
영향
completed/failed 태스크를 budgetLimitUsd: 2000로 재실행하면 워커가 예외로 실패, 사용자에게는 500.
재현/확인
1) 태스크 정상 완료 → completed. 2) 같은 태스크에 budgetLimitUsd: 2000로 POST /tasks/:id/run. 3) throw 확인.
권장 수정
정책 평가를 queued 이전에 수행, 차단 시 completed → planned → policy_blocked 또는 policy_blocked 직행 전이 추가.
추가 테스트
D 표 4.

#4 — API 인증 부재 + owner 필터 부재 → 데이터 노출/위조

Critical 분류: 보안 · 확신도: 높음
관련 설계
docs/foundations/auth-rbac-readiness.md — 인증이 의도적으로 미구현(readiness만). 그러나 requesterUserId는 도메인 객체 전체에 존재하고 감사 로그가 사용.
현재 구현
apps/api/src/main.ts에 인증 미들웨어 없음. POST /tasks는 requesterUserId를 body에서 받고 미지정 시 packages/core/src/schemas/domain.ts:70에서 "user_demo_admin"로 폴백. GET /tasks//usage//branch-leases 등 모든 리스트 라우트가 owner 필터 없이 전체 반환.
영향
다른 사용자 태스크/PR URL/사용 비용 익명 조회 가능. 감사 로그 actor 위조 가능.
권장 수정
단기 — X-Aichestra-Actor 트러스트 헤더 강제 + 요청 컨텍스트 actor 주입. 중기 — JWT 미들웨어. 모든 리스트 쿼리에 requesterUserId 필터 추가.
추가 테스트
D 표 7 / 8 / 9.

#5 — POST /merge-simulations repoPath 임의 경로 허용

High 분류: 보안 (path traversal / 정보 노출) · 확신도: 높음
관련 설계
Conflict Manager v1 — "local-only, no fetch/push" 명시되어 있으나 어떤 repo가 허용되는지 allowlist 정의 누락.
현재 구현
apps/api/src/main.ts:2024에서 body의 repoPath를 simulator로 직접 전달. packages/adapters/src/git/merge-simulators.ts:119–141은 git -C <repoPath> rev-parse --is-inside-work-tree로 git 워크트리 여부만 확인. isUnsafePath는 path === path.root만 차단.
영향
호스트의 임의 git 워크트리 대상으로 git diff/git merge-tree 실행 → 파일명/diff 노출.
권장 수정
AICHESTRA_ALLOWED_REPO_PATHS 환경변수 + 절대경로 prefix 매칭, 또는 등록된 Repo 엔티티의 path만 허용.
추가 테스트
D 표 10.

#6 — Skill/Harness/Instruction (name, version) 유니크 부재 + 발행본 불변성 부재

High 분류: 도메인 불변식 / 데이터 정합성 · 확신도: 높음
관련 설계
정확한 버전 핀(v0). v3 패키징은 replace_draft_only 모드로 부분 표현.
현재 구현
packages/db/prisma/schema.prisma의 세 모델에 @@unique([name, version]) 없음. RegistryService.createSkill(packages/registry/src/index.ts:2117–2133)이 중복 검사 없이 push. updateSkill은 active+approved 엔트리의 invocationRules/payload 직접 수정 가능.
영향
같은 name@version 다중 존재 시 highestCompatible 비결정. 발행본 in-place 수정 시 버전 핀 의미 상실.
권장 수정
Prisma @@unique([name, version]) + InMemory 중복 거부. updateSkill 등이 active+approved 엔트리에서 변경 가능 필드 명시 제한(owner, tags 등).
추가 테스트
D 표 5 / 6.

#10 — instruction_set_hash가 LLM usage 메타에 누락

Medium 분류: 설계 불일치 / 원인 추적 · 확신도: 높음
