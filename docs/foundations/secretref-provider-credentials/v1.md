# SecretRef-backed Provider Credentials v1

## What v1 Implements

SecretRef-backed Provider Credentials v1 adds a unified credential resolution boundary for controlled provider credentials used by Real Git Adapter v2, LLM Gateway v1/v2, Enterprise Provider Abstraction metadata, GitHub webhook secret verification, and future MCP server credentials.

The default runtime remains safe and mock-first. v1 does not add Vault, cloud secret managers, BYOK, OAuth/device-code/WIF/IAM, credential cache reads, vendor CLI calls, runner secret injection, or Local Agent secret forwarding.

## SecretRef Model

`SecretRef` remains metadata-only. v1 extends it with provider credential fields:

- `provider`: `mock`, `env`, or future cloud-secret placeholders.
- `secretKind`: `github_token`, `github_webhook_secret`, `llm_api_key`, `provider_api_key`, `webhook_secret`, or future token/identity placeholders.
- `envKey`: an optional environment variable reference for the explicit env provider.
- `status`: `active`, `disabled`, or `revoked`.

`SecretRef` never contains raw secret values. API create/update paths reject raw token fields, credential cache paths, and raw secret-like metadata.

## EnvSecretProvider

`EnvSecretProvider` is disabled by default and only reads a single requested env key referenced by an active `SecretRef`.

Required config:

```bash
AICHESTRA_ENABLE_ENV_SECRET_PROVIDER=true
AICHESTRA_ALLOWED_SECRET_ENV_KEYS=AICHESTRA_GITHUB_TOKEN,AICHESTRA_LLM_API_KEY
```

If `AICHESTRA_ALLOWED_SECRET_ENV_KEYS` is configured, the `SecretRef.envKey` must be in the allowlist. The provider never enumerates environment variables and never returns values through API, health, dashboard, or audit DTOs.

## Credential Flow

The v1 resolution path is:

```text
Git / LLM / Provider metadata
  -> AuthorizationService / AuthContext
  -> purpose-specific PolicyEngine check
  -> SecurityControlService credential manager
  -> active SecretRef
  -> provider.credential.resolve policy decision
  -> metadata SecretLease request/issue policy decisions
  -> EnvSecretProvider single-key lookup
  -> transient internal value at adapter boundary only
  -> sanitized audit/redaction
```

Public `CredentialResolutionResult` DTOs include status, blocked reason, policy/audit ids, and a metadata-only `CredentialHandle`. They never include raw values. `resolveCredentialForInternalUse` is the only internal method that can carry a transient value to the GitHub/OpenAI-compatible adapter boundary.

When `AuthorizationService` is available, credential resolution uses the request `AuthContext` or an explicit mock system context for internal service operations. Authorization denial and policy denial are audited before any env variable is read.

## GitHub Credential Flow

Preferred v1 config:

```bash
AICHESTRA_GITHUB_TOKEN_SECRET_REF=secretref_github_token
AICHESTRA_ENABLE_ENV_SECRET_PROVIDER=true
AICHESTRA_ALLOWED_SECRET_ENV_KEYS=AICHESTRA_GITHUB_TOKEN
```

Remote Git operations still require all Real Git Adapter v2/v1 gates: GitHub provider selection, remote Git enabled, operation-specific branch/PR/read gates, repo allowlist, branch prefix, Auth/RBAC allow, policy allow, and credential resolution allow. `AICHESTRA_GITHUB_TOKEN` remains a legacy env fallback only when no GitHub token SecretRef is configured.

GitHub tokens are never stored in repos, pull request records, health, dashboard, API responses, or audit metadata.

## GitHub Webhook Secret Flow

Preferred v1 config:

```bash
AICHESTRA_GITHUB_WEBHOOK_SECRET_REF=secretref_github_webhook_secret
AICHESTRA_ENABLE_ENV_SECRET_PROVIDER=true
AICHESTRA_ALLOWED_SECRET_ENV_KEYS=AICHESTRA_GITHUB_WEBHOOK_SECRET
```

Real Git Adapter v2 may resolve webhook secrets through `SecurityControlService` before constructing the HMAC verifier. Webhook handling still requires `AICHESTRA_ENABLE_GITHUB_WEBHOOKS=true`, an allowed repo, signature verification, Auth/RBAC or system context, and policy allow. The webhook secret value is never stored, returned, or written to audit.

## LLM Credential Flow

Preferred v1 config:

```bash
AICHESTRA_LLM_API_KEY_SECRET_REF=secretref_llm_api_key
AICHESTRA_ENABLE_ENV_SECRET_PROVIDER=true
AICHESTRA_ALLOWED_SECRET_ENV_KEYS=AICHESTRA_LLM_API_KEY
```

Remote LLM completion still requires all LLM Gateway v1 gates: OpenAI-compatible provider selection, remote LLM enabled, completion enabled, base URL configured, model allowlist/default model, virtual key allowance, budget allow, policy allow, and credential resolution allow. LLM Gateway v2 adds route selection and fallback policy checks before invoking the same credential boundary for remote routes. `AICHESTRA_LLM_API_KEY` remains a legacy env fallback only when no LLM API key SecretRef is configured.

API keys are never stored in model catalog records, virtual keys, usage metadata, health, dashboard, API responses, or audit metadata.

## Enterprise Provider Abstraction

`ProviderAuth` API-key entries can reference `secretRef`. Local CLI providers remain `external_cli_session` with `credentialAccess=never_read_tokens`; they do not resolve provider API keys, read credential caches, or receive forwarded secrets.

## MCP Gateway v0

MCP Gateway v0 can model server/tool `requiredSecretRefs`, but tools do not receive SecretLeases or raw secret values. Secret-requiring MCP tools are blocked by default and audited as secret-denied metadata-only outcomes.

OAuth, device-code, workload identity, cloud IAM, and BYOK remain future placeholders.

## Legacy Env Fallback Compatibility

SecretRef takes precedence over legacy env credentials. If a `*_SECRET_REF` is configured but is missing, disabled, revoked, policy-denied, authorization-denied, or otherwise unresolved, the adapter reports the SecretRef failure and does not silently fall back to `AICHESTRA_GITHUB_TOKEN`, `AICHESTRA_GITHUB_WEBHOOK_SECRET`, or `AICHESTRA_LLM_API_KEY`.

When the API runtime intentionally uses a legacy env credential because no corresponding SecretRef is configured, it records a metadata-only `credential_legacy_env_fallback_used` security audit event. The event includes provider id, purpose, env key name, source, and gate metadata, but never the env value.

## Policy Integration

v1 uses Policy-as-code for:

- `provider.credential.resolve`
- `git.credential.resolve`
- `llm.credential.resolve`
- `secret.lease.request`
- `secret.lease.issue`
- `secret.read`
- `git.remote_operation`
- `llm.remote_completion`
- `provider.invoke`

`secret.read`, credential cache read/upload, runner secret injection, and Local Agent secret forwarding remain denied. Existing Git/LLM config gates remain authoritative even when credential policy allows resolution.

## Audit and Redaction

Security audit events include credential ref lifecycle, credential resolution, env provider disabled/non-allowlisted/missing outcomes, legacy env fallback usage, and value redaction events.
Credential resolution now distinguishes authorization-denied and policy-denied outcomes in audit metadata while keeping raw values out of the audit stream.

Audit must not include raw secrets, tokens, API keys, bearer tokens, env var values, raw prompts, or credential cache contents. Redaction covers GitHub token-like strings, provider key-like strings, bearer tokens, env dumps, `~/.codex/auth.json`, `~/.claude`, and Google credential cache references.

## API, Health, and Dashboard

Credential endpoints:

- `GET /security/credentials/refs`
- `POST /security/credentials/refs`
- `PATCH /security/credentials/refs/:id/status`
- `POST /security/credentials/resolve/check`
- `GET /security/credentials/audit`

Health and dashboard report credential manager kind, env secret provider enabled state, active SecretRef count, GitHub token, GitHub webhook secret, and LLM credential configured status, plus recent credential audit events. They expose no secret values or env var values.

## Test Strategy

`tests/secretref-provider-credentials-v1.test.ts` covers SecretRef validation, EnvSecretProvider gates, AuthorizationService and PolicyEngine before env reads, GitHub token/webhook and LLM SecretRef resolution, Enterprise Provider API-key SecretRef metadata, API/health/dashboard no-secret responses, and regression safety.

Existing Real Git Adapter v1/v2 and LLM Gateway v1/v2 tests continue to cover operation gates, routing decisions, and mocked provider boundaries.

## Known Limitations

- Not production-ready secret management.
- No Vault/cloud secret manager integration.
- No BYOK.
- No OAuth/device-code/WIF/IAM token exchange.
- No credential cache reads or uploads.
- No runner or Local Agent secret injection.
- Credential repositories remain in-memory by default.
- Legacy direct env fallback remains supported for compatibility when no `*_SECRET_REF` is configured.
