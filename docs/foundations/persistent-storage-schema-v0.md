# Persistent Storage Schema v0

## Scope

This is a Postgres-oriented schema design. Real Integration Foundation v0 introduced it as a plan; Persistent DB v1 now wires the core durable slice to an opt-in Postgres storage provider.

Conventions:

- Primary keys are text ids generated by application code.
- Timestamps use `timestamptz`.
- Domain enums remain text columns for v0 migration flexibility.
- Complex snapshots, refs, lists, and metadata use `jsonb`.
- Audit and revision tables are append-only.

## Core Tables

### `repos`

Purpose: durable repository metadata used by tasks, branch leases, and future Git adapters.

- Primary key: `id`
- Important columns: `provider`, `owner`, `name`, `default_branch`, `remote_url`, `status`, `created_at`, `updated_at`
- Indexes: `provider,status`
- Retention: project lifetime
- Real Git Adapter v1 notes: GitHub remote operations use `provider=github`, `owner`, `name`, and `default_branch` with environment allowlists. Tokens are not stored in this table.

### `tasks`

Purpose: durable task state.

- Primary key: `id`
- Important columns: `title`, `description`, `status`, `requester_user_id`, `repo_id`, `base_branch`, `branch_name`, `selected_agent`, `selected_model`, `selected_skill_ids jsonb`, `selected_harness_id`, `instruction_set_id`, `budget_limit_usd`, `conflict_risk_score`, `created_at`, `updated_at`
- Foreign keys: `repo_id -> repos.id`, `requester_user_id -> users.id` when user persistence is enabled
- Indexes: `repo_id`, `status`, `requester_user_id`, `created_at`
- Retention: project lifetime
- Migration notes: add partial index for active task statuses later

### `task_runs`

Purpose: durable execution attempts.

- Primary key: `id`
- Important columns: `task_id`, `attempt`, `status`, `agent`, `model`, `model_provider`, selected skill/harness ids and versions, registry refs as `jsonb`, `instruction_set_id`, timing, result, changed files, PR url, error
- Foreign keys: `task_id -> tasks.id`
- Indexes: `(task_id, attempt) unique`, `task_id`, `status`, `created_at`
- Retention: project lifetime
- Migration notes: add explicit TaskRun transition history later

### `usage_ledger_entries`

Purpose: durable LLM/runtime usage and cost attribution.

- Primary key: `id`
- Important columns: `task_id`, `task_run_id`, `user_id`, `repo_id`, `provider`, `model`, `event_type`, token counts, `cost_usd`, `latency_ms`, `skill_version`, `harness_version`, `metadata jsonb`, `created_at`
- Foreign keys: `task_id -> tasks.id`, `task_run_id -> task_runs.id`
- Indexes: `task_id`, `task_run_id`, `user_id`, `provider`, `created_at`
- Retention: billing and audit retention
- Migration notes: never drop task/run attribution

## LLM Gateway Tables

### `llm_models`

Purpose: durable model catalog for future real provider integrations.

- Primary key: `id`
- Important columns: `provider_kind`, `display_name`, `context_window`, `supports_tools`, `supports_streaming`, `input_token_cost_usd`, `output_token_cost_usd`, `status`, `metadata jsonb`, `created_at`, `updated_at`
- Indexes: `provider_kind,status`
- Retention: model catalog lifecycle
- Migration notes: v1 runtime still uses an in-memory catalog by default. `openai-compatible/default` is an Aichestra model id and can map to a provider-side model through gated config; provider API keys are not stored here.

### `virtual_model_keys`

Purpose: internal policy keys for model routing and budgets. These are not provider API keys.

- Primary key: `id`
- Important columns: `owner_kind`, `owner_id`, `display_name`, `allowed_provider_kinds jsonb`, `allowed_model_ids jsonb`, `monthly_budget_usd`, `per_task_budget_usd`, `rpm_limit`, `tpm_limit`, `status`, `created_at`, `updated_at`
- Indexes: `owner_kind,owner_id`, `status`
- Retention: policy lifecycle
- Migration notes: never store OpenAI, Anthropic, Gemini, Bedrock, or other provider secrets here.

### `llm_audit_events`

Purpose: append-only audit events for LLM routing, blocked calls, allowed calls, and provider validation.

- Primary key: `id`
- Important columns: `event_type`, `task_id`, `task_run_id`, `actor_id`, `provider_kind`, `model_id`, `result`, `reason`, `metadata jsonb`, `created_at`
- Indexes: `task_id`, `task_run_id`, `provider_kind,model_id`, `created_at`
- Retention: operational/compliance retention
- Migration notes: metadata must be sanitized and must not include provider API keys, bearer tokens, raw credentials, raw prompts, unredacted output, or credential cache contents. LLM Gateway v1 remote events use this table shape in future persistence.

### `llm_gateway_requests`

Purpose: sanitized request/result trace records for future real LLM calls.

- Primary key: `id`
- Important columns: `task_id`, `task_run_id`, `provider_kind`, `model_id`, `status`, `input_tokens`, `output_tokens`, `estimated_cost_usd`, `finish_reason`, `metadata jsonb`, `created_at`
- Indexes: `task_id`, `task_run_id`, `provider_kind,model_id`, `created_at`
- Retention: debug/audit retention
- Migration notes: v1 usage ledger remains the source of cost attribution. Request traces must contain sanitized metadata only; raw prompts and provider secrets are not stored.

### `provider_catalog_entries`

Purpose: future durable provider catalog for Enterprise LLM Provider Abstraction.

- Primary key: `id`
- Important columns: `display_name`, `vendor`, `kind`, `auth jsonb`, `supported_models jsonb`, `billing_mode`, `capabilities jsonb`, `default_enabled`, `status`, `policy_notes jsonb`, `metadata jsonb`, `created_at`, `updated_at`
- Indexes: `kind,status`
- Retention: provider lifecycle
- Migration notes: `auth` stores references/config only; it must not store raw provider tokens, OAuth tokens, keychain paths, or credential cache contents.

### `provider_audit_events`

Purpose: append-only provider validation, credential, invocation, parser, Local Agent, and redaction audit records.

- Primary key: `id`
- Important columns: `event_type`, `actor_id`, `task_id`, `task_run_id`, `provider_id`, `provider_kind`, `auth_type`, `operation`, `result`, `reason`, `metadata jsonb`, `created_at`
- Indexes: `provider_id,created_at`, `task_id,task_run_id`
- Retention: compliance/audit retention
- Migration notes: metadata must be sanitized and must not include raw prompts, provider secrets, bearer tokens, or credential cache content.

These tables are schema skeletons only in Local Agent Protocol v1. Runtime repositories remain in-memory and no real Local Agent daemon, network transport, PTY automation, vendor CLI execution, or credential-cache access is wired.

- Indexes: `user_id,status`, `host_id`, `last_seen_at`
- Retention: local agent lifecycle and connection history
- Migration notes: must not include vendor tokens, raw secrets, or credential cache content.

### `local_agent_sessions`

Purpose: future durable session metadata for connected Local Agents.

- Primary key: `id`
- Important columns: `agent_id`, `user_id`, `status`, `issued_at`, `expires_at`, `revoked_at`, `metadata jsonb`
- Indexes: `agent_id,status`, `user_id,status`, `expires_at`
- Retention: session lifecycle plus audit retention
- Migration notes: session rows store metadata only, not bearer tokens or provider credentials.

### `local_agent_channels`

Purpose: future durable mock/future channel lifecycle metadata.

- Primary key: `id`
- Important columns: `agent_id`, `user_id`, `channel_kind`, `status`, `handshake_status`, `created_at`, `established_at`, `expires_at`, `revoked_at`, `metadata jsonb`
- Indexes: `agent_id,status`, `user_id,status`, `channel_kind,status`, `expires_at`
- Retention: channel lifecycle evidence
- Migration notes: v1 stores mock metadata only; do not store production private keys, bearer tokens, or credential cache content.

### `local_agent_handshakes`

Purpose: future durable channel handshake metadata.

- Primary key: `id`
- Important columns: `channel_id`, `agent_id`, `challenge`, `response_status`, `issued_at`, `completed_at`, `expires_at`, `metadata jsonb`

- Migration notes: v1 challenge/response is deterministic mock metadata, not production cryptography.

### `local_agent_capability_advertisements`

Purpose: versioned Local Agent capability advertisements.

- Primary key: `id`

### `local_cli_compatibility_entries`

- Important columns: `vendor`, `command`, `version_range`, `provider_template_id`, `parser_mode`, `stdout_policy`, `stderr_policy`, `supported`, `notes`, `created_at`, `updated_at`
- Indexes: `provider_template_id`, `vendor,command`, `supported`
- Retention: provider template lifecycle
- Migration notes: v1 uses fixture versions only and does not detect or execute real CLIs.

### `local_cli_compatibility_results`

Purpose: append-only compatibility check results for Local Agent invocations.

- Primary key: `id`
- Important columns: `provider_id`, `agent_id`, `command`, `reported_version`, `compatible`, `reason`, `parser_mode`, `warnings jsonb`, `created_at`
- Indexes: `agent_id,created_at`, `provider_id,created_at`, `compatible`
- Retention: debug/audit retention
- Migration notes: results are deterministic fixture metadata and must not include CLI output or secrets.

### `local_agent_invocation_envelopes`

Purpose: sanitized invocation envelope metadata for local CLI provider coordination.

- Primary key: `id`
- Important columns: `task_id`, `task_run_id`, `provider_id`, `local_agent_id`, `workspace_ref`, `instruction_set_hash`, `prompt_ref`, `required_consent_level`, `sandbox_profile_id`, `network_policy_id`, `redaction_policy_id`, `secret_scope_ids jsonb`, `timeout_ms`, `signature_status`, `metadata jsonb`, `created_at`

### `local_agent_invocations`

### `local_agent_consent_decisions`

### `local_agent_invocation_streams`

- Important columns: `invocation_id`, `state`, `event_count`, `started_at`, `completed_at`
- Indexes: `invocation_id`, `state,started_at`
- Retention: bounded output lifecycle retention
- Migration notes: stream rows store lifecycle metadata only.

### `local_agent_stream_events`

Purpose: redacted ordered stdout/stderr/system stream event previews.

- Primary key: `id`
- Important columns: `stream_id`, `invocation_id`, `sequence`, `source`, `type`, `payload_preview jsonb`, `redacted`, `created_at`
- Indexes: `stream_id,sequence`, `invocation_id,sequence`, `source,type`
- Retention: bounded output retention under redaction policy
- Migration notes: store size-limited redacted previews only; raw unredacted output must not be stored.

### `local_agent_normalized_events`

Purpose: redacted, normalized stdout/stderr/system event previews.

- Primary key: `id`
- Important columns: `invocation_id`, `source`, `type`, `payload jsonb`, `redacted`, `created_at`
- Indexes: `invocation_id,created_at`, `source,type`
- Retention: bounded output retention under redaction policy
- Migration notes: stdout and stderr remain separate; store size-limited redacted payloads only.

### `local_agent_protocol_audit_events`

Purpose: append-only protocol audit for registration, consent, dispatch, events, completion, blocked credential cache access, and blocked direct local CLI execution.

- Primary key: `id`
- Important columns: `event_type`, `actor_id`, `agent_id`, `invocation_id`, `task_id`, `task_run_id`, `result`, `reason`, `metadata jsonb`, `created_at`
- Indexes: `event_type,created_at`, `agent_id,created_at`, `invocation_id,created_at`, `task_id,task_run_id`
- Retention: long-lived compliance retention
- Migration notes: metadata must be sanitized and must not contain provider API keys, bearer tokens, OAuth tokens, local credential cache contents, or raw secrets.

## Secrets and Sandbox Tables

These tables are schema skeletons only in Secrets and Sandbox Design v0. Runtime repositories remain in-memory and no real secret backend or sandbox runtime is wired.

### `secret_refs`

- Primary key: `id`
- Columns: `provider`, `secret_kind`, `name`, `env_key`, `scope`, `description`, `status`, `metadata jsonb`, `created_at`, `updated_at`
- Indexes: `provider,status`, `secret_kind,status`, `scope`, `env_key`
- Retention: secret reference lifecycle
- Migration notes: never store raw secret values, OAuth tokens, API keys, keychain paths, env var values, or vendor credential cache paths. `env_key` is a reference only for SecretRef-backed Provider Credentials v1.

## Migration Considerations

- Start with append-only audit/revision tables and simple text enum columns.
- Add FK constraints only after migration order and seed data are stable.
- Keep JSONB payloads for registry snapshots, refs, and proposal patches.
- Add unique `(name, version)` constraints for registry entities and packages.
- Use repository contract tests before wiring persistent implementations.
- Persistent DB v1 contract tests run against Postgres only when `AICHESTRA_TEST_DATABASE_URL` is configured.
