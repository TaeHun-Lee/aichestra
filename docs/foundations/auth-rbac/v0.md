# Production Auth/RBAC Planning v0

## What v0 Implements

Production Auth/RBAC Planning v0 adds a provider-neutral identity and authorization boundary without implementing real login or production identity provider integration.

Implemented:

- auth/RBAC domain models in `@aichestra/auth`;
- in-memory repositories for principals, actors, teams, roles, permissions, role bindings, service accounts, identity providers, and auth audit events;
- `AuthProvider` with deterministic `MockAuthProvider` as the default;
- disabled future placeholders for OIDC, SAML, SCIM, and service-account providers;
- `AuthorizationService` to combine RBAC checks with Policy-as-code decisions;
- `RequestContextResolver` for API, system, and test request contexts;
- `AuthContext` to `PolicySubject` mapping with principal id, actor kind, roles, teams, auth mode, service account id, and mock actor marker;
- `/auth/*` read-model endpoints and a gated authorization check endpoint;
- `/health` and dashboard auth visibility;
- sanitized auth audit events.

This milestone is planning/scaffolding only. It does not make Aichestra production-auth ready.

## Why This Exists

Real Git Adapter v2 and LLM Gateway v1 introduced controlled real-provider boundaries. Before broader provider use, GitHub App flows, production webhook hardening, or multi-provider LLM routing, the system needs a stable way to attribute requests to explicit actors and feed those actors into policy decisions.

v0 answers:

- who is acting;
- which principal owns the actor;
- which teams, roles, and permissions are attached;
- which scope the role binding applies to;
- how the resolved actor maps to `PolicySubject`;
- how authorization and policy decisions are audited.

## What v0 Does Not Implement

v0 does not implement:

- real SSO;
- OAuth/OIDC login;
- SAML;
- SCIM directory sync;
- production sessions;
- production API-key issuance;
- password login;
- external identity provider calls;
- real token validation;
- credential cache reads;
- production tenant isolation.

All real auth providers remain disabled placeholders.

## Domain Model

`Principal` is the durable identity metadata for a user, service account, system actor, local agent, or external integration.

`Actor` is the request-time identity derived from a principal. Actors carry role names, team ids, status, kind, and sanitized metadata.

`Team` is a grouping primitive for role bindings.

`Permission` describes an action/resource-kind pair and a risk level.

`Role` groups permission ids. Default roles are `system_admin`, `platform_admin`, `developer`, `reviewer`, `security_admin`, `service_account_runner`, and `viewer`.

`ResourceScope` describes global, org, team, repo, project, task, registry, provider, or local-agent scope.

`RoleBinding` attaches a role to a principal or team at a resource scope.

`ServiceAccount` models non-human actors and allowed scopes. v0 stores no service-account secret material.

`IdentityProvider` records the active mock provider and disabled future provider placeholders.

`AuthContext` carries the resolved actor, principal, teams, roles, permissions, role bindings, auth mode, request id, source, and metadata.

No auth model includes passwords, OAuth tokens, SSO tokens, session secrets, API key raw values, provider credentials, or credential-cache paths.

## AuthProvider

`AuthProvider` defines:

- `getProviderKind()`;
- `resolveAuthContext(request)`;
- `validateActor(actor)`;
- `listRoles()`;
- `listPermissions()`;
- optional `close()`.

`MockAuthProvider` is the default. It resolves deterministic local actors such as `mock-admin`, `user_demo_developer`, `user_demo_viewer`, `user_demo_reviewer`, `user_security_admin`, and `svc_runner`. It marks the context with `authMode = mock` and `productionAuthEnabled = false`.

Future provider placeholders throw disabled/not-implemented errors and do not perform network calls or token validation.

## AuthorizationService

`AuthorizationService` is the central RBAC-to-policy bridge.

It:

- resolves auth context through the configured provider;
- checks whether the actor has an active role binding granting the requested permission;
- maps `AuthContext` into `PolicySubject`;
- calls `PolicyService` when the action/resource is policy-mappable;
- records `authorization_allowed`, `authorization_denied`, and `role_binding_evaluated` audit events;
- keeps policy denial authoritative.

Admin-like roles do not bypass PolicyEngine denials. For example, `platform_admin` has the RBAC permission for future `git.merge`, but Policy-as-code still denies provider merge.

SecretRef-backed Provider Credentials v1 uses this bridge before env-backed credential resolution when `AuthorizationService` is available. Credential checks carry actor/principal/auth mode metadata, deny unauthorized actors before any env read, and still keep PolicyEngine authoritative for provider, Git, LLM, and lease decisions.

## Policy Integration

`PolicySubject` now supports:

- `actorId`;
- `principalId`;
- `actorKind`;
- `roles`;
- `teams`;
- `authMode`;
- `serviceAccountId`;
- `isMockActor`;
- sanitized metadata.

Policy-as-code v0 adds `dashboard.read`, `auth.read`, `auth.authorize.check`, and `auth.admin`. Dashboard/auth metadata reads are allowed for safe mock roles. Production auth administration remains denied.
SecretRef-backed Provider Credentials v1 also uses AuthContext-derived policy subjects for `provider.credential.resolve`, `git.credential.resolve`, and `llm.credential.resolve`.
MCP Gateway v0 uses AuthContext-derived policy subjects for `mcp.server.list`, `mcp.tool.list`, and low-risk mock MCP tool invocation.

Existing denials for remote Git, unverified webhooks, merge, rebase, branch deletion, remote LLM, runner command execution, unsafe MCP tool paths, secret reads, credential cache access, Local Agent secret forwarding, and improvement apply remain unchanged.

## Request Context

`RequestContextResolver` creates:

- API request contexts from request headers such as `x-aichestra-actor-id`, `x-aichestra-request-id`, and `x-aichestra-correlation-id`;
- explicit system contexts;
- explicit test contexts.

Header actor overrides are mock-only convenience for local/tests. They are not production authentication.

## Default Roles And Permissions

`viewer` can read dashboard data and list MCP server/tool metadata.

`developer` can create/read/run mock-safe tasks, use mock LLM/model paths, use runner execution boundaries, create mock branches/PRs when policy permits, invoke low-risk read-only mock MCP tools when policy permits, and read registry metadata.

`reviewer` can approve registry/improvement metadata and read review-related Git metadata.

`security_admin` can inspect security/policy/auth audit metadata and request metadata-only secret lease review paths, but cannot read raw secrets. Credential-resolution checks still require explicit provider gates and policy allow decisions.

`platform_admin` can manage broad platform metadata and run authorization checks, but policy still denies risky operations.

`service_account_runner` can run scoped task/runner/LLM/git branch and low-risk MCP actions only within its role-binding scope.

`system_admin` is a local mock system role and not production auth.

## API Endpoints

Read-model endpoints:

- `GET /auth/config`
- `GET /auth/me`
- `GET /auth/roles`
- `GET /auth/permissions`
- `GET /auth/teams`
- `GET /auth/actors`
- `GET /auth/service-accounts`
- `GET /auth/role-bindings`
- `GET /auth/principals`
- `GET /auth/identity-providers`
- `GET /auth/audit`

Authorization check:

- `POST /auth/authorize/check`

There are no login, logout, password, token, session, SSO, OAuth, SAML, OIDC, or SCIM endpoints in v0.

## Dashboard And Health

`/health` now includes auth provider kind, auth mode, production-auth disabled status, mock-actor enabled status, role catalog count, and permission catalog count.

The dashboard read model now includes auth mode, current actor, roles, teams, role/permission summaries, service-account summary, recent auth audit events, authorization examples, and a warning that mock auth is not production authentication.

No health, API, audit, or dashboard response exposes tokens, cookies, session secrets, raw credentials, password values, or provider secrets.

## Audit Events

Auth audit events include:

- `auth_context_resolved`
- `auth_context_missing`
- `mock_actor_used`
- `authorization_allowed`
- `authorization_denied`
- `role_binding_evaluated`
- `service_account_used`
- `production_auth_not_configured`
- `future_provider_disabled`

Audit metadata is sanitized and must not include tokens, cookies, passwords, API keys, provider credentials, session secrets, or credential-cache contents.

## Future SSO/OIDC/SAML/SCIM Plan

Future production auth work should add:

- request authentication middleware;
- production `AuthProvider` implementations for OIDC and/or SAML;
- SCIM directory ingestion for users, teams, and role assignments;
- durable auth repositories;
- production session management;
- production service-account credential issuance and rotation;
- RBAC administration workflows;
- tenant/org scoping;
- auth-aware DTO filters across service boundaries.

Each real provider must stay behind explicit config gates, SecretRef-backed credential boundaries, policy checks, and sanitized audit.

## Test Strategy

`tests/auth-rbac-v0.test.ts` covers:

- domain repository creation for principals, actors, teams, roles, permissions, role bindings, and service accounts;
- absence of credential material in auth models;
- mock actor resolution;
- future provider disabled behavior;
- RBAC allow/deny decisions;
- policy denial winning over admin-like RBAC;
- scoped service-account authorization;
- `AuthContext` to `PolicySubject` mapping;
- request context creation;
- `/auth/*` endpoints;
- absence of login/token/password endpoints;
- health/dashboard auth visibility;
- no secret/token exposure.

Dashboard read-model tests also cover `/dashboard/auth` and renderer visibility.

## Known Limitations

- Runtime auth repositories are in-memory.
- API handlers outside `/auth/*` still use existing service-specific actor id strings in many places.
- No production auth middleware exists.
- No tenant isolation exists.
- No real identity provider integration exists.
- Postgres auth tables are schema skeletons only.
- Mock header actor overrides are for local/test visibility only.
