# Observability / Audit Retention v0

## Status

Observability / Audit Retention v0 is implemented as a mock-first, read-only foundation.

It adds a shared audit envelope, retention classes, redaction classes, source coverage, in-memory metric and trace skeletons, read-only API endpoints, and a dashboard Observability panel. It does not add a production observability backend, external exporter, alert delivery path, SIEM integration, or retention deletion job.

## What v0 Implements

- `@aichestra/observability` provider-neutral audit, retention, redaction, metric, trace, and correlation models.
- `AuditEventEnvelope` normalization over existing module-specific audit repositories where practical.
- Audit-specific sanitizer for metadata before envelope creation.
- Read-only `ObservabilityService` query and summary APIs.
- Retention policy read models with deletion disabled.
- In-memory metric snapshots and trace-span skeletons derived from normalized audit data.
- Read-only `/observability/*` API endpoints.
- `/dashboard/observability` and web dashboard rendering.
- No-secret/no-token/no-raw-payload tests across sanitizer, API, and dashboard surfaces.

## Common Audit Taxonomy

`AuditEventEnvelope` normalizes module-specific audit events into these categories:

- `auth`
- `policy`
- `credential`
- `git`
- `git_webhook`
- `llm`
- `mcp`
- `runner`
- `registry`
- `improvement`
- `local_agent`
- `security`
- `dashboard`
- `system`

Severity values are `debug`, `info`, `warning`, `error`, and `critical`.

Outcome values are `success`, `denied`, `blocked`, `failed`, `skipped`, and `unknown`.

The envelope surfaces common dimensions when available: actor id, principal id, auth mode, request id, correlation id, task id, task run id, agent run id, repo id, provider id/kind, model id, tool id, secret ref id, and policy decision id.

## AuditEventEnvelope

The envelope stores only sanitized metadata:

- `id`
- `timestamp`
- `category`
- `eventType`
- `severity`
- `outcome`
- common correlation/provider/task dimensions
- `sourceModule`
- `retentionClass`
- `redactionClass`
- `sanitizedSummary`
- `sanitizedMetadata`
- `createdAt`

Raw payloads, raw prompts, raw tool inputs, provider tokens, webhook secrets, credential cache contents, and raw provider credentials are never part of the common envelope.

## Retention Classes

`AuditRetentionClass` is modeled as read-only policy metadata:

- `short_debug`: short-lived debug/demo metadata.
- `operational`: medium-lived operational audit and status records.
- `security`: long-lived security and denial records.
- `compliance`: long-lived governance, registry, and proposal decision records.
- `ephemeral`: derived metrics/traces that can be rebuilt.

`AuditRetentionPolicy` records the expected TTL/export/deletion posture. In v0 every policy uses `deletionMode = none`; no data is deleted and no retention job runs.

## Redaction Classes

`AuditRedactionClass` names the expected sanitization posture:

- `public_metadata`
- `internal_metadata`
- `sensitive_metadata`
- `secret_adjacent`
- `contains_user_content_redacted`
- `never_store_raw`

The audit sanitizer masks API-key-like strings, GitHub tokens, webhook secrets, bearer tokens, JWT-like strings, env dumps, credential cache paths, raw password-like fields, `SESSION_SECRET`, `JWT_SECRET`, `AICHESTRA_LLM_API_KEY`, `AICHESTRA_GITHUB_TOKEN`, `AICHESTRA_GITHUB_WEBHOOK_SECRET`, and future secret backend env names such as Vault/AWS/GCP/Azure secret credentials. It also size-limits strings and large metadata.

## Audit Source Normalization

Integrated sources:

- Auth/RBAC audit events.
- Policy decision audit entries.
- SecretRef credential and Secrets/Sandbox security audit events.
- Git audit events.
- GitHub webhook audit events.
- LLM Gateway audit events.
- MCP Gateway audit events.
- Runner audit events.
- Registry audit logs.
- Improvement Governance audit events.
- Local Agent Protocol audit events.

Partially integrated sources:

- Enterprise Provider Abstraction provider audit events, because provider catalog and blocked invocation records are normalized but future provider-specific audit streams are not implemented.
- Deployment Readiness planning records, because readiness checks and production risks are not audit events but are exposed as observability blocker context.

The full source inventory is in `docs/reference/audit-source-inventory.md`.

## Metric Skeleton

Metric v0 defines deterministic read-only models:

- `MetricDefinition`
- `MetricPoint`
- `MetricSnapshot`

The snapshot includes total audit event counts, denied/blocked/failed counts, security event counts, audit source coverage, retention policy counts, and `observability.external_exporter.enabled = 0`.

GitHub App / Production Webhook Hardening Planning v0 adds planned metric definitions for:

- `github.webhook.deliveries.received`
- `github.webhook.deliveries.verified`
- `github.webhook.deliveries.rejected`
- `github.webhook.duplicate_deliveries`
- `github.webhook.dead_letters`
- `github.pr_sync.success`
- `github.pr_sync.failure`
- `github.branch_sync.success`
- `github.branch_sync.failure`
- `github.api.rate_limit_warnings`
- `github.webhook.processing_latency_ms`

These remain definitions/read-model metadata only. No exporter, alert delivery, or production time-series backend is wired.

Persistent DB Production Operations v1 adds database operations readiness metadata for migration files, index review, retention/audit growth, webhook persistence, backup/restore, and pooling. These are read-only readiness signals only; no production DB metric exporter, retention job, or audit deletion job is wired.

Secret Backend Migration Planning v0 adds planned metric definitions for credential resolution attempts/denials, env fallback usage, disabled/revoked SecretRef usage attempts, rotation overdue count, lease issue/revoke counts, backend errors, and no-secret exposure violations. These are readiness definitions only; no secret backend exporter, alert delivery, or backend poller is wired.

Metrics are in-memory and derived from current service state. No Prometheus, OpenTelemetry Collector, Datadog, Grafana, CloudWatch, Honeycomb, Sentry, Splunk, Loki, Jaeger, or SIEM exporter exists in v0.

## Trace Skeleton

Trace v0 defines:

- `TraceSpan`
- `CorrelationContext`

Trace spans are derived read-model skeletons from normalized audit events that have useful request, correlation, task, run, or provider dimensions. They are not OpenTelemetry spans and are not exported.

## API Endpoints

Read-only endpoints:

- `GET /observability/config`
- `GET /observability/audit/events`
- `GET /observability/audit/summary`
- `GET /observability/audit/retention-classes`
- `GET /observability/audit/redaction-classes`
- `GET /observability/audit/sources`
- `GET /observability/metrics`
- `GET /observability/metrics/snapshot`
- `GET /observability/traces`

All responses are DTO-shaped, query limits are enforced, and non-GET requests are rejected.

## Dashboard Panel

Dashboard API-backed Read Model v0 now includes an Observability panel with:

- audit summary by category/outcome/severity;
- recent denied/blocked/security-relevant events;
- retention and redaction class summaries;
- metric snapshot;
- trace skeleton summary;
- audit source coverage;
- production readiness blockers related to observability;
- explicit no-secret status.

The dashboard does not call observability services, provider APIs, GitHub, LLMs, MCP transports, vendor CLIs, secret stores, or runner commands.

`/dashboard/github-app` separately surfaces GitHub App/webhook hardening readiness, including replay/dead-letter/credential/endpoint posture and no-secret status.

`/dashboard/secret-backend` separately surfaces Secret Backend Migration Planning v0 readiness, including backend options, migration phases, lease/rotation posture, env fallback warning, risks, blockers, and no-secret/no-env status.

## No-Secret Guarantees

v0 enforces:

- no raw secrets in common audit envelopes;
- no raw provider tokens;
- no raw webhook payloads;
- no unredacted credential cache paths;
- no raw unredacted prompts, tool input, or provider output in normalized audit metadata;
- sanitized metadata before envelope creation;
- size-limited metadata previews;
- no-secret API and dashboard regression tests.

## Known Limitations

- Aggregation is in-memory/read-only and adapts existing repositories at request time.
- Many source repositories remain in-memory.
- There is no durable audit event table for the common envelope.
- There is no tenant/org partitioning, legal hold, production export, or deletion workflow.
- Trace spans are derived skeletons, not distributed traces.
- Metrics are derived snapshots, not time-series storage.
- Alert rules and alert delivery remain planning only.
- GitHub App/webhook hardening metrics and audit events are planned definitions/read models only; production delivery tracking, alerting, and export remain future work.
- Persistent DB operations readiness is planning-only; production pooled DB metrics, backup/restore telemetry, retention enforcement evidence, and durable common audit storage remain future work.
- Secret backend migration readiness is planning-only; production backend metrics, rotation evidence, backend audit export, stale credential detection, and env fallback enforcement remain future work.

## Production Blockers

Production remains blocked until:

- durable common audit storage exists;
- production retention jobs and legal hold controls are designed and tested;
- audit export is implemented through a reviewed secure pipeline;
- structured logs, metrics, traces, SLOs, and alerting are implemented;
- tenant/data-residency scoping exists;
- auth, secret backend, policy bundles, database operations, backup/restore, and deployment controls are production-ready.

## Future OpenTelemetry / SIEM / Export Plan

Future work may add OpenTelemetry-compatible logs/metrics/traces, local exporters, SIEM export, audit export checkpoints, alerting, and retention enforcement only after production identity, tenant scoping, SecretRef-backed backend credentials, policy gates, storage operations, and export review controls exist.

Those integrations must remain disabled by default and must not bypass Auth/RBAC, Policy-as-code, SecretRef, sandbox, Git, LLM, MCP, Local Agent, Runner, or Dashboard safety gates.
