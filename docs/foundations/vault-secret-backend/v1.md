# Vault-backed Secret Backend v1

Status: `v1_implemented`

Vault-backed Secret Backend v1 adds the first concrete production-oriented SecretRef provider boundary. It does not make Vault the default backend and does not make Aichestra production-ready for secrets.

Vault Integration-Test Profile v1 now adds a first-class skipped-by-default readiness surface for the optional live Vault KV v2 smoke path. That profile reports gate status, test-case status, safety checks, health metadata, and dashboard data without calling Vault in default runtime/tests.

Vault Live Integration Enablement v1 now adds a stricter manual live-validation readiness layer over that profile. It reports `ready_for_manual_live_test`, `ready_if_configured`, `skipped`, or `blocked_unsafe` metadata, exposes a safe runbook, strengthens test-only path and allowlist checks, and keeps production Vault rollout false.

## What v1 Implements

- `provider: vault` support for `SecretRef` metadata.
- A Vault client boundary with disabled, mock, and gated HTTP client implementations.
- Vault KV v2-style read request mapping from SecretRef metadata.
- A `VaultSecretProvider` integrated with `SecurityControlService`.
- CredentialManager and TokenResolver-compatible credential handles for Vault-backed Git, webhook, LLM, and provider API credentials.
- Auth/RBAC, RequestContext Propagation v1 metadata, Service Account Actor Boundary v1 attribution, Tenant/Repo/Provider Scope Model v1 SecretRef scope metadata, and Policy-as-code checks before any Vault client read.
- SecretLease metadata issuance for successful internal resolutions.
- Vault-specific metadata-only security audit events.
- Vault token/string redaction support.
- Read-only Vault config, health, readiness, resolve-check, audit, health metadata, and dashboard read models.
- Deterministic mock Vault tests plus a skipped-by-default live Vault test skeleton.
- Vault Live Integration Enablement v1 readiness/check/runbook/summary metadata for a safer manual live validation path.

## What v1 Does Not Implement

- No production Vault rollout.
- No Vault HA, unseal, storage backend, cluster bootstrap, Terraform, Helm, Kubernetes, Vault Agent, CSI, or sidecar integration.
- No default Vault connection in runtime or tests.
- No production secret rotation jobs.
- No destructive secret migration.
- No BYOK.
- No OAuth, device-code, WIF, IAM, or cloud identity token exchange.
- No credential cache reads or local CLI credential reads.
- No AWS Secrets Manager, GCP Secret Manager, Azure Key Vault, or custom backend implementation.
- No production readiness claim.

## Provider Config

Vault remains disabled unless all required gates are explicitly configured:

```bash
AICHESTRA_SECRET_BACKEND_PROVIDER=vault
AICHESTRA_ENABLE_VAULT_SECRET_PROVIDER=true
AICHESTRA_VAULT_ADDR=...
AICHESTRA_VAULT_AUTH_METHOD=token
AICHESTRA_VAULT_TOKEN=...
AICHESTRA_VAULT_KV_MOUNT=secret
AICHESTRA_VAULT_ALLOWED_PATH_PREFIXES=aichestra/
AICHESTRA_VAULT_REQUEST_TIMEOUT_MS=5000
AICHESTRA_VAULT_INTEGRATION_TESTS=false
```

`AICHESTRA_VAULT_NAMESPACE` is optional. `AICHESTRA_VAULT_TOKEN_SECRET_REF`, AppRole role id, and AppRole secret id references remain future metadata only.

Health, readiness, dashboard, API, and audit output may expose booleans, counts, status, auth method, and client kind. They must not expose Vault address values, token values, secret values, raw env values, private keys, webhook secrets, provider API keys, or credential cache paths.

## SecretRef Metadata

Vault-backed refs reuse the existing metadata-only `SecretRef` model:

```json
{
  "provider": "vault",
  "secretKind": "llm_api_key",
  "metadata": {
    "vaultMount": "secret",
    "vaultPath": "aichestra/integration/llm",
    "vaultKey": "api_key",
    "vaultVersion": 1,
    "vaultNamespace": "team-a",
    "dataShape": "single_key"
  }
}
```

`vaultPath` and `vaultKey` are references, not secret values. Disabled or revoked SecretRefs fail closed before any backend read.

## KV v2 Mapping

`VaultSecretProvider` maps SecretRef metadata to a KV v2 read request:

- mount: `metadata.vaultMount` or configured default mount;
- path: `metadata.vaultPath`;
- key: `metadata.vaultKey`;
- version: optional `metadata.vaultVersion`;
- namespace: optional `metadata.vaultNamespace` or configured namespace;
- data shape: `single_key` by default, `full_object` remains a metadata shape for future use.

The gated HTTP client isolates the only Vault HTTP boundary. Default runtime and tests use `DisabledVaultClient` or `MockVaultClient`.

## Auth Strategy

v1 models token auth as a gated integration-test path only. The token is process-local client configuration and is never returned by DTOs, health, dashboard, audit, tests, or errors. AppRole and workload identity are future design items.

## Resolution Flow

```text
Git / webhook / LLM / provider credential request
  -> RequestContext / AuthContext / AuthorizationService
  -> purpose PolicyEngine check
  -> provider.credential.resolve policy check
  -> secret.lease request/issue policy checks
  -> Vault path allowlist check
  -> VaultClient read through VaultSecretProvider
  -> metadata-only SecretLease and CredentialHandle
  -> transient internal value only for immediate adapter boundary
  -> sanitized audit/redaction
```

`secret.read`, runner secret injection, Local Agent secret forwarding, local CLI credential access, and credential cache reads remain denied.

## Git And Webhook Flow

The GitHub token, GitHub webhook secret, and GitHub App private-key metadata SecretRefs may use `provider: vault`. This only changes credential resolution support; it does not enable remote Git, webhooks, GitHub App private-key signing, live installation token exchange, auto-merge, force-push, branch deletion, or production webhook traffic.

## LLM Flow

LLM API key and provider API key SecretRefs may use `provider: vault`. Remote LLM completion still requires the existing LLM Gateway gates, model allowlists, budget policy, Auth/RBAC, Policy-as-code, and SecretRef checks. No default test calls an LLM provider.

## MCP And Local Agent Flow

Future MCP tool secrets and Local Agent pairing secrets can be represented as Vault-backed SecretRefs, but MCP tool secret forwarding, Local Agent secret forwarding, and local CLI credential reads remain disabled.

## Audit And Redaction

v1 adds Vault-specific audit events for provider selection, config validation/missing state, path allowlist checks, resolution request/allow/deny/missing/forbidden/unavailable outcomes, and value redaction. Audit metadata includes ids, purpose, provider kind, actor/principal ids when available, policy/authorization decision ids, and sanitized booleans.

Audit must not include Vault tokens, secret values, GitHub tokens, webhook secrets, LLM API keys, GitHub App private keys, raw provider credentials, raw env values, or credential cache paths. Selected Vault-backed credential checks carry sanitized requestId, correlationId, actorId, principalId, authMode, and source metadata.

## API, Health, And Dashboard

Read-only endpoints:

- `GET /readiness/secrets/vault/config`
- `GET /readiness/secrets/vault/checks`
- `GET /readiness/secrets/vault/summary`
- `GET /readiness/secrets/vault/live-readiness`
- `GET /readiness/secrets/vault/live-checks`
- `GET /readiness/secrets/vault/live-runbook`
- `GET /readiness/secrets/vault/live-summary`
- `GET /security/secrets/vault/config`
- `GET /security/secrets/vault/health`
- `POST /security/secrets/vault/resolve/check`
- `GET /security/secrets/vault/audit`
- `GET /dashboard/vault-secret-backend`

`POST /security/secrets/vault/resolve/check` returns resolution status only. It never returns a Vault secret value.

`GET /health` exposes safe Vault metadata: selected backend, enabled/configured booleans, auth method, allowed path prefix count, integration-test gate status, client kind/status, `productionSecretBackendImplemented: false`, and `envFallbackProductionAllowed: false`.

## Optional Live Tests

The live Vault test skeleton is skipped unless every explicit gate is configured:

- `AICHESTRA_VAULT_INTEGRATION_TESTS=true`
- `AICHESTRA_SECRET_BACKEND_PROVIDER=vault`
- `AICHESTRA_ENABLE_VAULT_SECRET_PROVIDER=true`
- `AICHESTRA_VAULT_ADDR`
- `AICHESTRA_VAULT_AUTH_METHOD=token`
- `AICHESTRA_VAULT_TOKEN`
- `AICHESTRA_VAULT_KV_MOUNT`
- `AICHESTRA_VAULT_ALLOWED_PATH_PREFIXES`
- `AICHESTRA_TEST_VAULT_SECRET_PATH`
- `AICHESTRA_TEST_VAULT_SECRET_KEY`

The test path must be allowlisted and non-production. The test reads status/metadata only, never writes, deletes, lists broad paths, rotates secrets, or prints a secret value.

Vault Live Integration Enablement v1 further requires the path allowlist prefix and the test path to look test-only, rejects production-looking path segments, warns on sensitive unmarked key names, and surfaces the result without returning the raw path or key.

## Known Limitations

- The live HTTP client is intentionally isolated and not constructed by default.
- CredentialManager remains synchronous in v1; deterministic CredentialManager integration uses `MockVaultClient`. Optional live tests use the explicit async `VaultSecretProvider.resolveSecretAsync` provider boundary.
- Production Vault operations, identity, policy rollout, rotation, break-glass, audit-device ownership, backup/restore, and env fallback enforcement remain future work.
- Production Auth/RBAC and production policy bundle runtime are still planning-only.
- RequestContext Propagation v1 is mock-first attribution only and does not enable Vault by default.
- Vault transit signing is not implemented. Registry Signed Package / Artifact Trust v1 records only mock signature/provenance metadata and never asks Vault to sign or verify artifacts.
