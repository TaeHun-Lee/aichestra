# Persistent Storage Schema v0

## Scope

This is a Postgres-oriented schema design. Real Integration Foundation v0 introduced it as a plan; Persistent DB v1 now wires the core durable slice to an opt-in Postgres storage provider.

Conventions:

- Primary keys are text ids generated by application code.
- Timestamps use `timestamptz`.
- Domain enums remain text columns for v0 migration flexibility.
- Complex snapshots, refs, lists, and metadata use `jsonb`.
- Audit and revision tables are append-only.

## Core Tables

### `repos`

Purpose: durable repository metadata used by tasks, branch leases, and future Git adapters.

- Primary key: `id`
- Important columns: `provider`, `owner`, `name`, `default_branch`, `remote_url`, `status`, `created_at`, `updated_at`
- Indexes: `provider,status`
- Retention: project lifetime

### `tasks`

Purpose: durable task state.

- Primary key: `id`
- Important columns: `title`, `description`, `status`, `requester_user_id`, `repo_id`, `base_branch`, `branch_name`, `selected_agent`, `selected_model`, `selected_skill_ids jsonb`, `selected_harness_id`, `instruction_set_id`, `budget_limit_usd`, `conflict_risk_score`, `created_at`, `updated_at`
- Foreign keys: `repo_id -> repos.id`, `requester_user_id -> users.id` when user persistence is enabled
- Indexes: `repo_id`, `status`, `requester_user_id`, `created_at`
- Retention: project lifetime
- Migration notes: add partial index for active task statuses later

### `task_runs`

Purpose: durable execution attempts.

- Primary key: `id`
- Important columns: `task_id`, `attempt`, `status`, `agent`, `model`, `model_provider`, selected skill/harness ids and versions, registry refs as `jsonb`, `instruction_set_id`, timing, result, changed files, PR url, error
- Foreign keys: `task_id -> tasks.id`
- Indexes: `(task_id, attempt) unique`, `task_id`, `status`, `created_at`
- Retention: project lifetime
- Migration notes: add explicit TaskRun transition history later

### `usage_ledger_entries`

Purpose: durable LLM/runtime usage and cost attribution.

- Primary key: `id`
- Important columns: `task_id`, `task_run_id`, `user_id`, `repo_id`, `provider`, `model`, `event_type`, token counts, `cost_usd`, `latency_ms`, `skill_version`, `harness_version`, `metadata jsonb`, `created_at`
- Foreign keys: `task_id -> tasks.id`, `task_run_id -> task_runs.id`
- Indexes: `task_id`, `task_run_id`, `user_id`, `provider`, `created_at`
- Retention: billing and audit retention
- Migration notes: never drop task/run attribution

## LLM Gateway Tables

### `llm_models`

Purpose: durable model catalog for future real provider integrations.

- Primary key: `id`
- Important columns: `provider_kind`, `display_name`, `context_window`, `supports_tools`, `supports_streaming`, `input_token_cost_usd`, `output_token_cost_usd`, `status`, `metadata jsonb`, `created_at`, `updated_at`
- Indexes: `provider_kind,status`
- Retention: model catalog lifecycle
- Migration notes: v0 runtime uses an in-memory catalog; persistent wiring is future work.

### `virtual_model_keys`

Purpose: internal policy keys for model routing and budgets. These are not provider API keys.

- Primary key: `id`
- Important columns: `owner_kind`, `owner_id`, `display_name`, `allowed_provider_kinds jsonb`, `allowed_model_ids jsonb`, `monthly_budget_usd`, `per_task_budget_usd`, `rpm_limit`, `tpm_limit`, `status`, `created_at`, `updated_at`
- Indexes: `owner_kind,owner_id`, `status`
- Retention: policy lifecycle
- Migration notes: never store OpenAI, Anthropic, Gemini, Bedrock, or other provider secrets here.

### `llm_audit_events`

Purpose: append-only audit events for LLM routing, blocked calls, allowed calls, and provider validation.

- Primary key: `id`
- Important columns: `event_type`, `task_id`, `task_run_id`, `actor_id`, `provider_kind`, `model_id`, `result`, `reason`, `metadata jsonb`, `created_at`
- Indexes: `task_id`, `task_run_id`, `provider_kind,model_id`, `created_at`
- Retention: operational/compliance retention
- Migration notes: metadata must be sanitized and must not include provider API keys.

### `llm_gateway_requests`

Purpose: sanitized request/result trace records for future real LLM calls.

- Primary key: `id`
- Important columns: `task_id`, `task_run_id`, `provider_kind`, `model_id`, `status`, `input_tokens`, `output_tokens`, `estimated_cost_usd`, `finish_reason`, `metadata jsonb`, `created_at`
- Indexes: `task_id`, `task_run_id`, `provider_kind,model_id`, `created_at`
- Retention: debug/audit retention
- Migration notes: v0 usage ledger remains the source of cost attribution.

### `audit_events`

Purpose: common task and system audit events.

- Primary key: `id`
- Important columns: `actor_user_id`, `action`, `target_type`, `target_id`, `task_id`, `repo_id`, `metadata jsonb`, `created_at`
- Indexes: `actor_user_id`, `target_type,target_id`, `task_id`, `repo_id`, `created_at`
- Retention: compliance retention
- Migration notes: keep append-only

### `pull_requests`

Purpose: durable mock/current PR metadata before real Git provider integration.

- Primary key: `id`
- Important columns: `task_id`, `repo_id`, `provider`, `external_id`, `url`, `status`, `created_at`, `updated_at`
- Indexes: `task_id`, `repo_id,status`
- Retention: task/PR lifetime

### `instruction_sets`

Purpose: durable assembled instruction context for TaskRuns.

- Primary key: `id`
- Important columns: `task_run_id`, `artifacts jsonb`, `assembled_hash`, `max_context_bytes`, `created_at`
- Indexes: `task_run_id`
- Retention: task run lifetime plus audit retention

## Conflict Manager Tables

- Indexes: `repo_id,status`, `task_run_id`, `branch_name`, `expires_at`
- Retention: task/PR lifetime plus audit retention

### `conflict_risks`

- Primary key: `id`
- Columns: `repo_id`, `source_task_run_id`, `target_task_run_id`, `source_lease_id`, `target_lease_id`, `overlap_files jsonb`, `risk_score`, `risk_level`, `reasons jsonb`, `recommendation`, `created_at`
- Indexes: `repo_id`, `source_task_run_id`, `target_task_run_id`, `risk_score`, `created_at`

### `merge_queue_entries`

- Indexes: `repo_id,status`, `priority`, `task_run_id`, `branch_lease_id`
- Retention: PR/task lifetime

### `merge_simulation_results`

- Primary key: `id`
- Columns: `repo_id`, `repo_path`, `base_ref`, `source_ref`, `target_ref`, `task_run_id`, `branch_lease_id`, `mode`, `status`, `conflicting_files jsonb`, `changed_files jsonb`, `summary`, `raw_command_metadata jsonb`, `risk_contribution`, `created_at`
- Indexes: `repo_id`, `task_run_id`, `branch_lease_id`, `status`, `created_at`
- Retention: conflict evidence retention
- Migration notes: raw command metadata must be sanitized

## Registry Tables

### `skills`, `harnesses`, `instructions`

Purpose: durable registry entities.

- Primary key: `id`
- Shared columns: `name`, `version`, `description`, `status`, `approval_status`, `eval_status`, `owner`, `dependencies jsonb`, `created_at`, `updated_at`
- Skill-specific columns: compatible agents/models, required tools/harnesses, invocation rules, instruction refs/body, eval refs, tags
- Harness-specific columns: runtime type/image, tools, MCP servers, secret scopes, network policy, test commands, instruction loading policy
- Instruction-specific columns: type, scope, path/body, checksum metadata, precedence, applies-to fields, max context bytes
- Indexes: `(name, version) unique`, `status`, `approval_status`, `eval_status`, owner
- Retention: registry lifetime

### `registry_audit_logs`

- Primary key: `id`
- Columns: `actor_id`, `action`, `target_kind`, `target_id`, `target_name`, `target_version`, `before jsonb`, `after jsonb`, `reason`, `created_at`
- Indexes: `target_kind,target_id`, `actor_id`, `action`, `created_at`
- Retention: append-only compliance retention

### `registry_revisions`

- Primary key: `id`
- Columns: `target_kind`, `target_id`, `target_name`, `target_version`, `revision_number`, `snapshot jsonb`, `snapshot_checksum`, `change_reason`, `created_by`, `source_audit_log_id`, `created_at`
- Indexes: `(target_kind,target_id,revision_number) unique`, `source_audit_log_id`, `created_at`
- Retention: append-only registry lifetime

### `registry_packages`

- Primary key: `id`
- Columns: `schema_version`, `package_kind`, `name`, `version`, `description`, `owner`, `manifest_version`, `entries jsonb`, `dependencies jsonb`, `checksum`, `checksum_algorithm`, `created_by`, `tags jsonb`, `metadata jsonb`, `created_at`
- Indexes: `(name, version) unique`, `package_kind`, `checksum`
- Retention: registry package lifetime

### `registry_eval_results`

- Primary key: `id`
- Columns: `target_kind`, `target_id`, `target_name`, `target_version`, `eval_name`, `eval_type`, `status`, `score`, `max_score`, `summary`, `details`, `attached_by`, `source`, `artifact_ref`, `attached_at`
- Indexes: `target_kind,target_id`, `status`, `attached_at`
- Retention: eval evidence retention

### `registry_approval_queue_view`

Purpose: optional read model/view for pending registry approvals.

- Source: `skills`, `harnesses`, `instructions`
- Indexes: derived from source tables
- Retention: ephemeral/rebuildable

## Migration Considerations

- Start with append-only audit/revision tables and simple text enum columns.
- Add FK constraints only after migration order and seed data are stable.
- Keep JSONB payloads for registry snapshots, refs, and proposal patches.
- Add unique `(name, version)` constraints for registry entities and packages.
- Use repository contract tests before wiring persistent implementations.
- Persistent DB v1 contract tests run against Postgres only when `AICHESTRA_TEST_DATABASE_URL` is configured.
